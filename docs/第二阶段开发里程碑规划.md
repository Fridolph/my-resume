# 第二阶段开发里程碑规划

## 规划目标

第二阶段的目标不再是“把结构搭起来”，而是把第一阶段已经稳定下来的结构真正接入业务运行能力。

如果说第一阶段解决的是“怎么重构”，那么第二阶段解决的就是“怎么真正跑起来”。

因此，第二阶段会重点围绕以下方向展开：

- mock 数据替换为真实服务
- 后台内容持久化
- 前后台接口打通
- 内容发布链路细化
- 测试与部署能力提升

## 第二阶段总原则

第二阶段建议继续沿用第一阶段的节奏控制原则：

- 先打通数据链路，再扩展业务能力
- 先明确共享契约，再接实现
- 先完成闭环，再做优化
- 保持 Web 与 Admin 同步演进，但避免同时展开过多模块

也就是说，第二阶段不建议“一口气上数据库、鉴权、部署、测试、CMS 全都做”，而是要继续分阶段推进。

## 推荐里程碑总览

建议第二阶段拆为以下 6 个阶段：

- `P1`：真实数据层与 API 基础设施
- `P2`：后台持久化与内容读写闭环
- `P3`：前后台内容打通
- `P4`：内容发布流与版本能力
- `P5`：测试体系与质量保障
- `P6`：部署、环境与上线准备

下面是每个阶段的建议范围。

---

## 里程碑 P1：真实数据层与 API 基础设施

### 目标

建立真实服务层基础设施，为后续后台和前台的内容读写提供统一入口。

### 重点任务

- 选择并确定数据库方案
- 选择并确定 ORM / 数据访问层
- 建立 `server` 或 API route 基础结构
- 把 `packages/sdk` 从纯 mock 逐步演化为真实请求能力
- 明确共享 DTO 与数据库模型的关系
- 设计基础环境变量结构

### 完成标准

- 项目具备真实 API 基础结构
- 共享 DTO 和后端模型边界明确
- 后续后台模块具备接入真实读写的基础条件

### 建议优先级

这是第二阶段的起点，优先级最高。

---

## 里程碑 P2：后台持久化与内容读写闭环

### 目标

让后台从“本地状态管理器”变为“真实内容管理后台”。

### 重点任务

- 用户、文案、简历、项目、站点配置逐步接入数据库
- 建立后台读取、创建、更新接口
- 建立基础错误处理和校验
- 建立通用表单保存反馈机制
- 明确各模块的表结构或文档结构

### 完成标准

- 后台核心模块的增删改查不再依赖本地 `useState`
- 后台内容可以真实持久化
- 基础 API 与后台页面形成闭环

### 建议优先级

在 `P1` 之后立即开始，是第二阶段最关键的业务主线之一。

---

## 里程碑 P3：前后台内容打通

### 目标

让前台真正消费后台管理结果，而不是继续消费 SDK mock 数据。

### 重点任务

- 前台简历页接入后台简历内容
- 前台项目列表和详情页接入后台项目内容
- 前台 UI 文案与后台翻译结果建立边界
- 前台站点默认设置接入后台站点配置
- 调整 `packages/sdk` 到真实内容读取模式

### 完成标准

- Web 主要公开内容由后台真实数据提供
- Web 与 Admin 不再只是“结构对齐”，而是真正打通
- mock 数据可以逐步退场

### 建议优先级

这是第二阶段最能体现重构成果的阶段，建议作为 `P2` 之后的核心推进项。

---

## 里程碑 P4：内容发布流与版本能力

### 目标

把当前简单的状态字段升级为更完整的内容工作流。

### 重点任务

- 细化 `draft / reviewing / published / archived` 的流转规则
- 增加发布时间、更新人、审核人等字段
- 设计内容版本记录能力
- 设计回滚能力或历史快照能力
- 设计文案、简历、项目的一致性发布策略

### 完成标准

- 内容模块具备基本发布流
- 关键内容可追踪变更和状态流转
- 后台开始从“管理页面”升级为“内容工作流后台” 

### 建议优先级

在前后台打通后再进入这个阶段最合适，否则工作流会建立在不稳定的数据基础上。

---

## 里程碑 P5：测试体系与质量保障

### 目标

为第二阶段及后续演进建立更稳定的自动化质量基础。

### 重点任务

- 为关键 composable 建立单元测试
- 为核心页面和后台模块建立关键流程测试
- 为 API 建立基础集成测试
- 为 SEO 端点建立基础输出测试
- 为 i18n、主题和权限链路建立关键验证

### 完成标准

- 核心路径具备自动化回归保障
- 结构调整不再完全依赖手工验证
- 发布风险明显下降

### 建议优先级

不建议拖到很后面，但也不建议在真实服务还没稳定前大面积铺开。推荐在 `P2` / `P3` 后逐步补充。

---

## 里程碑 P6：部署、环境与上线准备

### 目标

让项目从“本地可开发”进入“可部署、可联调、可上线”的阶段。

### 重点任务

- 确定部署平台与部署方式
- 管理环境变量与多环境配置
- 确定数据库迁移与初始化流程
- 确定前后台部署关系
- 增加线上 SEO / 站点配置校验
- 补充上线前检查清单

### 完成标准

- 项目具备稳定部署方式
- 环境配置可复制、可迁移
- 具备从开发到预发 / 生产的基本路径

### 建议优先级

这是第二阶段的收口阶段，适合在前几项基础能力稳定后集中推进。

---

## 推荐执行顺序

推荐第二阶段按以下顺序推进：

1. `P1` 真实数据层与 API 基础设施
2. `P2` 后台持久化与内容读写闭环
3. `P3` 前后台内容打通
4. `P4` 内容发布流与版本能力
5. `P5` 测试体系与质量保障
6. `P6` 部署、环境与上线准备

这个顺序的核心逻辑是：

- 先搭底层
- 再接后台
- 再接前台
- 再补工作流
- 再补质量保障
- 最后做上线收口

## 建议的起步动作

如果马上进入第二阶段，我建议第一步不要直接写所有 API，而是先完成这三件事：

- 确认数据库和 ORM 方案
- 明确共享 DTO 和数据库模型如何映射
- 选出第一个打通模块作为样板

在当前项目里，最适合作为第二阶段样板模块的是：

- `站点配置`
- 或 `项目管理`

原因是这两个模块结构相对清晰、跨前后台价值高，而且比完整鉴权系统改造更适合作为真实数据链路的起点。

## 总结建议

如果要用一句话概括第二阶段，我建议这样定义：

> 第二阶段的核心不是继续“搭结构”，而是把第一阶段稳定下来的结构真正接入真实数据、真实工作流和真实部署路径。

只要继续保持第一阶段这种按里程碑推进、每步都落文档和复盘的方式，第二阶段也会推进得比较稳。
