# Web 前端测试与部署策略

## 当前测试策略

由于当前仍处于重构第一阶段，前端暂时没有引入完整的自动化测试体系，当前主要采用以下验证方式：

- `pnpm --dir apps/web typecheck`
- 本地 `Nuxt dev` 运行验证
- 关键页面与关键 SEO 端点的 HTTP 检查

这套策略虽然轻量，但对当前阶段是有效的，因为它能优先保证：

- 目录边界和类型边界稳定
- 页面可以实际访问
- 内容读取链路正常
- SEO 关键端点可访问

## 建议的下一阶段测试补充

当 Web 第二阶段开始接入真实 API 或更复杂交互时，建议补充：

- composable 单元测试
- 页面级关键流程测试
- 多语言切换测试
- 主题切换测试
- `robots.txt` 与 `sitemap.xml` 输出测试

如果后续要补自动化测试，建议优先从最稳定的 pure logic 和关键页面入口开始，而不是一开始就全量覆盖。

## 当前部署约束

当前前端部署仍以第一阶段收尾为目标，因此部署说明保持在基础级别。当前需要关注的部署前提包括：

- 配置正确的站点域名
- 确保 `runtimeConfig.public.siteUrl` 指向真实线上地址
- 确保公开静态资源可访问，例如 `og-default.svg`
- 确保 `robots.txt` 与 `sitemap.xml` 可以被线上环境正常输出

## 当前建议的部署检查项

部署前建议检查以下内容：

- 首页、简历页、项目页是否能正常访问
- 项目详情页是否能正常访问
- canonical 是否指向正确域名
- `robots.txt` 是否可访问
- `sitemap.xml` 是否可访问
- OG 图地址是否可访问

## 后续部署演进方向

后续如果继续推进部署能力，建议重点处理：

- 生产环境域名注入方式
- SSG / SWR 策略与内容更新策略衔接
- 线上缓存策略
- OG 图和 sitemap 的自动化检查

这样可以让当前第一阶段已经完成的 Web 结构，更平滑地进入生产可维护状态。
