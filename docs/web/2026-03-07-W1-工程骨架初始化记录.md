# Web W1：Nuxt 4 展示站工程骨架初始化记录

## 背景

旧项目是一个基于 `Vite + Vue 3` 的单页简历站，它完成了展示目标，但已经不适合继续承载新的产品形态。这一轮重构里，前端展示站会迁移到 `Nuxt 4 + TypeScript + Nuxt UI`，并纳入 Monorepo。

W1 的目标非常明确：先把前端展示站的工程壳搭起来，不急着迁移业务页面。

## 本阶段目标

W1 只解决“站点能不能作为新工程活起来”这个问题。

本阶段完成了以下内容：

- 创建 `apps/web`
- 初始化 `Nuxt 4`
- 接入 `TypeScript`
- 接入 `Nuxt UI`
- 接入 Monorepo workspace
- 建立基础布局与默认首页
- 打通本地类型检查命令

## 为什么这样拆

如果一开始就在新 `web` 里直接重写首页、简历页和项目页，工程结构、共享配置和目录边界都还没有稳定，后面会反复返工。

因此 W1 的重点不是“页面完成度”，而是“工程稳定性”。只有骨架稳定，后面的页面迁移、数据接入和 SEO 才有可靠基础。

## 本次实现

### 1. 建立 `apps/web`

`apps/web` 使用 `Nuxt UI` 模板初始化，然后裁剪掉模板自带的营销展示内容。这样既保留了官方推荐的基础集成方式，又避免把模板示例带进后续业务实现。

### 2. 接入根级 Monorepo

本次没有让 `apps/web` 维持自己的独立 workspace，而是移除了应用内的 `pnpm-workspace.yaml` 和锁文件，统一回到根仓库管理。

这样处理有两个直接好处：

- 后续 `web`、`admin`、`packages/*` 可以共享依赖安装与脚本入口。
- 共享包接入时不需要再额外做仓库间同步。

### 3. 保留最小首页骨架

W1 首页只保留三个能力：

- 能证明 `Nuxt UI` 已正确工作。
- 能说明 `web` 的后续职责。
- 能作为后面 W2 页面骨架迁移的临时落点。

这比直接保留模板营销页更合适，因为它更贴近当前项目的真实上下文。

### 4. 接入统一配置预留

当前阶段已经建立了根级 `pnpm-workspace.yaml`、`tsconfig.base.json` 和 `packages/*` 目录，这为后续共享类型、schema、sdk 和 UI 包提供了预留位。

虽然 W1 还没有真正消费这些共享包，但边界已经建立好了。

## 验证结果

本阶段完成后，执行了以下验证：

- `pnpm install`
- `pnpm --dir apps/web typecheck`

类型检查已经通过，说明 `web` 的基础骨架可继续承接后续开发。

## 当前结果

W1 完成后，`web` 已具备以下状态：

- 可以作为 `Nuxt 4` 应用独立启动
- 可以作为 workspace 子应用参与统一管理
- 具备 `Nuxt UI` 基础布局
- 具备后续接入页面路由、i18n 和内容读取层的稳定入口
