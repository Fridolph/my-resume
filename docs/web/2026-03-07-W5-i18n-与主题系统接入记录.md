# Web W5：i18n 与主题系统接入记录

## 背景

W4 完成后，Web 端已经具备了结构化内容读取能力，但它仍然缺少两个非常关键的能力：前台语言切换，以及用户可感知的主题状态管理。

如果这两个能力继续缺席，后续即使后台已经具备翻译管理能力，前台也无法真正消费这些结果；同时，主题切换也只能停留在组件级按钮，而不是可管理、可持久化的站点级状态。

因此，W5 的任务不是简单加一个切换按钮，而是要把“语言状态”“主题状态”“内容来源边界”这三件事一起建立起来。

## 本阶段目标

W5 聚焦以下几个目标：

- 为 Web 建立统一的 locale 状态。
- 为 Web 建立统一的 theme 状态。
- 让页面内容可以随 locale 变化而更新。
- 明确 UI 文案和业务内容的来源边界。
- 建立语言与主题的持久化策略。

这一阶段完成后，Web 端就不再只是“能展示内容”，而是具备了作为公开内容站继续扩展的基础状态系统。

## 为什么 W5 先不做多语言路由

W5 选择的是“统一路由 + locale 内容切换”，而不是一开始就做 `/en/...`、`/zh/...` 这样的多语言路由。

这样做有一个很现实的考虑：当前重构仍在第一阶段，内容契约、后台翻译接口、SEO 方案都还在逐步稳定。如果现在就把路由级 i18n 也一起引入，复杂度会显著增加，且会让页面、SEO、导航和内容层一起耦合升级。

因此，W5 先把更基础的边界稳定下来：

- locale 如何存储
- locale 如何影响页面内容
- UI 文案从哪里来
- 业务内容从哪里来
- theme 如何统一控制与持久化

等这些边界稳定后，再升级到多语言路由会更稳，也更容易和 W6 的 SEO 工作对齐。

## 本次实现

### 1. 建立共享 locale / theme 类型

W5 先在 `packages/types` 中新增了前台所需的基础类型：

- `WebLocale`
- `ThemePreference`

这样做的意义在于，前台语言和主题不再只是某个页面里的字符串常量，而是共享层可以识别的契约。后续如果后台或 SDK 继续扩展这些能力，也能沿用同一套类型边界。

### 2. 建立 `useWebLocale` composable

W5 新增了 `apps/web/app/composables/useWebLocale.ts`，它负责以下事情：

- 管理当前 locale 状态
- 提供切换语言的方法
- 维护前台界面文案词典
- 将 locale 持久化到 cookie 和 `localStorage`
- 在客户端同步更新 `html lang`

这里有一个很重要的设计点：W5 没有把所有文案都塞进 SDK，而是只把界面文案放到 `useWebLocale` 中。例如：

- 导航文案
- 加载态与错误态文案
- 页脚说明文案
- 按钮与主题标签文案

这些内容属于前端 UI copy，本质上是界面交互的一部分，而不是业务内容本身。

### 3. 建立 `useWebTheme` composable

W5 新增了 `apps/web/app/composables/useWebTheme.ts`，它建立了站点级主题状态封装。

实现方式是：

- 以 Nuxt 当前的 `color-mode` 能力为底座
- 在 Web 层提供统一的主题偏好状态
- 将偏好写入 `web-theme` cookie 和 `localStorage`
- 为界面提供已本地化的主题选项标签

这一步的价值在于，主题切换从“组件按钮”升级为“可管理的应用状态”。后续如果要把主题偏好写入用户设置，或者让后台管理更多展示策略，这层封装就能继续承接。

### 4. 让 SDK 支持多语言内容 DTO

W5 把 `packages/sdk` 中的 Web 内容数据升级为带 locale 参数的读取函数，包括：

- `getHomePageContent(locale)`
- `getResumePageContent(locale)`
- `getProjectsPageContent(locale)`
- `getProjectDetailContent(slug, locale)`

这意味着业务内容也开始具备明确的多语言输入边界。当前使用的是 SDK Mock，但调用方式已经和未来的后台 API 足够接近。

这个设计非常关键，因为它把“语言状态”和“业务内容读取”真正接上了，而不是只切换了几个 UI 标签。

### 5. 让页面在 locale 变化时自动刷新内容

W5 更新了 `useWebContent`，让页面级内容读取基于当前 locale 执行，并在 locale 变化时自动刷新。

这一步带来的结果是：

- 首页随语言切换更新
- 简历页随语言切换更新
- 项目列表页随语言切换更新
- 项目详情页随语言切换更新

这样页面层继续保持“编排层”职责，不需要自己关心数据从哪里切换，只要消费当前 locale 对应的数据即可。

### 6. 在头部建立语言与主题切换入口

W5 新增了头部交互组件：

- `WebLocaleSwitch`
- `WebThemeSwitch`

它们把 `useWebLocale` 和 `useWebTheme` 暴露出来的状态真正连接到 UI 层，让用户可以直接切换语言和主题。

同时，页脚也开始展示当前语言和主题状态。这让前台页面不仅“支持切换”，还可以清晰反馈当前站点状态。

## UI 文案与业务内容的边界

W5 一个非常重要的产出，不只是功能，而是边界定义。

当前项目采用以下分层：

- `useWebLocale`：管理 UI 文案
- `packages/sdk`：提供业务内容 DTO
- 页面组件：负责内容绑定与编排
- 展示组件：负责 UI 呈现

这样划分后，后续后台翻译模块接入时会非常清晰：

- 导航、按钮、状态文案可以继续作为前台 UI copy 管理
- 简历内容、项目内容、项目详情等业务内容可以由后台内容服务提供

这就避免了把所有翻译都混成一个层，也让前台和后台之间的协作边界更稳定。

## 持久化策略

### 语言持久化

W5 采用双重持久化：

- `web-locale` cookie
- `localStorage`

这样既能让 SSR 首屏读取到当前语言，也能让客户端在后续导航中保持一致状态。

### 主题持久化

W5 采用同样的思路：

- `web-theme` cookie
- `localStorage`
- `color-mode` 偏好同步

这样主题状态既可以在前台保持，又可以继续与 Nuxt UI 当前的主题生态兼容。

## 本地验收结果

### 1. 类型检查

执行通过：

- `pnpm --dir apps/web typecheck`

### 2. 本地服务

执行：

- `pnpm --dir apps/web dev --port 3000`

由于本地 `3000` 端口被占用，Nuxt 自动切换到 `3003`，最终在 `http://localhost:3003` 完成验收。

### 3. 多语言内容验证

通过注入 `web-locale` cookie，已验证：

- 首页可以返回英文导航与英文首页内容
- 简历页可以返回英文简历章节
- 项目详情页可以返回英文详情说明与按钮文案

已确认命中以下英文内容：

- `Home`
- `Resume`
- `Projects`
- `Profile`
- `Education`
- `Back to Projects`

### 4. 主题状态验证

通过注入 `web-theme=dark` cookie，已验证页脚可以返回当前主题状态，例如：

- `当前主题：深色`

这说明当前主题偏好已经进入前台状态系统，而不是停留在孤立组件中。

## 本阶段结果

W5 完成后，Web 端已经具备以下能力：

- 前台语言切换 composable
- 前台主题切换 composable
- locale 与 theme 持久化策略
- 支持双语内容的 SDK DTO
- 页面随 locale 刷新的内容读取链路
- UI 文案与业务内容的清晰边界

这意味着 Web 端已经从“数据驱动页面”进一步升级为“具备状态系统的公开内容站”。

W5 完成后，最自然的下一步是进入 W6：Web SEO 与内容站能力完善。

下一阶段会重点处理：

到那时，W5 建立的 locale 状态和内容边界会直接成为 SEO 设计的前提条件。
