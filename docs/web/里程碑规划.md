# Web 重构里程碑规划

## 文档说明

这份文档用于跟踪前端展示站的重构过程。它不是单纯的待办清单，而是一份可持续更新的实施记录。

使用方式如下：

- 每个里程碑开始前，先确认范围。
- 每个子任务完成后，补充“完成总结”。
- 如果实现过程中有方案调整，直接在对应里程碑下补充“变更记录”。
- 完成一个里程碑后，再进入下一个里程碑，避免并行过多导致范围失控。

---

## Web 重构目标

新的 `web` 应用承担以下职责：

- 作为面向外部访客的主站。
- 承载首页、简历页、项目页等公开内容。
- 支持多语言切换。
- 支持 SEO、Open Graph、Sitemap 等内容站能力。
- 从结构化数据中渲染页面，而不是直接写死内容。

---

## 里程碑 W4：Web 内容读取层接入

### 目标

让前台开始从结构化数据源读取内容，替代原项目中直接写死在组件里的内容。

### 子任务

- [x] 设计前台内容读取方式
- [x] 接入 `packages/sdk` 或服务端请求层
- [x] 定义首页内容 DTO
- [x] 定义简历内容 DTO
- [x] 定义项目列表 DTO
- [x] 建立内容加载状态处理
- [x] 建立请求错误处理
- [x] 建立公开内容缓存策略预留

### 完成标准

- 首页可读取结构化内容。
- 简历页可读取结构化内容。
- 项目列表页可读取结构化内容。
- 数据读取与组件展示边界清晰。

### 完成总结

> 已完成。W4 已将首页、简历页、项目列表页和项目详情页切换为通过 `packages/sdk` + `useWebContent` composable 读取结构化内容，并补齐页面级加载态与错误态。共享类型已新增首页、简历、项目列表、项目详情 DTO，页面层不再直接写死核心内容；同时通过统一的 `useAsyncData key` 为后续接入 SSR 缓存、静态 payload 与后台 API 保留了稳定边界。`pnpm --dir apps/web typecheck` 已通过，且本地 `Nuxt dev` 已验证四个页面访问正常。

### 变更记录

> W4 先采用 SDK Mock，而不是直接连后台接口。这样可以优先把“数据边界、DTO 结构、页面状态处理”稳定下来，后续只需替换 `packages/sdk` 的实现即可，不必再次重构页面组件。

## 里程碑 W5：Web i18n 与主题系统接入

### 目标

在新前端站点中建立统一的多语言与主题切换能力，并与后台可管理内容配合。

### 子任务

- [x] 设计前台多语言切换方案
- [x] 接入前台 i18n 基础设施
- [x] 定义公开文案与业务内容的读取边界
- [x] 建立主题切换 composable
- [x] 建立语言切换 composable
- [x] 建立语言状态持久化策略
- [x] 建立主题状态持久化策略
- [x] 验证多语言路由或多语言内容渲染

### 完成标准

- 前台具备稳定的语言切换能力。
- 前台具备稳定的主题切换能力。
- 页面内容与界面文案的来源边界明确。

### 完成总结

> 已完成。W5 已在 `apps/web` 中建立前台 locale 与 theme composable，接入 `web-locale` / `web-theme` 的 cookie 与 `localStorage` 持久化，并将首页、简历页、项目页与项目详情页切换为随 locale 变化而刷新内容。当前界面文案统一收口在 `useWebLocale`，业务内容统一由 `packages/sdk` 提供多语言 DTO，从而明确了“UI 文案”和“业务内容”的边界。`pnpm --dir apps/web typecheck` 已通过，本地 `Nuxt dev` 已验证中文、英文和主题状态显示都能正常工作。

### 变更记录

> W5 暂不引入完整路由级 i18n 模块，而先采用“统一路由 + locale 内容切换”的方式推进。这样可以先稳定内容契约、状态持久化和后台翻译对接边界，后续若需要再升级为多语言路由，不会影响当前页面与 DTO 结构。
