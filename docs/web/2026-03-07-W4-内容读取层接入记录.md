# Web W4：内容读取层接入记录

## 背景

W3 完成后，Web 端已经具备了稳定的页面骨架和展示组件体系，但页面中的核心内容仍然是写死在页面文件里的。这种方式适合快速搭建结构，不适合继续扩展，因为它会带来三个直接问题：

- 页面既负责布局，又负责维护内容文本。
- 后续接入后台接口时，页面层会被迫再次重构。
- 多语言、缓存和 SEO 内容很难形成统一边界。

因此，W4 的核心目标不是“把接口接上”，而是先建立一层稳定的内容读取边界，让页面开始真正转向数据驱动。

## 本阶段目标

W4 聚焦以下四件事：

- 为 Web 页面定义结构化 DTO。
- 在 `packages/sdk` 中提供统一内容读取入口。
- 在 Web 端封装 `useWebContent` composable。
- 为页面补齐加载态、错误态与缓存边界预留。

这一步完成后，首页、简历页、项目页和项目详情页就都不再直接依赖硬编码内容，而是依赖一层可替换的数据来源。

## 为什么先用 SDK Mock

W4 没有直接连接后台 API，而是先在 `packages/sdk` 中提供 mock 数据。这不是为了偷懒，而是为了控制重构节奏。

如果现在就把 Web 和后台强绑定，会出现两个问题：一是前后台需要同步推进，影响节奏；二是 DTO 还没有稳定时，接口结构很容易来回调整，最终把页面层也拖进反复修改中。

先用 SDK Mock 有三个好处：

- 可以先把页面的数据边界固定下来。
- 可以先验证组件体系是否真的适合数据驱动。
- 后续只需要替换 SDK 内部实现，就能平滑切换到真实接口。

这也是 Monorepo 下比较稳妥的推进方式：先稳定共享契约，再替换实现来源。

## 本次实现

### 1. 在共享类型中定义 Web DTO

W4 在 `packages/types/src/index.ts` 中新增了 Web 端内容模型，包括：

- `HomePageContent`
- `ResumePageContent`
- `ProjectsPageContent`
- `ProjectDetailContent`
- `WebStatItem`
- `WebFeatureItem`
- `ResumeSectionItem`
- `ProjectItem`

这些类型解决的不是“类型好不好看”的问题，而是把前后台之间未来要交换的数据结构先固定下来。

这样做以后，页面只关心“拿到了什么结构”，而不关心“这些数据来自哪里”。这正是后续接入 CMS、管理后台和多语言内容的基础。

### 2. 在 `packages/sdk` 中建立内容读取入口

W4 在 `packages/sdk/src/index.ts` 中新增了四个内容读取函数：

- `getHomePageContent()`
- `getResumePageContent()`
- `getProjectsPageContent()`
- `getProjectDetailContent(slug)`

当前这些函数返回的是 mock 数据，并通过异步函数模拟了真实请求形态。

这个设计让 Web 页面不直接依赖某个接口地址，也不直接依赖某个 JSON 文件，而是依赖“内容读取能力”。等后台 API 准备好后，只需要把 SDK 的内部实现替换掉，页面层和组件层都不需要改动。

### 3. 在 Web 端封装 `useWebContent`

W4 新增了 `apps/web/app/composables/useWebContent.ts`，把页面数据读取统一封装为 composable：

- `useHomePageContent()`
- `useResumePageContent()`
- `useProjectsPageContent()`
- `useProjectDetailContent(slug)`

这样做有两个直接收益。

第一，页面不需要知道 SDK 的具体实现细节。第二，后续如果要增加缓存策略、埋点、错误日志、请求头或 locale 参数，也只需要在 composable 层调整，而不必逐个修改页面。

这也是 W4 中“公开内容缓存策略预留”的具体落点：缓存能力暂时没有真正落到 HTTP 层，但页面已经统一从稳定 key 的 `useAsyncData` 边界读取内容，后续可以平滑接入 payload 缓存、静态化策略或服务端缓存。

### 4. 页面改为内容驱动渲染

W4 已重构以下页面：

- `apps/web/app/pages/index.vue`
- `apps/web/app/pages/resume.vue`
- `apps/web/app/pages/projects/index.vue`
- `apps/web/app/pages/projects/[slug].vue`

每个页面都通过对应的 composable 读取数据，再把数据传递给 W3 已经建立好的展示组件，例如：

- `WebSectionIntro`
- `WebStatCard`
- `WebFeatureCard`
- `WebResumeBlockCard`
- `WebProjectPreviewCard`

这样一来，页面层真正退化为“编排层”，展示组件继续负责 UI 呈现，内容读取层负责提供结构化数据，这三层边界已经比 W3 更清晰了。

### 5. 建立加载态与错误态

W4 没有把 `useAsyncData` 的状态忽略掉，而是把它们明确体现在页面里：

- `pending` 时展示加载中的 `UCard`
- `error` 或无数据时展示 `UAlert`
- 成功时再渲染正式内容

这一步很重要，因为它决定了未来无论数据来自 mock、后台 API，还是静态内容源，页面都有一致的状态表达，不会在后面接接口时再补一次基础体验。

## 本地验收结果

### 1. 类型检查

执行通过：

- `pnpm --dir apps/web typecheck`

### 2. 本地服务

执行：

- `pnpm --dir apps/web dev --port 3000`

由于本地 `3000` 端口被占用，Nuxt 自动切换到 `3002`，最终在 `http://localhost:3002` 完成验收。

### 3. 页面验证

已验证以下页面可以正常访问，并返回 W4 对应内容：

- `/`
- `/resume`
- `/projects`
- `/projects/resume-platform`

同时确认页面返回内容中已经包含以下关键标识：

- `W4`
- `Resume DTO`
- `Projects DTO`
- `Project Detail DTO`
- `SDK Mock`

这说明 W4 的内容读取层已经真正参与页面渲染，而不是停留在类型定义阶段。

## 本阶段结果

W4 完成后，Web 端已经具备以下能力：

- 结构化的页面 DTO
- 可替换的数据读取入口
- 页面级内容 composable
- 明确的加载态和错误态
- 面向未来缓存与 API 切换的稳定边界

这意味着 Web 端已经从“组件化页面”正式进入“数据驱动页面”阶段。

## 下一步

W4 完成后，最自然的下一步是进入 W5：前台 i18n 与主题系统接入。

下一阶段会重点处理：

- 语言切换能力
- 内容与界面文案边界
- 多语言内容接入策略
- 主题状态持久化

到那时，当前 W4 建立的 DTO 与内容读取层会直接成为前台 i18n 的基础承载面。
