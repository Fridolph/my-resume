# Web W6：SEO 与内容站能力完善记录

## 背景

W5 完成后，Web 端已经具备了内容驱动、语言切换和主题状态能力，但它仍然缺少作为公开内容站最关键的一层能力：SEO 与可抓取性。

如果这些能力缺失，站点虽然可以正常访问，但它还不算一个真正适合长期公开展示的内容站。搜索引擎无法稳定理解页面边界，分享链接也缺少统一的 Open Graph 表达，而爬虫也无法通过 `robots.txt` 和 `sitemap.xml` 快速识别站点结构。

因此，W6 的任务不是再做页面本身，而是补齐内容站的“可被搜索引擎和分享平台理解”的基础设施。

## 本阶段目标

W6 聚焦以下几个目标：

- 建立站点级 SEO 配置。
- 为核心页面补齐标题、描述和 canonical。
- 补齐 Open Graph 和 Twitter 分享信息。
- 提供默认 OG 图资源。
- 提供 `robots.txt` 和 `sitemap.xml`。
- 明确主要页面与 SEO 端点的 SSR / SSG 策略。

这一步完成后，Web 端就不再只是一个可访问页面集合，而是开始具备内容站的基本搜索友好能力。

## 为什么 W6 先不引入额外 SEO 模块

W6 的实现思路是：优先用当前 Nuxt 项目内已有能力把 SEO 闭环搭起来，而不是立即引入一整套额外 SEO 模块。

这样做的原因很现实：当前重构仍在第一阶段，站点结构、内容边界和多语言策略都还在逐步稳定。如果现在就叠加更复杂的 SEO 模块生态，会让改动面迅速扩大，反而不利于先把最关键的 URL、meta、OG、robots 和 sitemap 基础能力固定下来。

所以 W6 先完成：

- 站点 URL 和站点信息配置
- 页面 canonical
- 页面 OG / Twitter meta
- 默认 OG 图资源
- `robots.txt`
- `sitemap.xml`
- 页面级结构化数据

这为下一阶段继续深化 SEO 能力提供了足够稳定的基础。

## 本次实现

### 1. 建立站点级 SEO 配置入口

W6 新增了 `apps/web/app/composables/useSiteSeoConfig.ts`，用于统一管理：

- 站点 URL
- 站点名称
- 站点默认描述
- 默认 locale
- 站点支持语言
- 默认 OG 图 URL

同时，`apps/web/nuxt.config.ts` 中新增了 `runtimeConfig.public` 相关字段，把站点基础 SEO 信息提升为全局配置，而不是散落在页面文件里。

这样做以后，页面和 SEO composable 不再依赖硬编码地址，后续如果域名或站点描述变化，只需要修改统一配置即可。

### 2. 重构 `usePageSeo`

W6 扩展了 `usePageSeo` 的能力，使它不再只是简单写入标题和描述，而是可以统一处理：

- `title`
- `description`
- `canonical`
- `og:url`
- `og:image`
- `og:type`
- `twitter:image`
- `robots`
- 页面级 JSON-LD 结构化数据

这样做的价值很明显：页面层不需要分别处理多个 SEO 片段，而是可以把它们作为一组页面级元信息统一交给 composable。

### 3. 为核心页面补齐 canonical 与结构化数据

W6 为以下页面补齐了基础 SEO 信息：

- 首页
- 简历页
- 项目列表页
- 项目详情页

同时，为这些页面增加了基础结构化数据：

- 首页：`WebSite`
- 简历页：`ProfilePage`
- 项目列表页：`CollectionPage`
- 项目详情页：`CreativeWork`

这样不仅页面标题和描述更完整，搜索引擎也更容易理解每个页面承担的内容角色。

### 4. 提供默认 OG 图资源

W6 新增了 `apps/web/public/og-default.svg`，作为站点默认 Open Graph 图片。这样当页面未配置单独图片时，分享链接仍然可以稳定展示品牌化预览图，而不是依赖空白状态。

这个能力虽然简单，但非常实用，因为它让整个站点在社交分享和预览场景下有了最小可用的统一视觉表达。

### 5. 提供 `robots.txt`

W6 新增了 `apps/web/server/routes/robots.txt.ts`，输出允许抓取规则，并显式指向 `sitemap.xml`。这样爬虫在访问站点时，可以快速知道站点允许被索引的范围，以及 sitemap 的位置。

### 6. 提供 `sitemap.xml`

W6 新增了 `apps/web/server/routes/sitemap.xml.ts`，基于当前站点静态页面和项目详情页动态生成 XML 输出，当前已覆盖：

- `/`
- `/resume`
- `/projects`
- `/projects/resume-platform`
- `/projects/design-system-draft`
- `/projects/i18n-content-hub`

这意味着前台项目详情页已经被纳入站点可发现结构，而不再只是页面内部可跳转的路径。

### 7. 明确 SSR / SSG 策略

W6 在 `apps/web/nuxt.config.ts` 中补充了 `routeRules`，当前策略是：

- 首页、简历页、项目页：`prerender`
- `robots.txt` 与 `sitemap.xml`：`prerender`
- 项目详情页：`swr: 3600`

这个策略的意义在于，站点已经开始明确哪些内容适合静态预渲染，哪些内容适合保留一定的动态更新弹性。

虽然这还不是最终部署策略，但它已经让第一阶段的内容站具备了明确的渲染边界。

## 本地验收结果

### 1. 类型检查

执行通过：

- `pnpm --dir apps/web typecheck`

### 2. 本地服务

执行：

- `pnpm --dir apps/web dev --port 3000`

由于本地 `3000` 端口被占用，Nuxt 自动切换到 `3002`，最终在 `http://localhost:3002` 完成验收。

### 3. 页面 meta 验证

已验证：

- 首页返回 `<title>Fridolph Web</title>`
- 首页返回 canonical：`https://fridolph.com/`
- 首页返回默认 `og:image`：`https://fridolph.com/og-default.svg`
- 首页返回 JSON-LD 结构化数据标记
- 简历页返回 canonical：`https://fridolph.com/resume`
- 简历页返回 `og:type=profile`

### 4. SEO 端点验证

已验证：

- `http://localhost:3002/robots.txt` 可以正常返回
- `http://localhost:3002/sitemap.xml` 可以正常返回
- `http://localhost:3002/og-default.svg` 可以正常返回 `200`

并确认 sitemap 中已经包含首页、简历页、项目页和三个项目详情页的 URL。

## 本阶段结果

W6 完成后，Web 端已经具备以下能力：

- 站点级 SEO 配置
- 页面级 canonical URL
- 页面级 Open Graph / Twitter meta
- 默认 OG 图资源
- `robots.txt`
- `sitemap.xml`
- 页面级结构化数据
- 初步明确的 SSR / SSG 策略

这意味着 Web 端已经从“具备状态系统的公开内容站”，进一步升级为“具备基础搜索友好能力的内容站”。
