# Monorepo 实施蓝图：从架构设计到第一阶段落地

## 前言

这份文档承接上一份《Monorepo 架构设计与重构方案》。如果说上一份文档解决的是“为什么这样设计”和“最终要长成什么样”，那么这一份文档解决的是“接下来到底怎么做”。

它会聚焦以下问题：

- 仓库如何初始化。
- 目录如何一步步搭起来。
- 哪些模块先做，哪些模块后做。
- 第一阶段应该交付什么，避免范围失控。
- 每一层代码应该放在哪里。

这份蓝图的目标不是一次性把所有功能规划到极致，而是给后续编码提供一个稳定、可执行、可拆阶段推进的落地路线。

## 一、实施原则

在真正开始编码之前，我先明确这次实施的四个原则。

### 1. 先建立骨架，再填业务

不要一上来就写页面和表单。第一步应该先把 Monorepo、共享配置、应用边界、类型和内容模型搭好。只有骨架稳定，后面的业务代码才不会反复迁移。

### 2. 先打通最小闭环，再扩展模块

第一阶段不是要一次性做完“个人官网 + 简历系统 + 博客系统 + 用户系统 + 多语言 CMS”。第一阶段只需要打通一个最小可运行闭环：

- 后台可登录。
- 后台可管理用户。
- 后台可管理 i18n 文案。
- 后台可管理简历内容与项目内容。
- 前台可以读取并渲染已发布内容。

### 3. 优先共享模型，不优先共享页面

在 Monorepo 里，真正值得优先共享的是：

- 类型。
- schema。
- DTO。
- 请求层。
- 设计 token。

不是所有组件都需要第一时间共享。页面级组件和高度业务化模块，一开始放在各自应用内更清晰。

### 4. 尽量避免过早独立服务

如果第一阶段就把 API、鉴权、文件服务、发布服务全部独立出来，复杂度会显著上升。当前更合理的方式是先利用 `Nuxt 4` 的服务端能力完成闭环，等边界稳定后再拆。

## 二、第一阶段目标定义

第一阶段的目标不是“完整平台上线”，而是完成一版最小可用平台。

### 第一阶段交付目标

- 建立 `pnpm workspace` Monorepo。
- 建立 `apps/web` 和 `apps/admin`。
- 建立 `packages/types`、`packages/content-schema`、`packages/sdk`、`packages/config`。
- 接入统一 TypeScript 配置与 ESLint 规则。
- 完成基础数据库 schema。
- 完成后台登录与权限守卫。
- 完成用户管理、i18n 管理、简历管理、项目管理的基础 CRUD。
- 完成前台首页、简历页、项目列表页与多语言渲染。

### 第一阶段明确不做的内容

为了避免范围蔓延，以下内容不建议放入第一阶段：

- 博客编辑器。
- 评论系统。
- 消息通知中心。
- 可视化拖拽页面搭建。
- 多租户。
- 审批流引擎。
- 独立 API 微服务拆分。

## 三、推荐实施顺序总览

整个实施过程建议分成七个步骤。

### 步骤一：初始化 Monorepo 基础设施

目标是建立统一的仓库结构与开发约束。

### 步骤二：初始化 `web` 与 `admin`

目标是让两个核心应用都可以启动，并具备统一的基础配置。

### 步骤三：建立共享包

目标是沉淀跨应用共享的类型、schema、请求层和配置层。

### 步骤四：设计数据库与服务端结构

目标是固定核心领域模型与数据访问边界。

### 步骤五：先打通后台最小闭环

目标是让后台具备真实的管理能力。

### 步骤六：建设前台展示站

目标是让前台开始消费结构化内容，而不是本地写死数据。

### 步骤七：收尾与工程化增强

目标是补齐测试、脚本、部署配置和文档。

## 四、仓库初始化方案

### 1. 推荐顶层目录结构

```txt
my-resume-platform/
├─ apps/
│  ├─ web/
│  └─ admin/
├─ packages/
│  ├─ config/
│  ├─ types/
│  ├─ content-schema/
│  ├─ sdk/
│  └─ ui/
├─ docs/
├─ package.json
├─ pnpm-workspace.yaml
├─ tsconfig.base.json
└─ turbo.json
```

### 2. 顶层 `package.json` 建议职责

顶层 `package.json` 不承担业务逻辑，它主要负责：

- workspace 统一脚本。
- 开发工具依赖。
- CI 命令入口。
- 构建、lint、typecheck 的任务编排。

推荐脚本如下：

- `dev:web`
- `dev:admin`
- `build`
- `lint`
- `typecheck`
- `test`

### 3. `pnpm-workspace.yaml` 规划

建议至少纳入以下目录：

- `apps/*`
- `packages/*`

这样未来新增 `api`、`db`、`shared` 等包时，不需要再改整体结构。

## 五、应用初始化方案

### 1. `apps/web` 初始化目标

`web` 应用要先满足以下条件：

- 使用 `Nuxt 4`。
- 使用 `TypeScript`。
- 接入 `Nuxt UI`。
- 具备基础路由和布局。
- 支持多语言切换能力预留。
- 具备统一 SEO 配置入口。

### 2. `apps/admin` 初始化目标

`admin` 应用要先满足以下条件：

- 使用 `Nuxt 4`。
- 使用 `TypeScript`。
- 接入 `Nuxt UI`。
- 具备登录页与受保护布局。
- 支持基础导航。
- 具备路由守卫和权限判断能力预留。

### 3. 为什么前后台都用 Nuxt 4

第一阶段统一技术栈的收益非常高：

- 配置方式一致。
- 目录风格一致。
- composables 使用方式一致。
- 组件心智一致。
- 后续共享包接入方式一致。

这会显著降低第一阶段的维护和重构成本。

## 六、共享包实施顺序

共享包建议按“必要程度”建立，而不是一次性全部做满。

### 1. `packages/config`

这个包应最早建立，因为它为整个仓库提供统一的工程约束。

建议包含：

- `tsconfig` 基础配置。
- ESLint 共享配置。
- 可选的 Tailwind preset。
- 统一的路径别名约定。

### 2. `packages/types`

这个包在第一阶段非常关键，因为它是共享领域语言的基础。

建议先定义这些类型：

- `User`
- `Role`
- `Permission`
- `Locale`
- `TranslationNamespace`
- `TranslationKey`
- `TranslationValue`
- `ResumeProfile`
- `Project`
- `PublishStatus`

### 3. `packages/content-schema`

这个包应在 `types` 之后建立，用于承接输入校验与内容约束。

建议先实现：

- 登录表单 schema。
- 用户创建 schema。
- 翻译项编辑 schema。
- 简历条目 schema。
- 项目条目 schema。

如果采用 `zod`，这一层也可以顺带产出前后端共享 DTO。

### 4. `packages/sdk`

这个包在后台 CRUD 和前台数据拉取阶段最有价值。

建议第一批接口封装为：

- `auth`
- `users`
- `translations`
- `resume`
- `projects`
- `site-settings`

### 5. `packages/ui`

这个包建议晚一点建立，不要在第一周就强推所有组件共享。

第一阶段只需要共享以下低风险内容：

- 通用页面容器。
- 通用搜索栏。
- 通用筛选工具条。
- 通用空状态与错误状态。
- 通用卡片与表单节区块。

## 七、数据库与领域模型实施顺序

### 1. 第一批必须落地的表

我建议第一阶段只先做最核心的一组表：

- `users`
- `roles`
- `permissions`
- `user_roles`
- `role_permissions`
- `locales`
- `translation_namespaces`
- `translation_keys`
- `translation_values`
- `resume_profiles`
- `resume_profile_translations`
- `projects`
- `project_translations`
- `site_settings`

### 2. 可以暂缓的表

这些表在第一阶段不是必须：

- `articles`
- `article_translations`
- `media_assets`
- `audit_logs`
- `publish_jobs`
- `notifications`

### 3. 表结构设计原则

设计数据库时，需要优先遵循以下原则：

- 先满足内容与权限闭环。
- 一类业务一组主表与翻译表。
- 通用文案与业务内容分开。
- 发布状态放在核心内容模型中，而不是散落在前端逻辑里。
- 尽量让前台只读取稳定的已发布数据。

## 八、服务端结构蓝图

虽然第一阶段不拆独立 API 服务，但服务端代码仍然要分层。

### 1. 推荐目录结构

可以放在 `apps/admin/server/` 或未来单独拆到 `apps/api/`。第一阶段建议如下：

```txt
server/
├─ api/
│  ├─ auth/
│  ├─ users/
│  ├─ translations/
│  ├─ resumes/
│  ├─ projects/
│  └─ settings/
├─ services/
├─ repositories/
├─ utils/
└─ middleware/
```

### 2. 每层职责

#### API Route 层

负责：

- 接收请求。
- 参数校验。
- 调用 service。
- 返回统一响应格式。

#### Service 层

负责：

- 业务规则。
- 权限判断。
- 发布逻辑。
- 跨表写入流程。

#### Repository 层

负责：

- 数据库查询。
- 插入、更新、删除。
- 组合查询结果。

这种分层方式可以避免 Route 文件越来越肥，也方便后续把 Service 或 Repository 抽到共享服务中。

## 九、后台第一阶段模块实施顺序

后台是第一阶段的核心，因为它决定内容系统是否成立。

### 1. 登录与权限守卫

这是后台最优先的模块。

第一阶段应完成：

- 登录页。
- 登录态存储。
- 管理端基础中间件。
- 基于角色或权限的页面访问控制。

### 2. 用户管理

用户管理模块应尽量先简单可用。

建议第一版支持：

- 用户列表。
- 创建用户。
- 启用或禁用用户。
- 分配角色。
- 查看权限集合。

### 3. 国际化文案管理

这是整个系统非常关键的模块。

建议第一版支持：

- 按 namespace 查看文案。
- 按语言筛选文案。
- 编辑单条文案。
- 查看翻译缺失情况。
- 标记草稿与发布状态。

### 4. 简历管理

建议先支持最核心的内容：

- 基本信息。
- 教育经历。
- 工作经历。
- 技能。
- 联系方式。

如果时间有限，兴趣爱好和扩展信息可以第二阶段补。

### 5. 项目管理

建议第一版支持：

- 项目列表。
- 项目排序。
- 项目外链。
- 技术栈标签。
- 多语言说明。
- 发布状态。

## 十、前台第一阶段模块实施顺序

前台不建议一开始就做太多页面，而是先完成“能消费后台内容”的最小展示闭环。

### 1. 首页

首页可以包含：

- Hero 区域。
- 简要介绍。
- 精选项目。
- 联系方式入口。

### 2. 简历页

简历页是原项目最重要的内容沉淀，应该优先接通。

建议由后台返回结构化内容，再由前台进行组件化渲染。

### 3. 项目列表页

项目列表页用于验证：

- 多语言项目内容。
- 列表查询。
- 标签展示。
- 已发布内容的公开渲染。

### 4. 主题与多语言

第一阶段前台要保留两类基础能力：

- 主题切换。
- 多语言切换。

但这两者都不应该再像原项目那样散落在页面内，而是通过 composables 和统一配置驱动。

## 十一、推荐页面与模块边界

### 1. `web` 路由层边界

页面组件应尽量保持为“组合层”，不要把复杂业务逻辑直接堆在页面里。

例如：

- `pages/resume.vue` 负责页面级 SEO、数据请求和区块组合。
- `components/resume/ResumeSection.vue` 负责章节渲染。
- `components/resume/ExperienceList.vue` 负责经历列表。

### 2. `admin` 页面层边界

管理后台的页面也应遵循相同原则。

例如：

- `pages/translations/index.vue` 负责路由入口和查询参数。
- `components/translation/TranslationTable.vue` 负责表格显示。
- `components/translation/TranslationEditorDrawer.vue` 负责编辑界面。

### 3. composables 边界

推荐在第一阶段就抽离以下 composables：

- `useAuth`
- `usePermissions`
- `useLocale`
- `useTheme`
- `usePagination`
- `useTableFilters`
- `usePublishStatus`

这些能力会被多个模块重复使用，适合尽早沉淀。

## 十二、接口设计建议

为了让后续接入更稳定，第一阶段就应该建立统一接口风格。

### 1. 路径风格建议

推荐风格如下：

- `/api/auth/login`
- `/api/users`
- `/api/users/:id`
- `/api/translations`
- `/api/translations/:id`
- `/api/resumes`
- `/api/projects`
- `/api/site-settings`

### 2. 响应风格建议

建议统一响应结构，至少包含：

- `data`
- `message`
- `meta`

如果出现错误，也建议保持稳定结构，方便前台统一处理。

### 3. 查询能力建议

列表接口建议尽早统一以下查询参数：

- `page`
- `pageSize`
- `keyword`
- `status`
- `locale`
- `sort`

这样后台表格与前台列表都更容易复用查询逻辑。

## 十三、工程化实施清单

### 1. 类型检查

每个应用和共享包都应具备可独立运行的类型检查能力。

### 2. Lint

建议统一到仓库级命令，避免每个包各自维护一套风格。

### 3. 环境变量

从第一阶段开始就应区分：

- 本地开发环境。
- 测试环境。
- 生产环境。

至少需要提前规划这些环境变量：

- 数据库连接。
- 鉴权密钥。
- 默认语言。
- 站点域名。
- 存储服务配置。

### 4. 脚本

建议尽早统一这些脚本：

- 安装依赖。
- 启动前台。
- 启动后台。
- 数据库迁移。
- 数据库 seed。
- lint。
- typecheck。

## 十四、第一阶段里程碑建议

为了让推进更稳定，我建议第一阶段拆成四个里程碑。

### 里程碑一：工程骨架完成

完成条件：

- Monorepo 初始化完成。
- `web` 和 `admin` 可运行。
- `packages/config`、`packages/types` 已建立。
- 基础 lint/typecheck 可执行。

### 里程碑二：数据层完成

完成条件：

- 数据库 schema 初版完成。
- 迁移脚本可运行。
- 用户、翻译、简历、项目的基础 repository 和 service 建立。

### 里程碑三：后台闭环完成

完成条件：

- 可以登录后台。
- 可以管理用户。
- 可以管理翻译文案。
- 可以管理简历与项目内容。

### 里程碑四：前台闭环完成

完成条件：

- 前台能展示结构化主页。
- 前台能展示简历页。
- 前台能展示项目列表页。
- 前台支持多语言读取已发布数据。

## 十五、建议的开发顺序清单

如果进入实际编码，我建议严格按下面的顺序推进：

1. 初始化 `pnpm workspace`。
2. 创建 `apps/web`。
3. 创建 `apps/admin`。
4. 建立 `packages/config`。
5. 建立 `packages/types`。
6. 建立 `packages/content-schema`。
7. 配置统一 lint/typecheck。
8. 引入数据库与 ORM。
9. 建立用户与权限 schema。
10. 建立翻译与内容 schema。
11. 实现后台鉴权。
12. 实现后台用户管理。
13. 实现后台 i18n 管理。
14. 实现后台简历管理。
15. 实现后台项目管理。
16. 实现前台首页。
17. 实现前台简历页。
18. 实现前台项目页。
19. 接入发布态数据读取。
20. 收尾并补文档。

## 十六、第一阶段编码策略建议

为了减少返工，我建议编码时遵循以下策略：

### 1. 每个模块先做最小数据闭环

不要先做很复杂的 UI。先保证“能创建、能读取、能更新、能展示”。

### 2. 不要过早抽象所有组件

很多共享组件只有在使用两次以上、且边界稳定时才值得抽出来。

### 3. 页面只做编排，逻辑尽早下沉

一旦某个页面开始出现分页、过滤、状态同步、权限判断，就应尽快把这部分逻辑抽到 composables 或服务层。

### 4. 尽量让 schema 成为单一事实来源

表单校验、接口校验、服务端入参校验应尽量共享同一套 schema，避免规则分叉。

## 十七、风险点与规避建议

### 1. 风险：第一阶段范围过大

规避方式：

- 严格遵守 MVP 范围。
- 博客、媒体库、审计日志全部延后。

### 2. 风险：过早抽象共享 UI

规避方式：

- 优先共享类型、schema、sdk。
- 共享 UI 从低风险通用组件开始。

### 3. 风险：把后台表单结构直接暴露给前台

规避方式：

- 增加 service 层或公开 DTO 层。
- 前台只消费稳定输出结构。

### 4. 风险：i18n 与业务内容混在一个模型里

规避方式：

- UI 文案和业务内容分开建模。
- 翻译表和业务翻译表分开设计。

## 十八、实施完成后的预期结果

当第一阶段完成后，项目应具备以下能力：

- 具备标准 Monorepo 结构。
- 具备两个独立 Nuxt 应用。
- 具备共享类型与 schema。
- 具备后台登录和权限控制。
- 具备文案国际化管理能力。
- 具备简历和项目内容管理能力。
- 具备前台公开展示能力。
- 具备继续扩展博客、媒体库和更多模块的基础。

## 结语

架构设计解决的是方向问题，实施蓝图解决的是节奏问题。对于这次重构来说，最重要的不是某个单点技术，而是用合适的顺序把系统搭起来。

只要按“骨架优先、闭环优先、共享模型优先、范围受控”这四个原则推进，这个项目就可以从一个单页简历稳步演进成一个真正可维护、可扩展、可运营的内容平台。

这份蓝图确认后，下一步最适合进入的是编码前准备阶段，包括：

- 输出第一版目录脚手架。
- 初始化 Monorepo。
- 建立 `web` 与 `admin`。
- 搭建共享配置与类型包。
