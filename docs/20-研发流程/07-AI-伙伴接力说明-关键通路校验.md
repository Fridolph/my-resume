# AI 伙伴接力说明：关键通路校验

这份文档写给进入本仓库协作的 AI 伙伴。

它不负责解释全部背景，
只负责让你在当前仓库里更快接住节奏，
并避免在公开仓库里引入不必要的外部术语或私有表达。

---

## 1. 先看清你在接什么仓

这是一个正在渐进重构中的教学型 `monorepo` 项目。

当前默认节奏不是“一次做满”，而是：

```text
里程碑
-> Issue
-> Plan
-> TDD / 测试设计
-> 实现
-> Review
-> 自测
-> 关键通路校验
-> review-ready
-> 日志 / PR / 合并
```

你在这里的首要职责不是“多做一点”，
而是：

- 看清当前 Issue 真正在承诺什么
- 控制范围，不扩张
- 保证交付可解释、可验证、可回滚

---

## 2. 进入任务的最短顺序

接新任务时，建议至少先看：

1. 仓库根目录 `AGENTS.md`
2. 当前 Issue / 相关开发日志
3. `docs/20-研发流程/06-review-ready-前关键通路校验.md`
4. 与当前任务直接相关的测试和模块文件

如果要接持久化、发布流、联调收束这类任务，
不要只看代码文件，
还要先看对应开发日志里的：

- 本次目标
- 非目标
- TDD / 测试设计
- Review 记录
- 自测结果

---

## 3. 你在公开仓库里应该怎么说

本仓库默认使用公开、中性的工程语言。

因此在以下产物里：

- 文档
- Issue
- PR
- commit message
- 交接说明

统一使用这些表达：

- 本轮主承诺
- 关键通路
- 校验证据
- 是否进入 `review-ready`

不要主动引入：

- 未公开的方法论名称
- 私有协作框架名称
- 只在内部语境里成立的比喻和隐喻

如果你心里有自己的工作模型，请把它翻译成仓库内可公开使用的工程语言后再落文档。

---

## 4. 什么时候要做关键通路校验

以下任务，默认应该补关键通路校验：

- 服务从一种实现切到另一种实现
- 持久化改造
- 发布流 / 状态流 / 权限流改造
- 跨多个模块的联调收束
- 表面测试会过、但主线容易假通的任务

以下任务可不强制：

- 很小的样式修正
- 单文件文案修改
- 非主线的微型重构

---

## 5. 最小交接格式

如果你要给下一个 AI 或人类协作者交接，至少交清：

```text
目标 / 状态 / 下一步 / 校验结果 / 文档锚点
```

最短句式：

```text
【接棒】我接的是 X，我下一步先做 Y。
```

如果当前已进入 `review-ready` 前夜，
建议再补一段更聚焦的话：

```text
本轮主承诺是 X，关键通路已校验到 Y，当前仍缺 Z。
```
