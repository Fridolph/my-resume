# 从单页简历到内容平台：我的 Monorepo 架构设计与重构方案

## 前言

这份文档既是一篇重构记录，也是一份正式的架构设计说明。它的目标不是简单地把原项目从 `Vite + Vue 3` 迁移到 `Nuxt 4`，而是把一个单页在线简历项目升级成一个可持续演进的内容平台。

新的平台会包含两个核心应用：

- 面向访客的前端展示站。
- 面向运营与维护的管理后台。

在这个平台里，我可以自由扩展模块、维护不同语言的内容、统一管理用户和权限，并让前台展示与后台管理共享同一套领域模型。

## 一、为什么要重构

### 1. 原项目已经完成了它的阶段性使命

原项目是一份很典型的前端在线简历站。它基于 `Vue 3 + TypeScript + Vite + TailwindCSS + vue-i18n + Pinia` 实现，结构清晰，展示效果也足够完整。

从当前代码来看，它具备以下特征：

- 使用单入口页面进行内容编排。
- 通过多个 `views` 模块组织简历章节。
- 提供主题切换、语言切换、锚点导航、下载 PDF 等能力。
- 以组件复用的方式抽象了部分结构，例如 `Section`、`ListWrap`、`ProjectWrap`。

这个项目在“个人在线简历”这个目标下是成立的，也已经完成了展示和练手的价值。

### 2. 原项目的限制也开始明显暴露

随着项目目标从“展示一份简历”变成“长期维护一个个人内容平台”，原架构的边界开始变得明显。

#### 内容与页面结构耦合

当前项目的内容大多直接写在视图组件中。这样做在项目很小时非常直接，但一旦需要管理多语言、多版本简历、多类型页面，维护成本就会快速上升。

#### 缺少真正的后台管理能力

目前项目可以展示内容，但不能管理内容。每次修改文案、增删项目、更新经历，仍然需要改代码、重新构建、重新发布。

#### 国际化仍然偏静态

当前的国际化方案更适合小型展示页。文案以本地文件为主，缺少后台录入、审核、发布、版本控制等能力，不适合后续扩展成真正的内容系统。

#### 状态与模块边界不够稳定

项目体量小的时候，这不是问题。但如果继续叠加博客、项目详情、配置管理、用户体系、权限控制，这种结构很快会失去可维护性。

### 3. 这次重构的目标不是“升级技术”，而是“升级产品形态”

我这次重构的核心目标有三点：

- 把单页简历升级成前后台分离的内容平台。
- 把静态文案升级成可管理、可扩展、可国际化的结构化内容。
- 把单仓库应用升级成适合长期演进的 Monorepo 工程。

## 二、原项目现状分析

### 1. 技术栈现状

原项目使用的主要技术如下：

- `Vue 3`
- `TypeScript`
- `Vite`
- `TailwindCSS`
- `Pinia`
- `vue-i18n`

它是一个轻量的前端展示项目，运行时依赖并不复杂，适合快速开发与发布。

### 2. 当前结构特点

原项目采用“组件层 + 视图层”的组织方式：

- `components/`：通用展示组件。
- `views/`：简历的各个内容章节。
- `hooks/`：少量组合式逻辑。
- `locales/`：本地多语言文案。
- `store/`：全局状态。

这种方式适合单页项目，但它仍然以页面展示为中心，而不是以领域模型和内容管理为中心。

### 3. 当前最值得保留的部分

虽然架构会重做，但原项目有几部分思路值得保留：

- 简历内容章节划分清晰。
- 组件抽象已经有雏形。
- 视觉风格较稳定。
- 展示型页面目标明确，没有引入过多无关复杂度。

因此，新的架构不会否定原项目，而是把它的展示经验迁移到一个更可持续的工程体系里。

## 三、重构后的目标形态

重构后的项目不再只是“一个简历网页”，而是一个围绕个人内容展示与后台运营构建的平台。

### 1. 面向外部的前端展示站

前端站点承担以下职责：

- 展示个人主页与个人品牌信息。
- 展示简历、项目、文章、作品和联系方式。
- 提供 SEO、站点地图、Open Graph、分享图等能力。
- 支持多语言切换。
- 支持结构化内容渲染，而不是把内容写死在组件里。

### 2. 面向内部的管理后台

管理后台承担以下职责：

- 管理用户与权限。
- 管理国际化文案。
- 管理简历内容、项目内容、文章内容。
- 管理站点配置，例如站点标题、社交链接、默认语言、下载资源。
- 管理内容发布状态与版本记录。

### 3. 共享领域模型与基础设施

前台与后台会共享以下基础设施：

- TypeScript 类型。
- 内容 Schema。
- API DTO。
- 权限定义。
- 工具函数。
- 统一的组件和设计约束。

## 四、为什么选择 Monorepo

我选择 Monorepo，不是为了追求“看起来高级”，而是为了控制复杂度。

### 1. 前后台本质上是一个产品的两个界面

展示站和管理后台虽然界面不同，但它们服务的是同一套内容与同一套业务模型。把它们分散到多个仓库，会造成以下问题：

- 类型重复定义。
- 接口契约分散。
- 权限定义不一致。
- 维护成本上升。

Monorepo 可以把这些共享资产放在统一位置。

### 2. 内容平台天然适合共享包

这个项目后续会非常依赖共享包，例如：

- `types`：共享领域类型。
- `content-schema`：共享校验规则。
- `sdk`：共享 API 调用封装。
- `ui`：共享业务组件与设计令牌。

如果不采用 Monorepo，这些内容很快会变成复制粘贴的负担。

### 3. Monorepo 有助于分阶段演进

我可以先完成 `web`，再补 `admin`，再沉淀共享包，而不需要反复拆仓库、迁移代码、同步依赖。

## 五、技术选型与原因

这一部分是本次重构最关键的决策说明。

### 1. 为什么前端展示站选择 Nuxt 4

前端展示站需要 SEO、SSR、页面级路由、良好的内容渲染体验和后续扩展能力。`Nuxt 4` 很适合这个方向。

当然熟悉 React 的同学用 Next.js 完全可以，我选只是因为我 Vue3 用顺手了。
选择 `Nuxt 4` 的其他原因如下：

- 天然支持 SSR 与静态生成
- 路由、布局、服务端能力、插件体系完整
- 对内容站、博客站、展示站非常友好
- 后续无论接内容系统还是鉴权系统，都有明确扩展路径

### 2. 为什么 UI 层选择 Nuxt UI

`Nuxt UI` 可以帮助我快速搭建一套统一、可维护、可主题化的界面体系。

选择它的原因如下：

- 与 Nuxt 生态集成紧密。（官方支持最好）
- 组件成熟，适合做后台表格、表单、抽屉、弹窗、导航和卡片布局
- 可以同时服务展示站和管理后台
- 能减少大量基础组件的重复开发

### 3. 为什么后台也建议使用 Nuxt 4

后台并不一定非要使用独立的 Node 框架。我本来想用 Next.js 来学习，但想了下，确实没必要。
对于当前阶段，后台同样使用 `Nuxt 4` 有明显优势：

- 技术栈统一，降低后续维护的心智负担
- 组件和 composables 可以复用（monorepo架构的优势发挥到最大）
- 管理后台并不需要特别复杂的高并发后端架构
- `Nuxt server routes` 足以支撑第一阶段的 API 能力

如果未来业务复杂度继续上升，再将 API 拆成独立服务也很自然。

### 4. 为什么使用 TypeScript 作为统一语言

这个项目会有多个应用、多个共享包、多个数据模型。`TypeScript` 可以保证以下几点：

- 类型在前台、后台、服务端之间共享
- 内容模型与接口定义一致
- 重构时更容易发现破坏性变更
- 管理后台的表单和接口联调更稳定

### 5. 为什么使用 PostgreSQL

这个项目会涉及用户、权限、内容、翻译、版本、发布状态等结构化数据。关系型数据库比纯文档型数据库更适合当前阶段。

当然，MySQL、mongoDB这些都可以，选择自己熟悉的就好，

`PostgreSQL` 的优势在于：

- 结构稳定，适合复杂关系
- 事务能力好
- 生态成熟
- 后续扩展全文检索、JSON 字段、审计记录也较方便

### 6. 为什么推荐 Drizzle ORM

我希望数据库层足够透明、类型友好，并且适合 Monorepo 共享 schema。

同上，选择自己熟悉的或者想学习的练手就好，`Drizzle ORM` 非常适合这个目标。

它的优势包括：

- 类型推导自然
- Schema 与 SQL 的距离较近
- 适合共享到多个应用和包里
- 对长期维护比较友好

### 7. 为什么国际化改成数据库驱动

原项目的本地 `locales/*.ts` 方案适合静态站点，但不适合“后台管理文案”。

数据库驱动的国际化方案可以支持：

- 在线编辑与审核
- 多语言覆盖率检查
- 发布态与草稿态管理
- 按模块、命名空间、页面维度管理文案

## 八、核心领域模型设计

这次重构最重要的事情，不是页面，而是领域建模。

### 1. 用户与权限

后台必须具备完整的用户与权限模型。

推荐的表包括：

- `users`
- `roles`
- `permissions`
- `user_roles`
- `role_permissions`

推荐权限命名方式：

- `user.read`
- `user.write`
- `translation.read`
- `translation.write`
- `resume.read`
- `resume.write`
- `project.read`
- `project.write`
- `site_config.write`
- `publish.execute`

这种命名清晰、可扩展，也方便在前端做权限判断。

### 2. 国际化文案模型

国际化不要再只靠本地文件，而是改成数据库管理。

推荐拆分为以下几张表：

- `locales`
- `translation_namespaces`
- `translation_keys`
- `translation_values`

推荐字段示意：

#### `locales`

- `id`
- `code`，例如 `zh-CN`、`en-US`
- `name`
- `is_enabled`

#### `translation_namespaces`

- `id`
- `name`，例如 `common`、`resume`、`project`、`seo`

#### `translation_keys`

- `id`
- `namespace_id`
- `key`，例如 `hero.title`
- `description`

#### `translation_values`

- `id`
- `translation_key_id`
- `locale_id`
- `value`
- `status`，例如 `draft`、`published`

### 3. 业务内容国际化模型

通用文案与业务内容应该分开。

推荐业务表如下：

- `resume_profiles`
- `resume_profile_translations`
- `projects`
- `project_translations`
- `articles`
- `article_translations`

这样可以做到：

- UI 文案由翻译系统管理
- 业务内容由业务模块管理
- 每类内容都支持多语言
- 每类内容都可单独校验、保存、发布

### 4. 站点配置模型

建议增加 `site_settings` 表，统一管理站点级配置。

它可以包含：

- 站点标题
- 默认语言
- 社交账号链接
- 下载文件地址
- 首页开关项
- SEO 默认描述

## 九、内容发布与数据流设计

### 1. 内容生命周期

为了让后台真正可用，内容需要具备状态。

推荐状态如下：

- `draft`
- `reviewing`
- `published`
- `archived`

### 2. 数据流建议

内容流转建议如下：

1. 后台编辑内容
2. 保存为草稿
3. 可选地进入审核
4. 发布后写入正式状态
5. 前台只消费已发布内容

### 3. 前台读取策略

前台不直接依赖“后台表单结构”，而是依赖稳定的公开内容接口或服务层

这样可以避免后台调整字段时直接影响展示层

## 十、API 分层建议

在第一阶段，我建议优先采用 `Nuxt server routes` 实现 API，而不是一开始就拆独立后端

### 1. 第一阶段 API 位置

推荐直接放在应用内：

- `apps/admin/server/api/*`
- 或者抽到未来的 `apps/api-server`

### 2. API 分层原则

建议至少分三层：

- Route 层：负责请求入参和响应格式。
- Service 层：负责业务逻辑。
- Repository 层：负责数据库读写。

### 3. 共享契约

所有 API 入参、出参类型，尽量从 `packages/content-schema` 和 `packages/types` 导出，避免应用内重复定义。

## 十一、前端展示站的页面规划

新的前端展示站不应再是单页拼装，而应是结构化站点。

- 管理教育经历
- 管理技能与标签
- 管理不同岗位版本

> 要具备可扩展性，比如投不同公司，不同行业，能让招聘方看到不同的版本（这个后续有时间再实现）

### 4. 项目管理

支持以下能力：

- 项目列表与排序
- 项目技术栈
- 项目封面图与外链
- 多语言项目说明

### 5. 站点设置

支持以下能力：

## 十三、组件与 Composable 设计原则

为了避免重构后再次回到“大组件堆叠”的状态，这次需要明确组件边界。

### 1. 路由页面只负责编排

无论是 `web` 还是 `admin`，页面组件都应该尽量薄。页面负责：

- 拉取数据
- 组合模块
- 处理页面级路由和 SEO

页面不应该承担复杂展示细节和可复用逻辑。

> 这里有些纸上谈兵，就简历而言SEO做不到哪去，仅供学习

### 2. 复杂逻辑下沉到 composables

例如以下逻辑适合抽到 composables：

- 语言切换
- 主题切换
- 列表筛选与分页
- 发布流程
- 权限判断

### 3. 展示组件保持单一职责

例如：

- `ResumeSection` 只负责展示简历章节
- `ProjectCard` 只负责展示项目卡片
- `TranslationEditor` 只负责编辑翻译项

## 十四、部署与演进策略

### 1. 第一阶段先求可用，不求过度拆分

我建议第一阶段保持以下策略：

- 使用 `pnpm workspace`
- 先建设 `web` 和 `admin`
- API 先内置在 Nuxt 服务端
- 数据库只覆盖核心领域

### 2. 第二阶段再做独立化

如果后续需要，可以逐步演进：

- 抽独立 `api` 应用
- 引入任务队列
- 引入审计日志
- 引入更完整的发布流程
- 对接对象存储与 CDN

> 心动吗？到这里就是一套完整的 monorepo 全栈项目了。后续我们还可以接入AI接着玩

## 十五、推荐的实施顺序

为了控制风险，我建议按以下顺序推进。

### 阶段一：搭建 Monorepo 基础设施

- 初始化 `pnpm workspace`。
- 创建 `apps/web` 与 `apps/admin`。
- 创建 `packages/types`、`packages/content-schema`、`packages/sdk`、`packages/config`。
- 统一 TypeScript 和 ESLint 配置。

### 阶段二：确定数据库与领域模型

- 设计用户、权限、翻译、简历、项目、站点配置表
- 建立 Drizzle schema
- 设计内容发布状态

### 阶段三：先打通后台最小闭环

- 登录
- 用户与角色
- 国际化文案管理
- 简历内容管理
- 项目内容管理

### 阶段四：建设新版前台展示站

- 主页
- 简历页
- 项目页
- 多语言与 SEO
- 对接后台正式数据

## 十六、MVP 范围建议

如果要尽快落地，建议第一版 MVP 只做以下内容：

- `web`：主页、简历页、项目列表页
- `admin`：登录、用户管理、i18n 管理、简历管理、项目管理
- 数据库：用户、角色、翻译、简历、项目、站点配置

不要在第一版就同时做：

- 博客系统
- 评论系统
- 复杂审核流
- 多租户
- 可视化拖拽编辑器

这些能力都可以等核心平台稳定后再扩展。是的，锦上添花的后面来，先让我们的博客能运行起来

## 十七、这次重构的价值

这次重构带来的价值，不只是“代码结构更漂亮”

它真正带来的提升包括：

- 从静态简历升级到可运营的内容平台
- 从手工改代码升级到后台管理内容
- 从本地静态 i18n 升级到系统化的翻译管理
- 从单应用升级到可长期演进的 Monorepo
- 从页面驱动开发升级到领域模型驱动开发

## 结语

原项目是一个很好的起点，因为它已经验证了内容展示、视觉风格和基础交互方向。现在要做的，不是推翻它，而是把它提升成一个真正可长期维护的产品架构。

接下来的实现阶段，我会以以下原则推进：

- 先稳定领域模型，再搭页面
- 先形成最小闭环，再做复杂扩展
- 先统一前后台边界，再沉淀共享包
- 先让内容可管理，再追求高级功能
