# Admin A2：登录与权限框架建立记录

## 背景

A1 完成后，管理后台已经具备了独立运行的工程骨架，但它仍然只是一个“可以打开的后台壳”。如果没有登录态、权限判断和路由守卫，后续用户管理、i18n 文案管理和内容管理模块都缺少稳定边界。

因此 A2 的目标不是做业务 CRUD，而是先把后台的访问控制框架搭起来。

## 本阶段目标

A2 主要解决以下问题：

- 后台未登录用户不能直接访问受保护页面。
- 后台需要有一个独立登录页。
- 登录后需要有持久化的登录态。
- 页面需要具备基础权限判断能力。
- 不同角色访问不同页面时，需要有明确的跳转策略。
- 退出登录流程需要明确可用。

## 方案选择

### 1. 为什么先用 mock 登录而不直接接后端

当前仓库还没有进入真正的用户系统和接口实现阶段。如果在 A2 就直接接数据库和真实鉴权接口，会把认证问题和数据层问题耦合在一起。

这会带来两个问题：

- 很难独立验证权限框架本身是否合理。
- 会把 A2 的边界拖进 A3 用户管理阶段。

因此本阶段选择了一个更稳妥的实现：先用本地 mock 账号完成前端登录闭环，把认证状态、路由守卫和权限判断全部跑通。等后续进入真实接口阶段，再将登录实现替换为后端服务即可。

### 2. 为什么登录态选择 Cookie + `useState`

后台需要兼顾页面跳转和 SSR 场景下的路由中间件判断，因此这次登录态没有只放在纯客户端状态里，而是采用：

- `useCookie` 持久化会话
- `useState` 提供应用内共享状态

这样处理的好处是：

- 页面刷新后可以恢复登录态。
- 服务端渲染时也能参与权限判断。
- 路由中间件可以直接基于当前会话做重定向。

## 本次实现

### 1. 建立 `useAuth`

A2 新增了 `useAuth` 组合式逻辑，用于统一处理：

- mock 账号定义
- 登录逻辑
- 会话构建
- 登录态存储
- 退出登录

目前定义了三类演示角色：

- `admin`
- `editor`
- `translator`

这样可以在没有后端接口的情况下，先验证不同角色对应的权限边界。

### 2. 建立 `usePermissions`

A2 新增了 `usePermissions`，把权限判断从页面逻辑中抽离出来。当前它主要提供：

- 当前权限集合读取
- `hasPermission`
- `requirePermission`

这个抽象很重要，因为后续无论是菜单显隐、按钮权限还是页面访问判断，都不应该散落在模板里硬编码。

### 3. 建立路由中间件

本阶段新增了两个中间件：

- `auth`：保护需要登录才能访问的页面
- `guest`：限制已登录用户重复访问登录页

这样处理后：

- 未登录访问后台首页会被重定向到 `/login`
- 已登录访问 `/login` 会回到首页

### 4. 建立登录页与未授权页

A2 新增了：

- `/login`
- `/unauthorized`

登录页用于承接 mock 登录闭环，未授权页用于承接“已登录但权限不足”的场景。这比直接显示空白页或返回 404 更清晰，也更符合后台管理端的交互预期。

### 5. 建立受保护页面样例

为了验证权限框架不是纸面设计，A2 还新增了几个实际受保护页面：

- `/`
- `/users`
- `/translations`
- `/settings`

其中：

- `/users` 需要 `user.read`
- `/translations` 需要 `translation.read`
- `/settings` 需要 `site.write`

这可以直接验证角色与权限是否按预期生效。

## 本地验收结果

本阶段不仅通过了类型检查，还实际跑通了本地服务与权限流。

### 1. 类型检查

执行通过：

- `pnpm --dir apps/admin typecheck`

### 2. 本地服务

执行：

- `pnpm --dir apps/admin dev --port 3001`

本地后台服务已成功启动。

### 3. HTTP 验证

已验证以下场景：

- `/login` 返回 `200 OK`
- 未登录访问 `/` 返回 `302`，跳转到 `/login?redirect=/`
- 携带管理员会话访问 `/` 返回 `200 OK`
- 携带管理员会话访问 `/users` 返回 `200 OK`
- 携带翻译角色会话访问 `/settings` 返回 `302`，跳转到 `/unauthorized`

这说明 A2 已经完成了从“页面骨架”到“权限骨架”的升级。

## 当前结果

A2 完成后，后台已经具备以下能力：

- 可独立访问登录页
- 可持久化登录态
- 可保护后台页面
- 可按权限控制页面访问
- 可处理未授权访问
- 可执行退出登录
