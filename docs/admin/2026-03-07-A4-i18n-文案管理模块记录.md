# Admin A4：i18n 文案管理模块记录

## 背景

A3 完成后，后台已经具备了第一个业务管理模块，但内容平台的核心能力还缺一块：文案管理。如果不能管理导航文案、页面标题、SEO 描述和业务模块标题，那么后续多语言展示就仍然会停留在硬编码阶段。

因此 A4 的目标是先把 i18n 文案管理模块的形态跑通。

## 本阶段目标

A4 主要解决以下问题：

- 建立文案列表页面。
- 按 namespace、locale 和状态筛选文案。
- 识别缺失翻译项。
- 展示不同语言的覆盖情况。
- 提供文案编辑入口。
- 提供草稿与发布状态切换。

## 方案选择

### 1. 为什么 A4 先做“翻译记录管理”，而不是直接做远程 i18n 接口

当前项目虽然已经进入后台模块阶段，但仍然处在前后台整体架构的早期重构期。如果 A4 一开始就深度绑定后端翻译接口和数据库表，会让页面结构、状态流转和接口契约同时变化，验证成本很高。

因此 A4 先选择本地 mock 翻译记录，重点验证：

- 翻译模型长什么样
- 页面筛选维度是否合理
- 缺失翻译如何表达
- 发布状态如何管理
- 编辑区应该怎样组织

等这些边界稳定后，再切到真实接口会更稳。

### 2. 为什么 namespace / locale / status 是首批筛选维度

一个真实的文案管理系统最先需要解决的，不是“怎么批量编辑”，而是“怎么快速找到要改的文案”。

因此 A4 先固定三个关键筛选维度：

- `namespace`
- `locale`
- `status`

这三个维度足够支撑第一版翻译管理体验，也能直接映射到后续数据库模型。

## 本次实现

### 1. 扩展共享翻译类型

A4 在共享类型中新增了：

- `TranslationNamespace`
- `TranslationRecord`

这一步意味着翻译管理不再只是一个页面概念，而是开始拥有自己的领域模型。

### 2. 建立 `useTranslationManagement`

A4 新增了 `useTranslationManagement`，用于统一处理：

- mock 翻译记录
- 关键字搜索
- namespace 筛选
- locale 筛选
- status 筛选
- 缺失项统计
- 覆盖率统计
- 编辑区状态
- 发布状态切换

这让文案管理页从一开始就具备了稳定的状态组织方式，而不是把所有逻辑堆在页面里。

### 3. 建立文案管理页面

A4 重写了 `/translations` 页面。当前页面已经具备：

- 顶部说明区
- 文案统计卡片
- 查询与筛选区
- 语言覆盖率区
- 文案列表区
- 编辑区

这套结构本身已经能支撑一个可用的第一版翻译管理页。

### 4. 展示缺失翻译与覆盖率

A4 特别加入了两种对内容平台很有价值的表达：

- 单条文案的“缺失”标记
- 按语言统计的覆盖率卡片

这样后续无论是内容运营还是翻译补齐，都有明确的可视反馈，而不是只能逐条翻找。

### 5. 建立发布状态切换

A4 支持文案在本地状态中进行：

- `draft`
- `reviewing`
- `published`
- `archived`

的表达和切换。虽然现在还没有完整的发布流，但状态模型已经先稳定下来，后续接真实接口时就不会重复改页面表达。

### 2. 本地服务

已验证以下场景：

- 携带管理员会话访问 `/translations` 返回 `200 OK`
- 页面中已返回 A4 文案管理模块关键文案

这说明 A4 的文案管理页已经具备实际可展示、可筛选、可编辑的模块形态。
