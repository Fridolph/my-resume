# Admin A1：Nuxt 4 管理后台工程骨架初始化记录

## 背景

这次重构的目标不是单纯重写前端页面，而是构建一个“展示站 + 管理后台”的内容平台。后台是这个平台的管理入口，因此需要先建立稳定的工程骨架。

A1 的任务不是做业务功能，而是把管理后台作为一个独立、可扩展的应用搭起来。

## 本阶段目标

A1 聚焦后台工程基础能力，主要完成：

- 创建 `apps/admin`
- 初始化 `Nuxt 4`
- 接入 `TypeScript`
- 接入 `Nuxt UI`
- 接入根级 Monorepo
- 建立默认后台布局
- 建立后台首页骨架
- 验证基础类型检查

## 为什么 A1 不直接做登录

登录和权限框架会直接影响中间件、布局、导航和接口约定。如果在 A1 阶段就急着写登录页，很容易让“工程骨架”和“权限边界”相互耦合。

因此本阶段只做后台壳：

- 应用可启动
- 基础布局存在
- 后台首页存在
- 可继续承接 A2 的权限设计

## 本次实现

### 1. 创建 `apps/admin`

后台应用同样基于 `Nuxt UI` 官方模板初始化，但随后移除了模板中与当前业务无关的宣传内容、组件和文件。

这样做的好处是可以快速获得稳定的 `Nuxt + Nuxt UI` 基础集成，同时避免把模板示例误当作业务结构延续下去。

### 2. 接入 Monorepo

`admin` 已纳入根级 `pnpm workspace`，不再保留应用内独立锁文件和局部 workspace 配置。

这样后续接入共享包时，后台可以直接消费：

- `@repo/types`
- `@repo/content-schema`
- `@repo/sdk`
- `@repo/ui`

### 3. 建立后台首页骨架

A1 首页没有直接实现用户表格或权限模块，而是保留了三个核心业务方向的占位：

- 用户管理
- 文案管理
- 内容管理

这使后台首页从一开始就围绕真实的产品职责组织，而不是延续模板营销页。

### 4. 为 A2 做边界预留

当前骨架已经具备后续接入以下能力的入口：

- 登录页
- 后台布局
- 中间件
- 权限判断 composable
- 用户与角色模块

这意味着 A2 可以直接在既有结构上扩展，而不必回头重做 A1。

## 验证结果

本阶段完成后，执行了以下验证：

- `pnpm install`
- `pnpm --dir apps/admin typecheck`

类型检查已通过，说明后台骨架可继续承接 A2 的实现。

## 当前结果

A1 完成后，`admin` 已具备以下状态：

- 可以独立作为 `Nuxt 4` 管理后台启动
- 可以纳入根级 Monorepo 管理
- 具备基础后台布局与首页入口
- 具备承接权限系统、用户管理和文案管理模块的结构基础
