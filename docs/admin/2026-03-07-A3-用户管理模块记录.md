# Admin A3：用户管理模块记录

## 背景

A2 解决了后台“谁能进来、谁能访问什么页面”的问题，但后台仍然缺少第一个真正的业务模块。为了验证管理端页面如何组织、列表模块如何设计、角色和状态如何表达，A3 选择先落地用户管理模块。

这个阶段的重点不是接数据库，而是把后台业务模块的基础形态跑通。

## 本阶段目标

A3 主要解决以下问题：

- 建立后台第一个真实的业务管理页面。
- 提供用户列表和管理入口。
- 支持查询与筛选。
- 支持创建与编辑。
- 支持角色切换与状态切换。
- 展示用户权限集合。

## 方案选择

### 1. 为什么 A3 仍然先用本地 mock 数据

当前后台还没有进入真正的 API 和数据库阶段。如果 A3 一开始就强依赖后端接口，用户管理模块会和数据层建设耦合在一起。

这样会导致两个问题：

- 无法单独验证管理端页面结构是否合理。
- 业务模块开发节奏会被数据库与接口设计拖慢。

因此 A3 继续沿用本地 mock 数据的策略，把重点放在：

- 列表查询区如何组织
- 用户卡片如何表达
- 表单如何承接创建与编辑
- 角色与状态如何更新

等这些边界稳定后，再切换到真实接口会更顺。

### 2. 为什么 A3 先做“列表型模块”

后台的大部分业务模块，本质上都会长成类似的结构：

- 统计区
- 查询区
- 列表区
- 编辑区
- 操作反馈

用户管理是最适合先验证这套模式的模块，因为它天然具备角色、状态、权限和编辑需求。A3 先把这套页面形态跑通，后面的文案管理、项目管理和简历管理就可以复用相同思路。

## 本次实现

### 1. 扩展共享用户类型

A3 在共享类型里新增了：

- `UserStatus`
- `UserRecord`

这样后台用户页面不再只依赖登录态类型，而是开始拥有独立的用户管理领域模型。

### 2. 建立 `useUserManagement`

A3 新增了 `useUserManagement` 组合式逻辑，用于统一处理：

- mock 用户数据源
- 关键字筛选
- 角色筛选
- 状态筛选
- 表单状态
- 创建/编辑保存
- 角色切换
- 启用/禁用切换

这一步很关键，因为它把列表模块的“状态管理”从页面模板里抽离了出来，后续接入 API 时，只需要替换数据来源，不必推翻整个页面结构。

### 3. 重建用户列表页

A3 重写了 `/users` 页面，使它从 A2 的占位提示页升级为真正的用户管理页面。

当前页面已经具备：

- 顶部说明区
- 用户统计卡片
- 查询筛选区
- 创建/编辑表单区
- 用户卡片列表
- 权限集合展示区
- 角色切换操作区

### 4. 角色与状态更新

A3 支持两种高频后台操作：

- 启用 / 禁用用户
- 直接切换用户角色

虽然当前仍然是本地状态驱动，但这已经足够验证交互是否顺手、页面结构是否清晰，以及“角色变化会带来权限集合变化”这一点是否表达明确。

### 5. 保持权限边界不变

A3 没有绕过 A2 建立的认证和权限体系，而是在其基础上继续扩展。

当前：

- `/users` 仍然必须具备 `user.read`
- 写操作入口仍然要求 `user.write`

这说明 A2 和 A3 之间的边界是衔接稳定的，而不是彼此推翻。

## 本地验收结果

### 1. 类型检查

执行通过：

- `pnpm --dir apps/admin typecheck`

### 2. 本地服务

执行并验证：

- `pnpm --dir apps/admin dev --port 3001`

### 3. HTTP 验证

已验证以下场景：

- 携带管理员会话访问 `/users` 返回 `200 OK`
- 页面中已返回用户管理模块关键文案
- 页面中已返回统计区、查询区和权限标签内容

这说明 A3 的用户管理模块已经不是占位结构，而是一个实际可展示的后台业务页。

## 当前结果

A3 完成后，后台已经具备以下能力：

- 用户列表展示
- 用户筛选
- 用户创建与编辑入口
- 启用/禁用切换
- 角色切换
- 权限集合展示
