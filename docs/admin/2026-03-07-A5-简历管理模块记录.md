# Admin A5：简历管理模块记录

## 背景

A4 完成后，后台已经具备了用户管理和文案管理能力，但简历内容仍然缺少真正的后台维护入口。虽然前台 `W4` 和 `W5` 已经开始走向结构化内容和多语言切换，但这些能力如果没有后台数据管理承接，最终仍然会卡在“前台能展示、后台不能维护”的状态。

因此，A5 的核心目标很明确：把简历内容正式收口到后台管理模块中，让基础信息、教育经历、工作经历、技能分组和联系方式都可以被统一维护，并且为后续接入真实 API 做好结构准备。

## 本阶段目标

A5 聚焦以下问题：

- 设计可扩展的简历内容结构。
- 建立基础信息管理界面。
- 建立教育经历、工作经历管理界面。
- 建立技能分组和联系方式管理界面。
- 建立多语言简历编辑方式。
- 建立简历发布状态表达方式。

这一阶段完成后，后台就不再只管理用户和翻译，而是开始真正承接前台公开内容。

## 为什么 A5 先采用统一简历文档模型

A5 没有把简历拆成很多分散的小状态，而是先建立一个统一的简历文档模型。这么做的原因很实际。

如果现在就把简历内容拆成多个彼此独立的 store 或多个临时接口，后续一旦需要增加版本管理、发布流、审核流和多语言同步，就会很难维护边界。相反，统一文档模型更适合作为后台内容管理的第一步。

这次实现中，简历文档统一包含：

- 文档级发布状态
- 文档级更新时间
- 多语言内容集合
- 每种语言下的基础信息、教育经历、工作经历、技能分组和联系方式

这样做以后，无论后面接数据库、接版本系统，还是让前台直接消费简历 DTO，都有比较稳定的承接面。

## 本次实现

### 1. 在共享类型中定义简历模型

A5 在 `packages/types` 中新增了简历管理所需的结构定义，包括：

- `ResumeBaseInfo`
- `ResumeEducationItem`
- `ResumeExperienceItem`
- `ResumeSkillGroup`
- `ResumeContactItem`
- `ResumeLocaleContent`
- `ResumeDocument`

这些类型不是单纯为了后台页面服务，它们同时也在为前后台共享契约做准备。后续如果 Web 端要直接消费后台的简历内容，或者 SDK 要从后台接口拿数据，都会直接受益于这层结构化定义。

### 2. 建立 `useResumeManagement` composable

A5 新增了 `apps/admin/app/composables/useResumeManagement.ts`，把简历管理的主要状态和行为都集中到一个 composable 中。它负责：

- 管理简历文档状态
- 管理当前编辑语言
- 管理发布状态
- 提供新增 / 删除教育经历、工作经历、技能分组和联系方式的方法
- 计算多语言覆盖情况
- 维护更新时间

这样做的价值很明确：页面不需要自己散落地处理所有数组状态和业务操作，而是把“简历怎么被管理”收口到统一逻辑层。

### 3. 建立简历管理页面

A5 新增了 `apps/admin/app/pages/resume/index.vue`，作为简历管理的主页面。页面中包含以下分区：

- 顶部模块说明和当前发布状态
- 多语言与发布状态区
- 基础信息区
- 教育经历区
- 工作经历区
- 技能分组区
- 联系方式区

这个页面继续沿用前面几个后台模块的设计风格：

- 用 `UCard` 分区
- 用表单字段承载输入
- 用显式按钮承载新增、删除和保存操作
- 用 `toast` 表达操作反馈

这样可以让后台模块体验保持一致，不会每新增一个模块就换一套交互方式。

### 4. 建立多语言简历编辑方式

A5 的多语言方案不是给每个字段单独塞翻译键，而是以 locale 为维度切换整个简历内容块。当前支持：

- `zh-CN`
- `en-US`

用户在后台切换语言后，会直接编辑对应 locale 下的完整简历内容，包括基础信息和各个章节数组。

这种方式的好处是很清楚：后台编辑者面对的是“某个语言版本的完整简历”，而不是分散的翻译碎片。对于简历这种强结构化但又具备整体语义的内容，这种管理方式比逐字段翻译更自然。

### 5. 建立发布状态管理方式

A5 继续复用已有的 `PublishStatus` 模型，把简历文档作为一个整体进行状态管理。当前支持：

- `draft`
- `reviewing`
- `published`
- `archived`

这一步很重要，因为它意味着简历内容不只是“能编辑”，还已经具备了进入内容工作流的雏形。后续如果要接版本记录、审核人或发布时间字段，都可以在这个基础上继续扩展。

### 6. 更新后台导航与首页入口

A5 完成后，后台导航已经增加了“简历”入口，后台首页的快捷入口也同步补上了简历管理卡片。

这意味着简历管理已经不再是一个孤立实验页，而是正式进入后台第一阶段的模块体系中。

## 多语言与发布状态边界

A5 一个很重要的产出，不只是表单页面，而是边界确认。

当前后台对简历内容采用如下分层：

- 文档级：发布时间、更新时间、发布状态
- 语言级：`zh-CN` / `en-US` 的完整简历内容
- 区块级：基础信息、教育经历、工作经历、技能分组、联系方式

这种分层方式让后续演进更顺畅：

- 可以继续增加更多 locale
- 可以继续增加简历章节
- 可以在文档级增加版本信息
- 可以在语言级增加翻译覆盖率校验

也就是说，A5 不是只完成了一个页面，而是把后台简历内容的演化方向先固定下来了。

## 本地验收结果

### 1. 类型检查

执行通过：

- `pnpm --dir apps/admin typecheck`

### 2. 本地服务

执行：

- `pnpm --dir apps/admin dev --port 3001`

由于本地 `3001` 端口被占用，Nuxt 自动切换到 `3002`，最终在 `http://localhost:3002` 完成验收。

### 3. 鉴权与页面访问验证

已验证：

- `/login` 可以正常返回 `200`
- 未登录访问 `/resume` 会被重定向到 `/login?redirect=/resume`
- 注入管理员 session cookie 后，`/resume` 页面可以正常访问

并确认页面已返回以下关键内容：

- `A5`
- `简历管理`
- `多语言与发布状态`
- `基础信息`
- `教育经历`
- `工作经历`
- `技能分组`
- `联系方式`

这说明 A5 页面已经真正进入后台路由和权限链路，而不是停留在孤立模板阶段。
