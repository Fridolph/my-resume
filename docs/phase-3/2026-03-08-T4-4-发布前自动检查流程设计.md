# T4-4：发布前自动检查流程设计

## 背景

`T4-1` 到 `T4-3` 已经先后补上了：

- Web 冒烟
- Admin 冒烟
- API 集成测试

到这个阶段，如果还没有一个统一的发布前检查入口，那么这些测试能力仍然是“分散存在”的：

- 有人知道怎么跑 Web 冒烟
- 有人知道怎么跑 Admin 冒烟
- 有人知道怎么跑 API 集成测试

但在真正准备发布或回归验证时，团队仍然需要自己拼接执行顺序。

所以 `T4-4` 的目标，是把当前第三阶段已经具备的这些检查能力，整理成一套更适合发布前执行的标准流程。

## 本轮目标

本轮主要完成两件事：

- 补齐统一检查脚本入口
- 整理一份发布前检查流程文档

## 设计原则

### 1. 先做可执行的最小检查链路

当前阶段不追求一次性覆盖 lint、构建产物分析、截图对比等所有动作，而是先把已经成熟的检查能力串起来。

### 2. 以当前三端真实链路为中心

优先覆盖：

- API 集成测试
- Web E2E 冒烟
- Admin E2E 冒烟

### 3. 同时保留文档化步骤

脚本负责“执行”，文档负责“解释什么时候跑、按什么顺序跑、看到失败后怎么理解”。

## 建议实现

## 1. 在根目录补统一检查脚本

建议新增：

- `check:api`
- `check:web`
- `check:admin`
- `check:phase3`

其中 `check:phase3` 负责顺序执行当前最关键的三条检查链路。

## 2. 整理发布前检查文档

文档应明确：

- 当前推荐执行顺序
- 每条命令的验证目标
- 本地运行前需要哪些服务或依赖
- 失败时优先排查哪里

## 完成标准

`T4-4` 完成后，应满足：

- 当前仓库拥有统一的发布前检查脚本入口
- 第三阶段已经形成一套清晰的本地发布前检查流程
- `T4` 质量体系扩展可以正式收口
