# T3-2：版本记录展示与差异理解能力记录

## 背景

`T3-1` 已经先把统一发布中心的信息密度和状态反馈补强了一轮，但第三阶段的“后台协作与运营体验增强”并不只发生在发布页。

对于当前后台来说，更高频、更贴近日常操作的其实是各内容模块自己的版本历史区块，也就是：

- 文案版本
- 简历版本
- 项目版本

第二阶段我们已经把版本保存、版本恢复和真实持久化链路打通了，但从实际使用体验看，这些版本区块仍然更偏“底层可用”，还不够“快速可理解”。

用户虽然能看到：

- 有多少个历史版本
- 每个版本的状态
- 谁在什么时候创建了版本

但很难第一眼判断：

- 这个历史版本和当前内容是不是一致
- 它主要改动了什么
- 恢复后最可能影响哪几个关键字段

所以 `T3-2` 的重点，不是继续扩充版本机制，而是让已有版本能力更容易被理解和使用。

## 本次实现

这一轮围绕“版本理解成本”做了三件事：统一差异摘要方式、补齐关键字段提示、降低恢复前判断成本。

### 1. 抽出统一的版本理解 composable

这次新增了：

- `apps/admin/app/composables/useContentVersionInsights.ts`

这个 composable 不负责保存版本，也不负责恢复版本，而是专门负责把历史快照和当前内容做轻量对比，并生成适合页面展示的结构化结果，包括：

- 是否有差异
- 差异提示标签
- 摘要信息项
- 当前与历史的关键字段对照

这样做的好处是，三个页面不需要各自零散写一套版本判断逻辑，后续如果继续补版本摘要规则，也可以沿着这一层统一收口。

### 2. 文案版本增加“与当前内容相比”的理解提示

在文案模块里，这次补上了更适合恢复前判断的信息：

- 与当前内容一致 / 有差异
- 文案内容是否不同
- 发布状态是否不同
- 历史文案长度与当前文案长度对比
- 历史文案与当前文案并排展示

这一点很关键，因为文案内容通常字段短、数量多、修改频繁。

如果页面只显示版本号和创建时间，用户每次都需要自己点进去读快照，判断成本会非常高。现在版本卡片已经能先回答一个核心问题：

> 恢复这个版本后，当前文案值会不会真的发生变化。

### 3. 简历版本增加结构级摘要

简历模块的快照结构比文案更复杂，这次没有引入重型 diff，而是优先做最有帮助的结构摘要：

- 当前语言标题是否一致
- 教育经历数量
- 工作经历数量
- 技能组数量
- 联系方式数量
- 历史标题与当前标题对比

这类信息虽然不是逐字段 diff，但已经足够帮助用户快速判断：

- 这更像一次小修订，还是结构变化
- 恢复后会不会改动当前简历的主要结构

尤其在运营或协作场景里，这种“结构级提示”比单纯堆快照字段更有价值。

### 4. 项目版本增加内容摘要与差异标签

项目模块这次补充的重点是：

- 当前语言标题是否一致
- 标签数量
- 排序值
- 外链状态
- 封面状态
- 历史标题与当前标题对比

对于项目管理来说，很多恢复操作其实不是为了改正文案，而是为了恢复：

- 排序
- 标签集合
- 外链配置
- 封面信息

所以项目版本区块的摘要信息必须覆盖这些字段，才能真正帮助用户在恢复前做判断。

### 5. 三个模块的版本理解方式开始统一

这次实现后，文案、简历、项目三个页面的版本区块虽然数据结构不同，但在认知模式上已经趋于一致：

- 先看版本元信息
- 再看与当前内容是否一致
- 再看高亮差异项
- 再看摘要卡片
- 最后看历史值与当前值对比，再决定是否恢复

这意味着后台版本系统开始从“有历史记录”升级为“可辅助判断的历史记录”。

### 浏览器级联调验证

本轮继续使用 Playwright CLI 做了真实浏览器联调，重点验证了：

- 使用 `root@fridolph.local / root123` 登录后台
- 进入 `/translations` 后，打开版本区块可看到：
  - 与当前内容一致 / 有差异的 badge
  - 差异高亮项
  - 历史状态、文案长度摘要
  - 历史文案 / 当前文案对照
- 进入 `/resume` 后，版本区块可看到：
  - 与当前简历结构一致 / 有差异
  - 教育、工作经历、技能组、联系方式摘要
  - 历史标题 / 当前标题对照
- 进入 `/projects` 后，编辑项目可看到：
  - 与当前项目结构一致 / 有差异
  - 标签数量、排序值、外链状态等摘要
  - 历史标题 / 当前标题对照

联调过程中还顺手确认了一个非常实际的问题：

- 当管理后台使用 `127.0.0.1`，而 API 使用 `localhost` 时，真实登录态 cookie 会因为 host 不一致导致接口表现异常。

所以本次联调统一使用：

- `http://localhost:3002`

来访问后台，确保会话 cookie 与 API 请求链路一致。

## 本次收获

`T3-2` 很能说明第三阶段的一个方向：

- 第二阶段关注“能力打通”
- 第三阶段关注“能力是否容易理解、是否适合持续使用”

版本系统在第二阶段已经成立，但如果用户每次恢复前都要先阅读底层快照，版本历史的价值其实并没有完全发挥出来。

这次补上轻量差异摘要之后，后台版本区块已经能更直接地回答几个真正重要的问题：

- 当前和历史是否一致
- 差异主要集中在哪些字段
- 恢复后大概率会改动什么

这让版本恢复从“谨慎试错”开始往“有依据地判断”靠近了一步。
