# T3-1：发布中心信息密度与状态反馈设计

## 背景

第三阶段当前临时调整了执行顺序。

由于当前阶段重点仍然是把本地三端链路持续打通、持续完善，并不急着立刻落正式线上环境，因此：

- `T1`：正式部署落地
- `T2`：运行稳定性与可观测性

先保留为后续实际上线前再补的里程碑。

在这个前提下，第三阶段当前最适合优先推进的是：

- `T3`：后台协作与运营体验增强

而 `T3` 的第一步，建议先从当前最关键、最贴近主链路的模块开始：

- 统一发布中心

因为当前系统虽然已经具备：

- 统一发布批次创建
- 批次激活
- 公开站点读取激活批次

但后台当前的发布中心还更偏“基础可用”，离“真正好用”还有一段距离。

具体来说，当前页面已经能完成操作，但仍然存在几个明显问题：

- 信息密度还不够高，用户要在多个卡片中自己拼装理解上下文
- 当前激活批次和历史批次之间的对比关系不够直观
- 状态反馈还不够强，哪些内容会对公开站点生效没有形成足够清晰的感知
- 缺少更明确的“发布影响范围”提示

所以 `T3-1` 的目标，不是新增新的发布机制，而是先把当前已有发布中心的“可理解性”和“可操作反馈”做得更好。

## 本轮目标

本轮聚焦以下四个方向：

- 提升发布中心页面的信息密度
- 提升当前激活批次与历史批次的区分度
- 增强对公开站点影响范围的反馈
- 让关键状态与操作结果更直观

本轮暂时不做：

- 复杂的多级审批流
- 批次差异比对算法
- 多人协作锁定机制
- 发布排期与定时发布

也就是说，`T3-1` 先解决“当前发布中心更好理解、更好使用”，而不是继续扩大发布机制本身。

## 当前问题分析

### 1. 当前激活批次信息虽然存在，但可读性一般

当前页面已经有“当前激活批次”卡片，但展示内容仍然偏基础字段堆叠。

它能说明：

- 名称
- 状态
- 版本数量
- 时间

但还不够突出：

- 它为什么重要
- 它当前究竟影响哪些公开内容
- 用户切换批次时会改变什么

### 2. 历史批次之间缺少更明显的比较线索

目前每个历史批次以卡片形式展示，但用户仍然需要逐个阅读字段，才能知道：

- 哪个是激活批次
- 哪个是历史批次
- 各批次覆盖了多少内容
- 哪个批次更适合作为回切候选

这对发布中心来说，认知负担偏高。

### 3. 状态反馈仍偏操作后 toast，而缺少页面内反馈

当前操作反馈主要依赖：

- toast 成功提示
- badge 状态标签

这对“动作是否成功”已经足够，但对“动作结果会影响什么”还不够。

例如切换激活批次后，页面里应该尽量更明确地反馈：

- 公开站点将读取该批次
- 当前批次包含简历 / 文案 / 项目数量
- 当前页展示已经刷新到最新状态

### 4. 页面还缺少更统一的摘要区

发布中心天然适合有一个更高层摘要区，用来快速回答：

- 当前有多少发布批次
- 当前激活批次是哪一个
- 当前公开站点覆盖了多少文案 / 项目 / 简历版本
- 最近一次激活时间是什么时候

这类摘要信息不需要很重，但非常适合提升页面的扫描效率。

## 设计原则

### 1. 不改变发布机制，只增强认知与反馈

本轮不引入新的工作流，只在现有批次模型上增强页面表现。

### 2. 先增强摘要与对比，再考虑更复杂的 diff 能力

当前阶段优先做：

- 摘要信息
- 结构化卡片信息
- 当前批次高亮
- 影响范围反馈

不急着做复杂版本差异比对。

### 3. 强调“当前公开站点正在消费什么”

发布中心页面的核心应该始终围绕一个问题展开：

> 当前公开站点到底正在消费哪一批内容。

所以页面设计要尽量帮助用户快速回答这个问题。

## 建议实现方向

## 1. 增加页面级摘要区

建议在页面顶部补一个摘要区，展示例如：

- 总批次数
- 当前激活批次名称
- 当前公开简历版本
- 当前公开项目数量
- 当前公开文案数量
- 最近激活时间

## 2. 强化“当前激活批次”视觉层级

建议：

- 当前激活批次卡片在页面上更突出
- 增加更明确的说明文案
- 强化“当前公开站点正在消费它”的提示

## 3. 统一历史批次卡片信息结构

建议让每个历史批次都统一展示：

- 批次名称
- 状态
- 简历版本号
- 项目数
- 文案数
- 创建人 / 激活人
- 创建时间 / 更新时间

并让“激活该批次”按钮的语义更明确。

## 4. 增强页面内状态反馈

例如：

- 操作后刷新当前摘要区
- 切换成功后页面中明确反映当前激活批次变化
- 当批次已处于激活状态时，给出清晰的不可操作反馈

## 5. 视情况增加轻量提示文案

例如页面顶部提示：

- 当前切换会影响 Web 公开内容读取
- 批次切换不会自动创建内容，只是切换公开读取目标

## 完成标准

`T3-1` 完成后，应满足：

- 发布中心页面信息密度明显提升
- 当前激活批次与历史批次关系更清晰
- 页面内状态反馈比现在更直观
- 用户更容易判断当前公开站点正在消费哪一批内容
