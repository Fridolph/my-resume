# T4-4：发布前自动检查流程记录

## 背景

当 `T4-1`、`T4-2`、`T4-3` 分别补上 Web 冒烟、Admin 冒烟和 API 集成测试之后，第三阶段的质量体系已经具备了比较完整的“能力”。

但如果这些能力没有被整理成一套统一的发布前检查流程，那么它们仍然只是：

- 知道的人会手动跑
- 不知道的人不知道先跑哪个
- 发布前检查没有一个统一入口

所以 `T4-4` 的核心目的，是把前面已经做好的质量能力收成一套真正能执行的流程。

## 本次实现

### 1. 在根目录补了统一检查脚本入口

这次在仓库根目录新增了：

- `check:api`
- `check:web`
- `check:admin`
- `check:phase3`

其中：

- `check:api` 负责执行 `apps/api-server` 的集成测试
- `check:web` 负责执行 `apps/web` 的 Playwright 冒烟
- `check:admin` 负责执行 `apps/admin` 的 Playwright 冒烟
- `check:phase3` 负责按顺序串起当前第三阶段最关键的三条质量链路

这让第三阶段首次拥有了一个统一的本地发布前检查入口。

### 2. 把当前推荐检查顺序真正固定下来

当前推荐顺序已经通过脚本收口为：

1. 先跑 API 集成测试
2. 再跑 Web 冒烟
3. 最后跑 Admin 冒烟

这个顺序非常合理，因为它遵循了依赖方向：

- 先确认后端接口层稳定
- 再确认公开展示端
- 最后确认后台管理端

这样如果链路出错，也更容易定位问题是从哪一层开始出现的。

### 3. 验证了整条第三阶段检查链路可以一条命令跑通

这一轮实际执行并通过了：

- `pnpm check:phase3`

这说明当前第三阶段已经不只是“零散地有测试”，而是已经具备：

- 一条命令
- 一套顺序
- 一次性覆盖三端关键质量入口

从流程角度看，这就是第三阶段质量体系最重要的一次收口。

## 本次自测

### 聚合检查命令

以下命令已通过：

- `pnpm check:phase3`

### 聚合检查覆盖范围

当前这条命令已经覆盖：

- API 8 条集成测试
- Web 4 条关键页面 E2E 冒烟
- Admin 4 条关键页面 E2E 冒烟

也就是说，当前第三阶段已经至少具备了：

- 16 条自动化验证
- 三端关键链路统一检查入口

## 本次收获

`T4-4` 的价值，并不是“多加了几个 script”，而是它真正把第三阶段的质量能力从“分散的成果”变成了“可执行的流程”。

这会直接影响后续协作体验：

- 发布前知道先跑什么
- 新人接手时知道当前质量底线在哪里
- 后续如果继续接 CI，也已经有明确可复用的命令入口

换句话说，`T4` 到这里已经可以正式收口，因为它不再只是补测试，而是已经形成了一个阶段性可复用的质量流程。
