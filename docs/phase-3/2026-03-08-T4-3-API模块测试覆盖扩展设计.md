# T4-3：API 模块测试覆盖扩展设计

## 背景

前两步 `T4-1` 和 `T4-2` 已经把 Web / Admin 的浏览器级 smoke 建立起来了。接下来 `T4-3` 需要补上的，就是 API Server 本身的模块级验证。

当前仓库已经存在一条 API 集成测试，覆盖了：

- 登录
- 受保护路由拦截
- 简历状态流转
- 简历版本恢复
- 统一发布批次创建与激活

但从模块覆盖角度看，还明显偏向：

- Auth / Resume / Release

而下面两个关键内容模块还缺少更明确的测试收口：

- Translation
n- Project

同时，角色权限边界目前虽然在运行中已经生效，但缺少足够明确的自动化验证。

## 本轮目标

`T4-3` 重点扩三块：

- 权限边界测试
- Translation 模块测试
- Project 模块测试

## 设计原则

### 1. 继续沿用集成测试，不拆成过多单测

当前 API 最重要的是模块协作是否正常，因此优先继续使用现有 `Vitest + Nest Test Application + supertest` 方式，验证真实接口链路。

### 2. 优先补业务价值最高的模块路径

本轮优先覆盖：

- 登录后不同角色访问受限接口
- 翻译更新、版本、公开发布链路
- 项目创建、状态流转、公开发布链路

### 3. 尽量让测试覆盖“模块闭环”

不仅验证接口能返回数据，还要尽量覆盖：

- 写入
- 版本变化
- 发布生效
- 公开接口读取结果

## 建议实现

## 1. 补角色权限边界测试

建议新增：

- translator 无法访问用户管理接口
- viewer 无法写入翻译内容

## 2. 补 Translation 模块测试

建议新增：

- 翻译内容更新后产生新版本
- 重新发布后，公开翻译接口可读到最新内容

## 3. 补 Project 模块测试

建议新增：

- 创建项目
- 项目状态从 draft -> reviewing -> published
- 发布后公开项目接口可读取到新项目
- 删除后重新发布，公开项目接口不再返回该项目

## 完成标准

`T4-3` 完成后，应满足：

- API 测试覆盖从 Resume / Release 扩展到 Translation / Project
- 权限边界具备明确自动化验证
- 公开内容链路在接口层也有可重复执行的校验

`T4-3` 完成后，下一步进入：
