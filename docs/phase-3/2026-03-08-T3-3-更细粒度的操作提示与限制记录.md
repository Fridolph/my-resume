# T3-3：更细粒度的操作提示与限制记录

## 背景

`T3-1` 解决了“发布页信息不够集中”的问题，`T3-2` 解决了“版本历史不够容易理解”的问题。

但第三阶段要让后台真正更适合协作使用，还需要再往前走一步，也就是把“操作边界”说清楚。

因为当后台已经接入真实 API、真实会话和真实数据库之后，页面里很多按钮已经不再是演示性质的交互，而是真正会影响当前内容状态的动作，例如：

- 恢复历史版本
- 切换发布状态
- 删除项目
- 创建并激活新发布批次
- 切换当前公开批次

如果这些动作缺少更细粒度的提示和限制，后台虽然功能完整，但仍然容易让用户在不完全理解后果的情况下继续操作。

所以 `T3-3` 的重点是：

- 把只读边界提示得更明确
- 把无意义操作前置限制掉
- 把危险操作的确认文案补上

## 本次实现

### 1. 抽出统一的操作提示 composable

这次新增了：

- `apps/admin/app/composables/useOperationGuidance.ts`

这个 composable 统一封装了几类页面级能力：

- 只读模式提示文案
- 轻量确认操作
- 版本恢复限制说明
- 项目排序边界说明

这样做的价值不只是在 `T3-3`，更重要的是它把“操作边界”从分散页面逻辑里抽了出来。后续如果后台继续扩模块，也可以沿着这一层复用。

### 2. 为内容页补上只读模式提示

这次在以下页面增加了页面级只读提示：

- 文案管理
- 简历管理
- 项目管理
- 统一发布中心

当账号没有写权限时，页面会明确告诉用户：

- 当前是只读模式
- 仍然可以查看内容、状态和版本信息
- 但不能执行保存、恢复、删除、发布或切换操作

这一点对协作非常重要，因为权限控制不应该只依赖“按钮没了”，而应该让用户明确知道自己当前的能力边界。

### 3. 为恢复操作补上禁用态和原因说明

这次把版本恢复按钮的逻辑进一步收紧了。

对于：

- 文案版本
- 简历版本
- 项目版本

如果某个历史版本与当前内容一致，则页面现在会：

- 禁用“恢复此版本”按钮
- 明确提示“无需重复恢复”

这样一来，后台不再只是允许“恢复”，而是开始帮助用户判断“这个恢复动作有没有意义”。

这和 `T3-2` 的差异摘要是连起来的：

- `T3-2` 负责让用户看懂差异
- `T3-3` 负责把无意义恢复直接拦在操作前

### 4. 为危险操作增加轻量确认

这次为多个真实写操作补上了确认提示，包括：

- 文案状态切换
- 文案版本恢复
- 简历状态切换
- 简历版本恢复
- 项目状态切换
- 项目删除
- 项目版本恢复
- 创建并激活新发布批次
- 切换当前公开批次

这些确认不追求复杂弹窗流程，而是先用轻量方式把最关键的信息说清楚：

- 将要操作什么
- 当前状态和目标状态是什么
- 操作后会影响什么

这让后台在不明显增加使用负担的前提下，降低了误操作概率。

### 5. 为项目排序动作补上边界限制

项目管理页里，排序操作本来已经是有效的，但之前还缺少更清晰的边界说明。

这次补上之后：

- 第一项会禁用“上移”
- 最后一项会禁用“下移”
- 操作区会给出当前不能移动，或移动后会发生什么的说明

这类细节虽然不大，但它会显著提升页面的“预期一致性”：

- 用户不会先点一下再发现没效果
- 页面会提前告诉用户当前动作为什么不可用

### 浏览器级联调验证

本轮继续使用 Playwright CLI 做真实浏览器联调，重点确认了以下内容：

  - 排序边界提示已经可见

这一轮联调重点验证的，不再只是“功能能不能调用 API”，而是“操作前的限制和提示是否真的在页面中出现”。

## 本次收获

`T3-3` 很像第三阶段的“交互边界收口”步骤。

因为到这个阶段，后台的主要问题已经不再是“有没有功能”，而是：

- 页面有没有把权限和边界解释清楚
- 用户会不会去执行一个没有意义的动作
- 用户在做危险操作前能不能提前意识到影响

这次补完之后，后台开始具备一种更稳定的协作感：

- 有权限边界提示
- 有操作前确认
- 有无意义操作限制
- 有可解释的禁用原因

这会让后台从“可用的功能集合”更进一步，开始接近“适合长期维护的运营后台”。
