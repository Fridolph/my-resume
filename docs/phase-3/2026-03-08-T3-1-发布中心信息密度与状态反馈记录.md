# T3-1：发布中心信息密度与状态反馈记录

## 背景

第三阶段当前先跳过了正式部署与可观测性建设，优先进入：

- `T3`：后台协作与运营体验增强

而 `T3` 的第一步，之所以先从发布中心开始，是因为当前系统最关键的一条业务链路其实就是：

- 后台管理内容
- 创建或切换统一发布批次
- Web 公开站点读取当前激活批次

也就是说，发布中心虽然不是后台里字段最多的页面，但它是最直接连接：

- 内容管理
- 统一发布
- 公开站点生效结果

的页面。

第二阶段已经让这个页面“可用”，第三阶段的 `T3-1` 则是要让它变得“更容易理解、更容易判断、更适合日常使用”。

## 本次实现

这一轮 `T3-1` 没有新增新的发布机制，而是围绕现有发布批次模型，重点增强了页面的信息密度和状态反馈。

### 1. 增加页面级摘要区

这次先补了一组更适合快速扫描的摘要信息，包括：

- 发布批次总数
- 当前激活批次
- 当前公开项目数量
- 当前公开文案数量
- 最近激活时间
- 历史归档批次数量

这一步非常重要，因为发布中心天然不应该只是“批次列表页”，它更应该先回答一个问题：

> 当前公开站点正在用什么，以及当前一共有多少可切换批次。

摘要区补上之后，页面的整体可读性明显提升了。

### 2. 强化当前激活批次的页面层级

这次对“当前激活批次”区域做了显著增强：

- 增加了更明确的高亮卡片
- 明确标记“公开站点正在使用”
- 强调当前激活批次会影响首页、简历页、项目页和公开文案
- 把简历版本、项目数量、文案数量等关键字段集中展示

这让页面的核心焦点从“所有批次平铺”变成了：

- 先看当前生效批次
- 再看历史批次和切换候选

### 3. 增强历史批次卡片的信息结构

这次还统一了历史批次卡片的信息结构，让每张卡片都更容易横向比较：

- 简历版本
- 项目版本数
- 文案版本数
- 最近更新时间
- 创建人 / 激活人
- 创建时间 / 激活时间

同时，按钮语义也做了增强：

- 对未激活批次使用“切换为当前公开批次”
- 对已激活批次显示“当前已生效”并禁用

这样用户在视觉上更容易判断：

- 哪个批次是当前生效的
- 哪个批次可以被切换回来
- 每个批次大概覆盖了什么范围

### 4. 增加页面内提示文案

这次还补了一条非常轻量但很重要的页面内提示：

- 当前批次切换会直接影响 Web 公开内容读取
- 发布中心不会自动创建内容，只会切换公开站点读取目标

这类提示并不复杂，但它对降低误解非常有帮助，尤其是后续如果不是当前作者自己来操作页面时，页面会更不容易被误读。

### 5. 增强操作成功后的反馈语义

之前操作成功主要是通用 toast，这次把 toast 描述也更明确地指向结果：

- 页面摘要已刷新
- 当前激活批次信息已同步更新
- 当前公开站点已切换到所选批次

这让操作结果不再只是“成功了”，而是更明确地说明“成功后页面和公开站点会发生什么”。

## 本次收获

这一轮 `T3-1` 很有代表性，因为它体现了第三阶段和第二阶段的一个明显区别：

- 第二阶段更关注“能力有没有”
- 第三阶段开始关注“能力好不好理解、好不好使用”

统一发布机制在第二阶段已经成立，但如果页面不能帮助用户快速理解：

- 当前站点正在消费哪一批内容
- 切换后会影响什么
- 哪个批次可以作为历史候选

那它仍然会停留在“能用但不够好用”的状态。

这次增强之后，发布中心已经开始从“功能可用页”往“真正的运营控制台页面”靠近了一步。
