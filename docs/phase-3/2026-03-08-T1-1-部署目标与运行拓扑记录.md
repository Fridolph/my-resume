# T1-1：部署目标与运行拓扑记录

## 背景

第三阶段的第一步不是直接部署，而是先把部署主线定下来。

因为当前项目已经不是一个单页面站点，而是一个真实三端结构：

- `web`
- `admin`
- `api-server`
- `SQLite + Drizzle ORM`

这意味着部署平台如果选错，后面一连串工作都会被放大：

- 数据持久化会出问题
- 域名和反向代理关系会混乱
- 部署手册和回滚方案也很难稳定下来

所以 `T1-1` 的价值，不是“做了多少代码”，而是先把第三阶段部署路线正式确定下来。

## 本次调研结论

这次 `T1-1` 结合当前项目约束和平台官方文档，最终得出的结论是：

### 当前阶段首选部署平台：Render

当前阶段推荐把：

- `Render`

作为第三阶段正式部署落地的首选平台。

### 原因一：它与当前三端结构匹配度更高

当前项目不是单一前端应用，而是三服务结构。

Render 更适合这种：

- 多服务并行
- 分服务配置环境变量
- 分服务管理运行状态

的模型。

这让它在当前阶段更容易承载：

- `web`
- `admin`
- `api-server`

的独立部署关系。

### 原因二：它支持当前阶段继续使用 SQLite

这一点其实是最关键的。

当前阶段明确继续坚持：

- `SQLite + Drizzle ORM`

所以部署平台必须允许：

- 数据文件写入持久化路径
- 服务重启和重新部署后仍能保留数据库文件

根据官方文档，Render 提供持久化磁盘，适合把本地文件系统中的数据保存下来。

这与当前项目的 SQLite 方案是兼容的。

### 原因三：它当前阶段的限制是可接受的

Render 的持久化磁盘也有明确限制，例如：

- 带磁盘的服务不能多实例横向扩容
- 带磁盘的部署不适合零停机切换

但这些限制在当前阶段并不是阻碍，因为当前阶段部署目标本来就是：

- 先正式上线
- 先稳定运行
- 先形成可复用部署流程

而不是现在就做高可用分布式架构。

所以这些限制在当前阶段是可以接受的，甚至和当前项目体量是匹配的。

## 备选方案判断

### Railway

Railway 也是可以支持当前项目的一种备选方案，因为它也支持 Volume 持久化。

但在 `T1-1` 这一步，不建议并行保留两条主部署路线。

更好的做法是：

- 当前先确定 Render 为主方案
- Railway 只保留为后续备选参考

这样后续 `T1-2 ~ T1-6` 的实施不会反复摇摆。

### Vercel

Vercel 并不适合作为当前阶段整套三端统一部署的主方案。

不是因为它不能部署 Nuxt，而是因为当前项目明确继续使用：

- `SQLite + Drizzle ORM`

而 Vercel 当前更适合：

- 静态站点
- 无服务器函数
- 不依赖本地文件数据库持久化的架构

所以当前阶段不建议把 Vercel 作为整套主部署平台。

## 本次确定的运行拓扑

`T1-1` 最终确定的推荐拓扑如下：

### 服务划分

- `web`：独立服务
- `admin`：独立服务
- `api-server`：独立服务
- `SQLite`：挂载在 `api-server` 上的持久化磁盘文件

### 域名划分

推荐采用：

- `example.com` 或 `www.example.com` -> `web`
- `admin.example.com` -> `admin`
- `api.example.com` -> `api-server`

### 当前判断

这套结构最大的价值在于：

- Web / Admin / API 的边界清楚
- CORS 关系清楚
- 公开内容与后台管理逻辑不混杂
- 后续部署、日志和监控的服务视角也更清楚

## 本次确定的 SQLite 持久化策略

本轮确定的策略是：

- 仅在 `api-server` 服务挂载持久化磁盘
- 通过 `REPO_DATABASE_PATH` 指向明确持久化路径
- 所有数据库迁移、备份、恢复都围绕这份数据库文件展开

推荐路径形式例如：

- `/var/data/platform.sqlite`

这一步的意义很大，因为它把“数据库到底落在哪里”从抽象概念变成了后续可执行部署动作。

## 对后续任务的影响

`T1-1` 完成以后，后续子任务就有了清晰方向：

- `T1-2`：继续围绕 Render + 三服务拓扑收口环境变量与生产配置
- `T1-3`：优先落地 API Server 与 SQLite 持久化
- `T1-4`：再接 Web / Admin 域名与访问链路
- `T1-5`：整理部署与回滚手册
- `T1-6`：完成部署后正式验收

也就是说，`T1-1` 解决的是一个非常基础但非常关键的问题：

> 后面的部署工作终于有统一方向了。

## 一个阶段结论

如果只看代码量，`T1-1` 可能不是一个“很重”的任务。

但从第三阶段整体推进看，它其实非常关键，因为它完成了：

- 部署平台主线确认
- 三端运行拓扑确认
- 域名边界确认
- SQLite 持久化方向确认

做到这里，第三阶段的部署落地工作终于不再是抽象讨论，而开始进入一条可以继续实施的真实路线。
