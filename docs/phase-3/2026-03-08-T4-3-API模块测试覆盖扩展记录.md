# T4-3：API 模块测试覆盖扩展记录

## 背景

`T4-1` 和 `T4-2` 已经分别把 Web 与 Admin 的浏览器级 smoke 跑起来了。到这里，第三阶段质量体系扩展还差一块最重要的拼图：

- API 模块本身的自动化覆盖

因为当前三端虽然都已经串起来，但如果 API 层只覆盖登录和简历主链路，仍然会出现一个问题：

- 页面能跑，但某些内容模块接口边界没有被系统性验证

所以 `T4-3` 的目标，是把当前 API 的自动化覆盖从“核心主线可用”继续扩展到：

- 权限边界
- Translation 模块
- Project 模块

## 本次实现

### 1. 保持现有集成测试模式，继续扩展真实接口链路

这一轮没有另起测试体系，而是继续沿用当前仓库已经存在的：

- `Vitest`
- `Nest Test Application`
- `supertest`
- 测试数据库隔离

这意味着新增测试不会偏离当前 API Server 的既有测试风格，也能继续覆盖真实的接口调用链路。

### 2. 补上角色权限边界测试

这次新增了角色权限边界验证，重点覆盖：

- `translator` 可以读取翻译内容，但不能访问用户管理接口
- `viewer` 可以读取翻译列表，但不能写入翻译内容

这一步很重要，因为它让当前权限系统不再只是依赖人工验证，而是开始有明确的接口级保障。

换句话说，权限边界现在已经进入自动化测试范围，而不再只是运行时约定。

### 3. 补上 Translation 模块的更新、版本与公开发布链路

这次为 Translation 模块增加了完整的一条链路验证：

- 获取现有翻译记录
- 更新翻译内容
- 检查版本列表新增 update 版本
- 创建新的统一发布批次
- 通过公开翻译接口确认最新内容已经生效

这一轮非常关键，因为它首次把：

- 后台翻译写入
- 版本生成
- 统一发布
- 公开接口读取

完整串成了一条可重复执行的自动化验证路径。

### 4. 补上 Project 模块的创建、状态流转、公开发布与删除链路

这次还为 Project 模块补上了一条更完整的业务链路：

- 创建新项目
- 从 `draft -> reviewing -> published` 状态流转
- 检查项目版本列表
- 创建统一发布批次
- 通过公开项目接口确认项目已对外可见
- 删除项目
- 再次发布并确认公开项目列表已移除该项目

这让 Project 模块的自动化覆盖，不再只停留在“能查列表”，而是已经覆盖到：

- 创建
- 发布
- 对外生效
- 删除
- 再次发布后的公开结果变化

## 本次自测

### API 集成测试命令

以下命令已通过：

- `pnpm --dir apps/api-server test:run`

### 当前通过结果

当前 API 集成测试总数已扩展到 8 条，覆盖：

- 未登录访问保护接口被拦截
- 登录与 session cookie 建立
- 简历状态流转
- 简历版本恢复
- 统一发布批次创建与激活
- 多角色权限边界
- Translation 更新 / 版本 / 发布链路
- Project 创建 / 发布 / 公开读取 / 删除链路

## 本次收获

`T4-3` 的核心价值，是把当前 API 的测试覆盖从“能证明主线可用”继续提升为“能证明各模块真的闭环可用”。

特别是这一轮新增后，测试已经不只是验证接口有没有返回 200，而是开始验证：

- 权限是否真的拦住了不该做的动作
- 内容更新后是否真的进入版本体系
- 发布后公开接口是否真的切换到最新结果
- 删除后公开数据是否真的被清理出当前发布结果

这让 API 层的可信度，比第二阶段明显往前推进了一步。
