# 第三阶段总体规划

## 背景

第二阶段已经完成了当前项目从“展示型项目”到“三端统一内容平台基础设施”的关键升级，当前已经具备：

- `apps/web`：Nuxt 4 展示端
- `apps/admin`：Nuxt 4 管理后台前端
- `apps/api-server`：NestJS API Server
- `SQLite + Drizzle ORM`：正式数据层
- 权限、工作流、版本、统一发布能力
- 测试、联调、环境与部署收口文档

这意味着项目已经不再处于“底座搭建期”。

第三阶段不应该继续重复第二阶段的基础建设，而应该进入：

- 更接近真实生产使用
- 更接近平台化运营
- 更接近长期维护能力
- 更接近产品化抽象

的阶段。

## 技术路线原则

第三阶段继续坚持当前正式技术主线，不做数据库与 ORM 方案切换：

- 数据库：`SQLite`
- ORM：`Drizzle ORM`

原因很明确：

- 以当前项目体量，这套方案完全够用
- 它已经在第二阶段中完成了真实验证
- 它足够轻量、足够稳定，也足够适合个人 IP 产品早期演进
- 它可以支持当前的后台管理、发布流、版本能力与正式部署需求

因此第三阶段明确约定：

> 不切换 `SQLite + Drizzle ORM` 主方案，继续围绕现有主线做稳定增强。

后续即使真的发展到更大规模，也应在明确业务需求后再评估迁移，而不是在当前阶段过早切换技术栈。

## 第三阶段目标

第三阶段的核心目标建议定义为：

> 把当前已具备三端闭环的内容平台，推进到“可正式运行、可持续维护、可继续扩展”的稳定阶段。

与第二阶段相比，第三阶段关注点将从“能不能搭起来”，转向：

- 能不能稳定运行
- 能不能更容易部署
- 能不能更好维护
- 能不能支持真实运营
- 能不能支持后续持续演进

## 第三阶段主线

第三阶段确认围绕以下四条主线推进：

### 主线一：部署落地

目标是让当前项目真正具备“线上可运行”的能力，而不是只停留在部署蓝图。

重点包括：

- 三端正式部署方案落地
- 运行环境配置与域名接入
- 数据持久化与备份策略落地
- 部署后基本验证流程
- 基础上线 / 回滚操作手册

### 主线二：可观测性与运维基础

目标是让系统具备最基础的“可看见、可排查、可恢复”能力。

重点包括：

- 健康检查增强
- 服务日志规范化
- 错误追踪与异常定位
- 基础监控指标梳理
- 关键链路故障排查文档

### 主线三：后台协作与内容运营能力增强

目标是让后台从“单人可用”进一步走向“多人协作更清晰”。

重点包括：

- 更清晰的内容审核与发布体验
- 更细粒度的操作边界
- 更友好的版本与发布管理视图
- 更完整的运营字段与后台提示
- 更强的内容协同效率

### 主线四：测试与质量体系扩展

目标是从第二阶段的关键链路验证，进一步扩展到更系统化的质量保障。

重点包括：

- Web / Admin E2E 冒烟
- 更细粒度的 API 模块测试
- 构建后验证
- 发布前自动检查
- 更稳定的回归基线

## 第三阶段边界

为了避免第三阶段范围失控，需要明确“不做什么”。

第三阶段暂时不优先做：

- 大规模视觉改版
- 全新业务线扩张
- 复杂微服务拆分
- 替换当前 `SQLite + Drizzle ORM` 主方案
- 一次性重写前端架构
- 过早引入过重的平台化基础设施

也就是说，第三阶段应该坚持：

- 延续第二阶段主架构
- 在现有基础上稳步增强
- 优先做“真实可用性”和“真实维护性”
- 不轻易引入新的大规模技术迁移

## 第三阶段建议里程碑结构

这里先给出高层拆分，后续再继续细化为标准执行任务。

## T1：正式部署落地

目标：

- 把当前三端真正部署到目标运行环境
- 形成可重复的部署步骤
- 验证部署后链路

建议范围：

- Web / Admin / API 三端部署
- 环境变量与域名配置
- SQLite 持久化目录
- 基础备份与回滚方案
- 部署后联调验证

## T2：运行稳定性与可观测性

目标：

- 建立基础运行观测能力
- 让问题更容易发现和排查

建议范围：

- 健康检查增强
- 关键日志补齐
- 错误响应与告警入口梳理
- 运维排查手册

## T3：后台协作与运营体验增强

目标：

- 提升内容管理效率
- 让后台更接近真实运营工具

建议范围：

- 发布中心体验增强
- 版本与差异信息增强
- 后台提示与状态反馈优化
- 更细粒度协作流设计

## T4：测试体系扩展

目标：

- 把当前“关键链路验证”升级为“更系统的质量保障”

建议范围：

- Web / Admin E2E 冒烟
- API 模块测试扩展
- 构建后验证
- 发布前自动检查脚本

## T5：第三阶段总结与下一阶段准备

目标：

- 完成第三阶段沉淀
- 为下一阶段继续扩展做准备

建议范围：

- 第三阶段总结
- 交付清单
- 风险与待办整理
- 下一阶段规划入口

## 建议执行顺序

建议继续按“先运行、再稳定、再增强、再扩测试、最后总结”的顺序推进：

1. `T1` 正式部署落地
2. `T2` 运行稳定性与可观测性
3. `T3` 后台协作与运营体验增强
4. `T4` 测试体系扩展
5. `T5` 第三阶段总结与下一阶段准备

这个顺序的好处是：

- 先把系统放到真实运行环境里
- 再补运行问题的观察和排查能力
- 然后再优化后台体验
- 最后扩测试，避免测试脱离真实使用场景

## 第三阶段完成标准

第三阶段整体完成标准建议定义为：

- 三端具备正式运行环境
- 至少一套真实部署流程可复用
- 基础运维与排查能力具备
- 后台协作和发布体验明显增强
- 测试体系较第二阶段有明显扩展
- 第三阶段形成完整总结与交付材料

## 当前下一步建议

第三阶段总体规划确认后，建议立刻进入：

- `T1`：正式部署落地

后续再把 `T1` 继续拆成 `T1-1 / T1-2 / T1-3 ...` 的子任务，并继续沿用当前标准流程推进：

1. 先出详细设计文档
2. 再开发实现
3. 完成本地或目标环境自测
4. 写技术记录文档
5. 更新里程碑
6. 分类别提交并推送
