# T3-2：版本记录展示与差异理解能力设计

## 背景

`T3-1` 已经增强了统一发布中心的信息密度和状态反馈，第三阶段当前继续沿着“后台协作与运营体验增强”的方向推进。

下一步最适合继续提升的，是各内容模块中的：

- 版本历史展示

因为当前系统虽然已经具备：

- 文案历史版本
- 简历历史版本
- 项目历史版本
- 版本恢复能力

但版本区块现在仍然更偏“底层可用”，还不够利于快速理解。

当前页面已经能展示：

- 版本号
- 状态
- 变更类型
- 创建人
- 创建时间
- 部分快照字段

但还存在一个明显问题：

> 用户能看到“有版本”，却不容易快速理解“这个版本和当前内容到底差在哪、值不值得恢复”。

所以 `T3-2` 的目标，不是新增版本机制，而是增强版本历史的可读性和差异理解能力。

## 本轮目标

本轮聚焦以下三个方向：

- 提升版本卡片的摘要能力
- 增强“与当前内容相比”的差异理解
- 让恢复操作前的判断成本更低

本轮暂时不做：

- 真正的全文 diff 算法
- 富文本差异高亮
- 逐字段可视化变更树
- 多版本并排比对器

也就是说，`T3-2` 先解决“能快速理解版本价值”，不追求一步做到复杂 diff 工具。

## 当前问题分析

### 1. 版本卡片信息仍偏原始字段展示

目前版本卡片虽然包含基础元信息，但对用户最重要的问题仍然回答得不够直接：

- 这个版本和当前内容是否一致
- 它主要影响哪些内容
- 恢复后最可能改变什么

### 2. 恢复动作缺少“恢复前理解”支撑

当前页面可以直接点击“恢复此版本”，但在操作前，用户还需要自己阅读快照字段，才能大致判断：

- 标题是否不同
- 文案内容是否不同
- 教育 / 经历 / 标签数量是否变化
- 这是一次小改动还是结构变化

这会增加恢复前的认知负担。

### 3. 三个模块的版本理解方式不统一

目前：

- 文案页只看文案值
- 简历页只看标题
- 项目页只看标题

整体仍然缺少一致的“版本摘要 + 差异提示”模式。

## 设计原则

### 1. 不引入重型 diff，只做轻量差异摘要

本轮优先做：

- 摘要字段
- 关键数量变化
- 当前版本对比提示
- 是否一致的快速判断

### 2. 三类模块尽量统一版本认知模式

虽然文案、简历、项目的数据结构不同，但版本卡片应尽量遵循统一模式：

- 元信息
- 快照摘要
- 与当前版本的差异提示
- 恢复操作

### 3. 只展示对恢复判断最有帮助的信息

本轮不追求全字段展示，而是优先展示最有助于“要不要恢复”的内容。

## 建议实现方向

## 1. 新增版本理解辅助 composable

建议增加一个轻量 composable，例如：

- `useContentVersionInsights()`

负责统一生成：

- 版本摘要
- 当前对比提示
- 模块级差异说明

## 2. 文案版本增强

建议新增：

- 当前文案与历史文案是否一致
- 历史文案长度或内容差异提示
- 当前状态与历史状态对比
- 快速说明“恢复后会替换当前文案值”

## 3. 简历版本增强

建议新增：

- 当前语言标题 / 历史标题是否一致
- 教育、经历、技能组、联系方式数量摘要
- 与当前简历结构的数量变化提示

## 4. 项目版本增强

建议新增：

- 当前语言标题 / 历史标题是否一致
- 标签数量、排序值、外链 / 封面状态摘要
- 与当前项目结构差异提示

## 5. 恢复按钮周边提示更明确

建议保留当前恢复按钮，但在版本卡片中增加更明确的语义提示，例如：

- 与当前内容一致
- 标题已变化
- 标签数量不同
- 文案内容不同

## 完成标准

`T3-2` 完成后，应满足：

- 三类模块的版本卡片信息更完整
- 用户更容易判断恢复某版本会带来什么变化
- 版本展示方式比当前更统一
- 不需要引入复杂 diff，也能明显降低理解成本
