# T4-1：Web 关键页面 E2E 冒烟记录

## 背景

第三阶段完成 `T3` 以后，后台页面在协作体验上已经有了比较完整的增强。接下来进入 `T4`，重点开始从“体验建设”转向“质量保障体系扩展”。

对当前项目来说，最先适合落地浏览器级验证的，是 Web 展示端。原因很直接：

- Web 是最终公开结果
- 后台内容、统一发布、公开 API 最终都会在 Web 呈现
- Web 页面数量相对可控，适合先建立第一批稳定 smoke 用例

所以 `T4-1` 的目标，是先为公开站点关键页面建立一条可重复执行的 E2E 冒烟链路。

## 本次实现

### 1. 为 `apps/web` 补齐 Playwright 测试基础设施

这次在 `apps/web` 中新增了：

- `@playwright/test`
- `apps/web/playwright.config.ts`
- `apps/web/tests/smoke.spec.ts`

同时补充了脚本：

- `pnpm --dir apps/web test:e2e`
- `pnpm --dir apps/web test:e2e:headed`

这意味着 Web 端从这一轮开始，已经拥有正式的浏览器级 smoke 测试入口，而不再只依赖手工打开页面检查。

### 2. 使用独立预览端口运行 Web 冒烟

为了避免和开发中的本地服务端口冲突，这次没有直接复用当前 `3000` 的手工开发环境，而是在 Playwright 配置里单独使用：

- Web 预览端口：`3100`
- API 健康检查：`3011`

同时，测试配置会优先：

- 复用已启动的 `api-server`
- 自动构建并启动 Web preview

这种方式比直接跑 `nuxt dev` 更稳定，也更适合作为后续持续集成或发布前检查的基础。

### 3. 补齐四条关键页面 smoke 用例

这次首批覆盖了四条关键公开页面：

- 首页 `/`
- 简历页 `/resume`
- 项目列表页 `/projects`
- 项目详情页 `/projects/resume-platform`

断言重点聚焦在稳定的页面能力，而不是复杂交互：

- 主标题是否出现
- 导航入口是否存在
- 页面关键模块是否渲染
- 项目详情页是否可以访问并返回列表页

这一轮的定位非常明确：先证明公开站点的关键路径在浏览器层是可达且可见的。

### 4. 顺手解决了两类测试落地问题

在把 Playwright 冒烟真正跑起来的过程中，还额外解决了两类很实际的问题。

#### 问题一：`nuxt preview` 参数传递方式不正确

一开始尝试通过命令行直接给 `preview` 传 host 和 port，结果 `Nuxt` 把参数解释成了错误的工作目录，导致提示找不到 `nitro.json`。

最终收口为更稳定的方式：

- 通过环境变量 `NITRO_HOST` / `NITRO_PORT` 控制 preview 服务
- 在 `playwright.config.ts` 中直接使用构建 + preview 命令

这样配置之后，Web 预览服务可以稳定启动在 `3100`。

#### 问题二：Playwright 运行时浏览器依赖缺失

初次执行测试时，Playwright 已经安装，但浏览器运行时依赖还没有完整下载，导致测试直接失败。

本轮通过执行：

- `pnpm --dir apps/web exec playwright install chromium`

补齐了：

- Chromium
- FFmpeg
- Headless Shell

这也意味着当前仓库已经具备运行 Web 冒烟测试所需的浏览器依赖。

### 通过结果

当前共通过 4 条用例：

这说明当前 Web 公开站点的核心访问路径，已经有了正式的浏览器级保障入口。

## 本次收获

`T4-1` 的意义不只是“多了 4 条测试”，更重要的是：

- 第三阶段第一次把浏览器级验证正式纳入仓库能力
- 公开站点的关键页面不再只靠人工点点看
- 后续 `T4-2` 的 Admin 冒烟和 `T4-4` 的发布前检查，已经有了直接可复用的模式

也就是说，这一轮真正建立起来的，是“关键页面可重复回归”的基础设施。
