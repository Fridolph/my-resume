# T4-1：Web 关键页面 E2E 冒烟设计

## 背景

第三阶段在完成 `T3` 之后，后台协作与运营体验已经有了比较完整的收口。接下来的重点，需要从“页面体验增强”切换到“关键链路验证体系增强”。

对于当前项目来说，Web 展示端虽然页面数量不算多，但它承载的是整个系统最直接的公开结果。也就是说，后台管理内容、统一发布批次、公开 API 输出，最终都会反映到 Web 页面上。

所以 `T4-1` 的目标，不是做全面的浏览器测试体系，而是先建立一套最小但稳定的 Web 冒烟验证，优先覆盖关键公开页面。

## 本轮目标

本轮优先为以下页面补 E2E 冒烟：

- `/`
- `/resume`
- `/projects`
- `/projects/[slug]`

测试重点不是复杂交互，而是确认：

- 页面能正常访问
- 关键标题与内容区块已渲染
- 列表页可以进入详情页
- 公开内容链路在浏览器层可用

## 设计原则

### 1. 先做关键路径冒烟，不追求大而全

当前阶段优先建立“有测试”与“可重复运行”的基础，不在 `T4-1` 一次性把所有语言、主题、异常态都覆盖完。

### 2. 尽量使用稳定断言点

优先选择：

- 页面标题
- 主标题
- 导航链接
- 稳定的关键文案

避免依赖容易随视觉微调而变化的 DOM 结构。

### 3. 让测试能在本地复用现有服务，也能独立启动

Playwright 配置应尽量支持：

- 本地已有 `api-server` / `web` 时直接复用
- 本地没有服务时自动启动所需服务

## 建议实现

## 1. 为 `apps/web` 增加 Playwright 依赖与脚本

建议增加：

- `@playwright/test`

并补充脚本：

- `test:e2e`

## 2. 新增 `playwright.config.ts`

配置中应包含：

- `baseURL: http://localhost:3000`
- 自动复用已存在的 `3000` 与 `3011` 服务
- 如果服务不存在，则自动启动 `apps/web` 与 `apps/api-server`

## 3. 新增 Web 冒烟用例

建议新增一个 smoke spec，覆盖：

- 首页主标题与导航
- 简历页主标题与内容块
- 项目列表页标题与卡片入口
- 项目详情页标题与返回按钮

## 完成标准

`T4-1` 完成后，应满足：

- `apps/web` 具备基础 Playwright 冒烟能力
- 关键公开页面可以通过一条命令重复验证
- Web 端已有明确可扩展的 E2E 入口
