# T1-1：部署目标与运行拓扑设计

## 背景

第三阶段的第一个里程碑是：

- `T1`：正式部署落地

而 `T1` 的第一步，并不是马上开始把服务往某个平台上推，而是先把最关键的前提确定清楚：

- 当前阶段到底选择什么部署目标
- 三端应该采用什么运行拓扑
- `SQLite + Drizzle ORM` 在线上环境下如何持久化
- 域名、反向代理和服务边界应该如何组织

如果这一步不先定下来，后续的部署实施就很容易出现几个问题：

- 选了不适合 `SQLite` 的平台，后面被迫返工
- Web / Admin / API 的部署方式不一致，后续维护成本高
- 域名与 CORS 关系先天不清晰
- 回滚、备份和联调都缺少统一基础

因此，`T1-1` 的目标，是先为第三阶段的部署工作选定一条当前阶段最稳妥、最适合现有架构的主线。

## 当前约束

当前项目的真实技术约束已经很明确：

- 三端架构：`web` / `admin` / `api-server`
- 后端运行时：Node.js + NestJS
- 数据层：`SQLite + Drizzle ORM`
- 当前体量：小规模、单实例可接受
- 当前重点：先完成稳定部署，而不是先追求高可用复杂架构

这意味着当前部署方案必须优先满足：

- 支持 Node 服务稳定运行
- 支持多服务部署
- 支持 `SQLite` 数据文件持久化
- 支持后续域名、CORS 和环境变量管理
- 不强迫当前项目过早切换数据库方案

## 平台选型结论

结合当前阶段目标，以及官方文档中对持久化能力的说明，`T1-1` 建议做出如下结论：

### 当前阶段首选平台：Render

本阶段推荐优先选择：

- `Render`

作为第三阶段正式部署落地的主目标平台。

### 推荐原因

#### 1. 它天然适合当前的三服务分离部署

当前项目正好需要：

- 一个 `web` 服务
- 一个 `admin` 服务
- 一个 `api-server` 服务

Render 对这种“多 Web Service / Private Service”模式比较自然，且支持通过 Blueprint 或服务面板统一管理。

#### 2. 它支持持久化磁盘，适合当前阶段继续使用 SQLite

根据 Render 官方文档，持久化磁盘可以挂载到服务的运行实例上，从而保留本地文件系统中的数据变更。

这对于当前项目非常关键，因为：

- `SQLite` 本质上就是文件数据库
- 当前阶段明确不切换 `SQLite + Drizzle ORM`
- 因此部署平台必须允许数据库文件落到持久化目录

Render 在这一点上和当前项目技术路线是兼容的。

#### 3. 它更适合当前阶段的“小规模、单实例”要求

Render 官方文档也明确说明：

- 挂载持久化磁盘的服务不能横向扩多实例
- 带磁盘的服务也不适合零停机切换

但这并不是当前阶段的问题，因为当前项目的部署目标本来就是：

- 先稳定上线
- 先完成正式运行
- 先建立可复用部署流程

而不是当前就追求高并发或多实例扩缩容。

所以这些限制在当前阶段是可以接受的。

### 备选平台：Railway

`Railway` 可以作为当前阶段的备选方案。

它同样提供：

- 持久化 Volume
- 多服务部署能力

所以从理论上讲，也能够支撑当前项目继续使用 `SQLite`。

但在 `T1-1` 阶段，仍然建议优先 Render，原因是：

- 当前更适合先确定一个主平台，不要并行维护两套部署思路
- Render 的持久化磁盘与当前项目拓扑更容易形成“单实例 + 持久化 SQLite”的清晰主线

### 当前阶段不建议作为主平台：Vercel

`Vercel` 并不适合作为当前阶段整套三端统一部署的主目标平台。

原因不是它不能部署 Nuxt，而是它与当前主数据方案不匹配。

根据 Vercel 官方文档：

- Nitro / Nuxt 部署到 Vercel 时，服务端逻辑会运行在 Functions 模型之上
- 其运行形态更偏向无服务器 / 托管函数
- 当前并没有为本项目提供与 `SQLite` 文件持久化完全匹配的主线方案

因此，在当前明确继续使用 `SQLite + Drizzle ORM` 的前提下，Vercel 更适合：

- 未来如果只部署纯前端站点时考虑
- 或在数据库方案发生变化后再重新评估

而不是当前阶段的首选统一部署平台。

## 推荐运行拓扑

`T1-1` 建议采用如下拓扑：

### 服务划分

- `web`：独立 Web Service
- `admin`：独立 Web Service
- `api-server`：独立 Web Service 或 Private Service
- `SQLite`：挂载到 `api-server` 的持久化磁盘目录

### 域名建议

建议使用三域名分离：

- `www.example.com` 或 `example.com` -> `apps/web`
- `admin.example.com` -> `apps/admin`
- `api.example.com` -> `apps/api-server`

### 当前推荐原因

这种三域名分离结构的优点是：

- Web 与 Admin 访问边界清晰
- API CORS 更容易配置
- 公开访问和管理访问更容易区分
- 后续如果要接入监控、CDN 或缓存策略，也更清楚

## SQLite 持久化方案

当前阶段继续坚持：

- `SQLite + Drizzle ORM`

因此部署时建议：

- 仅在 `api-server` 服务上挂载持久化磁盘
- 通过 `REPO_DATABASE_PATH` 指向磁盘目录内的数据库文件
- 所有 migration、备份和数据库检查都围绕这一路径执行

### 挂载建议

推荐使用清晰的持久化路径，例如：

- `/var/data/platform.sqlite`

并在环境变量中显式配置：

- `REPO_DATABASE_PATH=/var/data/platform.sqlite`

### 原则

- 数据文件不能留在临时构建目录
- 数据库备份必须早于版本切换
- migration 应在部署流程中明确执行
- 回滚时必须考虑数据库文件与应用版本兼容性

## 反向代理与请求边界

### Web

- 只访问 `/api/public/*`
- 不直接访问后台管理接口

### Admin

- 只访问 `/api/admin/*`
- 要求带登录态与权限边界

### API Server

- 统一暴露 `/api` 前缀
- 通过 `CORS_ORIGIN` 收口允许访问的来源

推荐的 CORS 关系：

- 允许 `web` 域名
- 允许 `admin` 域名
- 不默认放开所有来源

- `T1-3`
