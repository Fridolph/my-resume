# P2：站点设置模块正式迁移闭环记录

## 背景

正式版 P1 完成后，第二阶段已经拥有了独立的 `apps/api-server`、统一的 Drizzle 数据层，以及基础的数据库迁移机制。接下来的第一件事，不应该再去扩散更多模块，而是先把过渡阶段已经验证过的 `site settings` 样板，正式迁移到三端统一架构里。

这一步的意义非常直接。只要站点设置能完成从“Nuxt 内部临时 API”到“NestJS 正式 API Server”的迁移，就说明第二阶段新的架构方向已经不仅是理论成立，而是已经开始真正承载业务。

## 本次目标

P2 的任务聚焦在一条清晰链路上：

- 在 `apps/api-server` 中建立 `SiteSettingsModule`
- 提供站点设置的查询与更新接口
- 让 `packages/sdk` 负责统一请求封装
- 让 `apps/admin` 设置页通过 API Server 读写数据
- 让 `apps/web` 通过 API Server 读取站点设置
- 移除旧的 Nuxt 内部临时 API 路由

也就是说，P2 不是单纯“多写一个接口”，而是要完成第一条正式的三端迁移闭环。

## 本次实现

### 1. 建立 `SiteSettingsModule`

本次在 `apps/api-server` 中新增了站点设置模块，并建立了以下组成：

- `SiteSettingsModule`
- `SiteSettingsController`
- `SiteSettingsService`
- 站点设置请求校验 schema

这样处理后，站点设置已经不再依赖前端应用内部的 route，而是被收口到统一后端服务中。

### 2. 提供正式站点设置接口

本次新增的正式接口包括：

- `GET /api/site-settings`
- `PUT /api/site-settings`

这意味着站点设置已经正式进入 `api-server` 的模块化管理，而不再只是某个前端应用里的局部 server route。

### 3. 为 `packages/sdk` 增加站点设置 API Client

为了避免 Web 和 Admin 直接散落请求逻辑，本次在 `packages/sdk` 中增加了站点设置客户端封装。

这一步带来的好处有三个：

- 前后台调用方式统一
- API 地址和请求细节不需要散落在页面里
- 后续更多模块可以按同样模式继续接入

这也是第二阶段后续非常重要的约束：前端应用不要直接堆积接口细节，而是尽量通过 SDK 层统一访问 API Server。

### 4. Admin 设置页切换到 API Server

本次已经将 `apps/admin` 的站点设置页切换为通过 `packages/sdk` 调用 `apps/api-server`。

当前后台设置页已经不再依赖原先的 Nuxt 临时接口，而是通过正式的统一 API 读写站点配置。这一步非常重要，因为它标志着后台管理页面第一次真正接入独立后端服务。

### 5. Web 站点设置读取切换到 API Server

本次已经在 `apps/web` 中增加站点设置读取逻辑，并通过 API Server 获取站点标题、描述、默认语言和 OG 图相关配置。

虽然 Web 的公开内容主体仍有部分是 SDK mock，但站点级设置已经进入真实服务链路。这为后续 `P4` 的公开内容全面接入打下了第一层基础。

### 6. 移除旧的 Nuxt 临时 API 路由

P2 完成后，原先位于 `apps/admin` 与 `apps/web` 中的 `site-settings` 临时 API 路由已经被移除。

这一步非常关键。只有在正式接口已经稳定可用之后，及时删除旧链路，才能真正避免系统长期停留在“双轨并存”的状态。

## 本地自测结果

本次已完成以下验证。

### 1. 类型检查通过

执行通过：

- `pnpm --dir apps/api-server typecheck`
- `pnpm --dir apps/admin typecheck`
- `pnpm --dir apps/web typecheck`

### 2. API Server 读写通过

已验证：

- `GET /api/site-settings` 可以读取当前站点设置
- `PUT /api/site-settings` 可以更新站点设置
- 更新后的结果会返回最新 `updatedAt`

### 3. Admin 页面联调通过

已验证：

- Admin 设置页可通过 `packages/sdk` 请求正式接口
- 页面可以显示来自 API Server 的站点设置数据
- 页面文案已更新为 P2 正式迁移样板说明

### 4. Web 读取链路通过

已验证：

- Web 已通过 API Server 读取站点设置
- 返回的站点设置已经进入页面 SSR 数据中
- 站点标题与描述可由正式接口返回值驱动

### 5. 旧路由已失效

已验证：

- `http://localhost:3003/api/site-settings` 返回 `404`
- `http://localhost:3007/api/site-settings` 返回 `404`

这说明临时 Nuxt 路由已经成功退出主链路。

### 6. SQLite 数据落盘同步通过

已验证：

- 更新站点设置后，`data/platform.sqlite` 中的 `site_settings` 表已同步变化
- `updated_at` 与 `seo` 字段内容可以正确查询到最新值

## 当前结果

P2 完成后，第二阶段已经拥有第一条真正完成迁移的三端链路：

- Admin 通过 SDK 调用 API Server
- API Server 调用 Drizzle ORM
- Drizzle ORM 落盘到 SQLite
- Web 通过 API Server 读取同一份站点设置

这意味着第二阶段的统一架构已经不再只是“后端基座搭好了”，而是已经开始兑现真实业务迁移价值。

## 这次 P2 的意义

P2 的价值，不仅在于“站点设置可以读写”，而在于它证明了一件更关键的事：我们已经拥有了一套可以复制到其他模块的正式迁移模板。

接下来无论是用户、文案、简历还是项目模块，都可以沿着这条路径继续迁移：

- 先建 API Server 模块
- 再补 SDK 客户端
- 再切 Admin 页面
- 再切 Web 公开读取
- 最后移除旧临时链路

这会让第二阶段的推进节奏更可控，也让技术债不会继续累积。

## 下一步

P2 完成后，下一步建议进入 `P3`：后台核心内容模块接入。

下一阶段建议优先考虑以下迁移顺序：

- 用户管理
- i18n 文案管理
- 简历管理
- 项目管理

其中，优先级最高的建议是：先从你最想尽快看到真实管理效果的模块开始，而不是一次同时推四个模块。这样更有利于保持节奏和质量。
