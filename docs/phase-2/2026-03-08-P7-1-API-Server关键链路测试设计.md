# P7-1：API Server 关键链路测试设计

## 背景

到 `P6-5` 为止，`apps/api-server` 已经具备第二阶段的核心业务能力：

- 认证与登录态
- 角色权限与接口访问控制
- 内容状态流转
- 审计字段写入
- 历史版本快照
- 历史版本恢复
- 统一发布批次与公开读取

目前这些能力已经完成了多轮人工联调，但仓库里还没有正式测试基础设施。

这意味着当前系统虽然“能跑”，但仍然存在两个风险：

- 后续改动容易破坏核心链路而不自知
- 第二阶段收尾时缺少最基本的自动化质量保障

因此，`P7-1` 的目标，是先为 API Server 最关键的工作流能力建立一套最小可用、可重复执行的自动化测试基础。

## 本轮目标

本轮聚焦“关键链路测试”，只覆盖最容易回归、且业务价值最高的链路：

- 登录认证
- 受保护接口访问控制
- 内容状态流转
- 版本快照生成
- 历史版本恢复
- 统一发布批次创建与激活
- 公开接口读取激活批次内容

本轮暂时不做：

- 全量控制器覆盖
- UI 组件测试
- 浏览器级 E2E
- 覆盖率门禁
- 性能压测
- 模块级快照测试

也就是说，`P7-1` 先解决“核心能力自动化可验证”，不追求一步做到“大而全测试体系”。

## 核心难点

### 1. 当前数据库默认直连主开发库

目前 `packages/database` 默认把 SQLite 数据文件固定到：

- `data/platform.sqlite`

这对本地开发是方便的，但对测试并不安全。

如果测试直接操作主库，就会带来两个问题：

- 测试会污染日常开发数据
- 测试结果会依赖已有数据状态，变得不稳定

所以 `P7-1` 的第一步，不是先写测试用例，而是先补“测试数据库隔离能力”。

### 2. 当前数据库连接是全局单例

`packages/database/src/client.ts` 当前通过全局连接复用 SQLite 连接。

这对开发运行是合理的，但在测试环境里会带来额外要求：

- 测试前要能指定独立数据库路径
- 测试间要能在同一套测试库上稳定运行
- 测试结束后最好可以清理测试数据库

因此，本轮需要让数据库客户端支持通过环境变量切换测试库路径。

## 设计原则

### 1. 优先做 API 集成测试，而不是只做纯单元测试

本轮验证的目标是“关键链路”，而这些链路跨越：

- NestJS 控制器
- Guard / Decorator
- Service
- Database

如果只做纯函数级单元测试，很多真正关键的问题是测不到的，例如：

- Cookie 登录态是否正确生效
- 受保护接口是否真的被 Guard 拦截
- 控制器返回结构是否正确
- 统一发布批次接口是否能串到数据库层

因此本轮优先采用：

- **API 集成测试**

也就是直接起一个测试态 Nest 应用，然后用 HTTP 请求打真实接口。

### 2. 使用独立测试数据库

本轮测试必须使用单独的 SQLite 文件，例如：

- `data/platform.test.sqlite`

并通过环境变量注入，而不是硬编码改主库路径。

这样可以保证：

- 本地测试不会污染开发数据库
- 每次执行结果更稳定
- 测试失败时也更容易排查

### 3. 先覆盖“主链路冒烟级”场景

本轮不追求每个模块很多小用例，而是优先确保一条从登录到内容工作流再到统一发布的“主链路”可自动化跑通。

## 技术方案

### 测试框架

本轮建议在 `apps/api-server` 中引入：

- `vitest`
- `@nestjs/testing`
- `supertest`

其中：

- `vitest` 负责测试运行与断言
- `@nestjs/testing` 负责创建测试态 Nest 应用
- `supertest` 负责对 HTTP 接口发请求

### 测试目录建议

建议增加：

- `apps/api-server/test/`

并按“能力块”组织，例如：

- `auth.integration.test.ts`
- `workflow.integration.test.ts`
- `releases.integration.test.ts`
- `helpers/test-app.ts`

### 数据库隔离方案

建议在 `packages/database/src/client.ts` 中支持环境变量，例如：

- `REPO_DATABASE_PATH`

当存在该变量时：

- 优先使用该路径作为 SQLite 文件

同时在测试启动前：

- 删除旧的测试数据库文件
- 运行 migration
- 再启动测试应用

这样可以保证测试是“从干净数据库开始”。

## 本轮测试范围

### 场景一：认证与受保护接口

验证：

- 未登录访问后台接口返回未授权
- 正确账号密码可以登录
- 登录后可访问受保护接口

### 场景二：内容状态流转与审计字段

验证：

- 简历从 `draft -> reviewing -> published` 的合法流转成功
- 非法流转会被拒绝
- 审计字段如 `updatedBy`、`reviewedBy`、`publishedAt` 正常写入

### 场景三：版本快照与恢复

验证：

- 内容更新后会新增版本快照
- 恢复历史版本后会生成新的 `restore` 版本

### 场景四：统一发布批次

验证：

- 当存在已发布简历时，可以创建统一发布批次
- 新批次创建后会成为 `active`
- 再次创建新批次时，旧批次会自动归档
- 可以重新激活历史批次

### 场景五：公开接口读取激活批次

验证：

- `/api/public/release` 可返回当前激活批次
- `/api/public/resume`、`/api/public/projects`、`/api/public/translations` 能读取激活批次对应内容

## 命令与执行方式

本轮建议在 `apps/api-server/package.json` 中增加：

- `test`
- `test:run`

例如使用：

- `vitest`
- `vitest run`

后续根目录也可以视情况追加工作区级聚合脚本。

## 完成标准

`P7-1` 完成后，应满足：

- `apps/api-server` 拥有正式测试基础设施
- 测试运行使用独立 SQLite 测试数据库
- 至少覆盖认证、工作流、版本恢复、统一发布四类关键链路
- 测试命令可以在本地稳定执行
