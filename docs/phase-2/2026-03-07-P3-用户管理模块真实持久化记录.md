# P3：用户管理模块真实持久化记录

## 背景

在正式版 `P1` 建好 `api-server` 和 Drizzle 数据基座，并在 `P2` 完成站点设置模块正式迁移之后，第二阶段已经证明了一件事：三端统一架构不只是可以工作，而且已经可以稳定承载真实业务模块。

接下来进入 `P3` 时，最合理的顺序不是一次并行迁移所有模块，而是按优先级逐个拿下后台核心内容模块。第一步选择用户管理，是因为它同时覆盖了列表读取、创建、更新、角色切换、状态切换和权限展示这些高频后台能力，非常适合作为 `P3` 的第一条正式迁移链路。

## 本次目标

本次用户管理模块迁移的目标很明确：

- 建立 `users` 表结构
- 建立共享数据库仓储函数
- 在 `apps/api-server` 中增加 `UsersModule`
- 提供用户列表、创建和更新接口
- 在 `packages/sdk` 中补充用户 API Client
- 将 `apps/admin` 的用户管理页切换到真实 API

这一步完成后，后台核心模块就不再只停留在本地状态管理，而是开始真正进入持久化阶段。

## 本次实现

### 1. 建立 `users` 表结构

本次在 `packages/database` 中新增了用户表 schema，并补充了对应迁移文件。

当前用户表包含以下核心字段：

- `id`
- `name`
- `email`
- `role`
- `status`
- `permissions`
- `updated_at`

这里的 `permissions` 当前仍以 JSON 字符串形式存入 `text` 字段，以便和当前 Monorepo 的共享类型保持一致，同时也降低起步阶段的建模复杂度。

### 2. 建立用户仓储与默认种子

本次在共享数据库包中新增了用户仓储能力，主要包括：

- 用户初始种子写入
- 用户列表读取
- 创建用户
- 更新用户

同时，角色与权限映射也被统一放入数据库层，由仓储函数自动根据角色生成权限集合。这让后台页面不需要再单独维护“角色变化后权限应该怎么更新”的规则。

### 3. 在 `apps/api-server` 中建立 `UsersModule`

本次新增了：

- `UsersModule`
- `UsersController`
- `UsersService`
- 用户请求校验 schema

并补充了以下正式接口：

- `GET /api/users`
- `POST /api/users`
- `PUT /api/users/:userId`

这意味着用户管理已经正式进入 `api-server` 模块体系，而不再只是后台页面内部的局部状态逻辑。

### 4. 在 `packages/sdk` 中增加用户 API Client

为了保持和 `site settings` 模块一致的组织方式，本次在 `packages/sdk` 中补充了用户模块请求能力。

当前前端已经可以通过统一 SDK 完成：

- 用户列表查询
- 用户创建
- 用户更新

这一步的意义在于，后台页面不再直接散落 `fetch` 细节，而是通过 SDK 统一访问 API Server，后续扩展更多模块时也能保持同样的结构。

### 5. 将 `apps/admin` 用户管理页切换为真实 API

本次已经将后台用户页从本地 `useState` 驱动，迁移为通过 API Server 驱动。

当前页面已经完成以下变化：

- 页面加载时通过真实接口读取用户列表
- 新建用户通过 `POST /api/users` 持久化
- 编辑用户通过 `PUT /api/users/:userId` 持久化
- 角色切换通过正式接口更新
- 用户状态切换通过正式接口更新

同时，列表筛选、表单区、统计卡片和权限展示这些 UI 结构仍然保留，说明这一轮迁移是“替换数据源”，而不是推倒重写页面。

## 本地自测结果

本次已完成以下验证。

### 1. 数据库迁移通过

执行通过：

- `pnpm --dir packages/database db:migrate`

并已确认 `users` 表已经落盘。

### 2. 类型检查通过

执行通过：

- `pnpm --dir apps/api-server typecheck`
- `pnpm --dir apps/admin typecheck`

### 3. API 读写通过

已验证：

- `GET /api/users` 可以返回默认种子用户
- `POST /api/users` 可以创建新用户
- `PUT /api/users/:userId` 可以更新现有用户角色与状态

### 4. Admin 页面联调通过

已验证：

- 后台用户页可以显示来自真实接口的用户列表
- 页面 SSR 数据中已经包含新创建用户
- 角色切换结果会实时反映到页面与数据库中

### 5. SQLite 数据同步通过

已验证：

- `data/platform.sqlite` 中的 `users` 表已经写入默认用户种子
- 新建用户和角色更新结果可以直接在 SQLite 中查询到

## 当前结果

P3 的第一个正式模块已经完成，当前仓库已经拥有一条新的真实业务链路：

- Admin 用户页通过 SDK 请求 API Server
- API Server 的 UsersModule 处理列表、创建与更新
- Drizzle ORM 将数据落盘到 SQLite
- 角色变化会自动同步权限集合

这意味着第二阶段已经正式从“站点配置单模块迁移”进入“后台核心模块真实持久化”的阶段。

## 这次迁移的价值

用户管理模块的重要性不只在于它本身，而在于它验证了 `P3` 后续模块都能沿用同一套迁移模板：

- 数据库先建表
- API Server 建模块
- SDK 补客户端
- Admin 页切真实接口
- 数据库落盘验证

当这条路径已经在用户模块上验证成功之后，接下来的文案管理、简历管理和项目管理，就都可以按同样节奏推进。

## 下一步

用户管理完成后，下一步建议按既定顺序继续推进：

- `i18n` 文案管理
- 简历管理
- 项目管理

其中最优先建议的下一项仍然是：`i18n` 文案管理。因为它既是后台核心模块，也是后续 Web 公开内容全面接入后台数据的重要前置条件。
