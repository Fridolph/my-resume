# P6-4：版本回滚与恢复策略记录

## 背景

`P6-3` 已经让文案、简历、项目具备了统一历史快照能力，但当时的系统仍然只能“看历史”，还不能“恢复历史”。

如果后台只能知道：

- 某条内容以前改成过什么样
- 哪个版本是种子版本
- 哪个版本是更新版本

却不能把某个历史版本恢复成当前内容，那么版本系统就还只是“档案室”，还不是“可操作的工作流工具”。

所以这一轮 `P6-4` 的重点，就是把版本系统从“只读历史”推进到“可恢复历史”。

## 本次实现

### 1. 扩展版本来源类型 `restore`

本次先在共享类型中扩展了：

- `ContentVersionChangeType`

新增取值：

- `restore`

这一步很关键，因为恢复操作本身也应该是版本时间线中的正式事件，而不是隐式覆盖。

现在版本来源已经统一收口为四类：

- `seed`
- `create`
- `update`
- `restore`

这样版本链就能准确表达：

- 这条内容是初始化来的
- 还是新建来的
- 还是普通编辑来的
- 还是从历史版本恢复来的

### 2. 数据层增加按版本恢复能力

本次继续把核心能力收口在 `packages/database`，没有在控制器里直接拼恢复逻辑。

新增与调整包括：

- `packages/database/src/content-versions.ts`
  - 增加 `getContentVersionById()`
- `packages/database/src/translations.ts`
  - 支持 `update` / `restore` 两种写入来源
  - 新增 `restoreTranslationVersion()`
- `packages/database/src/resume-documents.ts`
  - 支持 `update` / `restore` 两种写入来源
  - 新增 `restoreResumeDocumentVersion()`
- `packages/database/src/projects.ts`
  - 支持 `update` / `restore` 两种写入来源
  - 新增 `restoreProjectVersion()`

这意味着恢复操作并没有绕过现有数据写入链路，而是继续复用：

- 主表写入逻辑
- 审计字段计算逻辑
- 版本快照生成逻辑
- 项目排序归一化逻辑

这样后续维护成本会低很多。

### 3. 审计字段按“恢复操作者”重新计算

本轮实现里，一个非常重要的决定是：

> 恢复的是历史内容结构，但不是历史操作者语义。

也就是说：

- 内容字段来自历史版本快照
- 但 `updatedBy` 应该写当前执行恢复的人
- 如果恢复后的状态是 `published`
  - `reviewedBy` 写当前恢复人
  - `publishedAt` 写当前恢复时间

这样后台看到的信息才符合真实语义：

- 当前这次恢复是谁做的
- 如果恢复后重新发布，那么是谁让它重新生效的

而不是错误延续旧版本中的历史发布时间与历史审核人。

### 4. API Server 增加恢复接口

本次在 NestJS 中增加了三类恢复接口：

- `POST /api/admin/translations/:translationId/versions/:versionId/restore`
- `POST /api/admin/resume/versions/:versionId/restore`
- `POST /api/admin/projects/:projectId/versions/:versionId/restore`

这些接口都要求：

- 已登录
- 具备对应模块写权限

并且会校验：

- 目标版本是否存在
- 目标版本是否属于当前模块
- 目标版本是否属于当前实体

这样可以避免跨实体、跨模块误恢复。

### 5. SDK 补齐恢复方法

本次在 `packages/sdk` 中补充了：

- `restoreTranslationVersion()`
- `restoreResumeVersion()`
- `restoreProjectVersion()`

这样后台页面仍然保持通过统一 SDK 调用 API，而不是在页面里手写请求路径。

### 6. Admin 页面补齐恢复操作入口

本次把恢复按钮接到了三个管理页面：

- 文案页：版本历史中可恢复指定版本
- 简历页：版本历史中可恢复指定版本
- 项目页：版本历史中可恢复指定版本

当前交互采用最轻量方式：

- 点击恢复
- 调用接口
- 刷新当前内容与版本历史
- Toast 提示成功或失败

本轮先不引入二次确认弹窗，保持开发节奏聚焦在链路闭环上。

## 本次实现的价值

这一轮最重要的价值，是让版本系统真正从“记录系统”升级为“恢复系统”。

做到这里之后，后台不再只是：

- 知道内容怎么变过

而是已经可以：

- 主动把历史版本恢复成当前内容
- 并且继续把这次恢复纳入历史版本链中

这意味着内容工作流已经开始具备真正的“纠错能力”和“回退能力”。

## 一个阶段结论

做到这里，内容工作流已经具备三层能力：

- `P6-1`：状态机
- `P6-2`：基础审计
