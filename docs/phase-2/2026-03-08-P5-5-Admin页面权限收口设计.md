# P5-5：Admin 页面权限收口设计

## 背景

`P5-4` 已经把权限控制落到后端接口层，但如果前端页面仍然把所有导航、入口和按钮都完整展示出来，那么体验上仍然不够一致。

因此，`P5-5` 的目标是让 `admin` 页面层也接入真实权限判断，让前后端权限边界形成闭环。

## 设计目标

这一阶段主要完成：

- 导航菜单按真实权限显示
- 后台首页入口卡片按真实权限显示
- 页面访问继续基于真实 `session.permissions` 判断
- 读写混合页面支持“只读查看 / 可编辑操作”两层能力

## 策略

### 页面访问控制

页面本身优先基于 `readPermission` 控制。

例如：

- 用户页 -> `user.read`
- 翻译页 -> `translation.read`
- 简历页 -> `resume.read`
- 项目页 -> `project.read`
- 设置页 -> `site.read`

### 页面内操作控制

按钮、保存动作、新增动作优先基于 `writePermission` 控制。

例如：

- 用户编辑 -> `user.write`
- 翻译保存 -> `translation.write`
- 简历保存 -> `resume.write`
- 项目创建/删除 -> `project.write`
- 设置保存 -> `site.write`

## 统一配置来源

这一轮不再让导航和首页入口分别维护自己的权限逻辑，而是新增统一的页面访问配置源。

这样做的好处是：

- 导航菜单可复用
- 首页入口卡片可复用
- 后续如果新增页面权限，只需要改一处

## 特殊处理：站点设置页

站点设置页这次会作为“读写分层页面”的代表样板。

原因是当前角色模型里已经存在：

- `site.read`
- `site.write`

这意味着设置页很适合展示两层能力：

- 低权限用户可以查看
- 高权限用户才可以编辑与保存
