# P5-2：基础鉴权机制设计

## 背景

`P5-1` 已经完成了统一角色模型和权限清单的收口，但那一轮解决的是“规则从哪里来”的问题，还没有解决“服务端如何识别当前用户”的问题。

如果继续往下推进：

- `admin` 真实登录态接入
- `api-server` 权限守卫
- 后台接口边界控制

那么 `api-server` 必须先具备基础鉴权能力。

因此，`P5-2` 的目标是先让后端拥有一条最小可用的正式鉴权链路：

- 登录
- 登出
- 获取当前用户
- 服务端会话识别

## 设计目标

本阶段不追求一次性实现完整权限系统，而是先建立一条稳定的身份识别基础链路。

具体目标：

- 在 `api-server` 建立独立 `AuthModule`
- 提供 `login / logout / me` 三个基础接口
- 让服务端能通过 cookie 识别当前登录用户
- 会话信息可失效、可登出、可在下一阶段复用
- SDK 预留 auth client，方便 `P5-3` 直接接入管理后台

## 为什么这一阶段选择 Cookie Session

这一轮没有直接上 JWT，也没有引入第三方 auth 框架，原因很明确：

- 当前阶段更关注“后端先具备正式登录态”
- `admin` 和 `api-server` 都运行在本地受控环境中
- 后续还会继续做权限守卫、接口边界和更完整认证设计

所以这次采用的是更适合当前阶段的轻量方案：

- 登录成功后，由服务端签发 session token
- token 通过 `httpOnly cookie` 返回给浏览器
- 后续请求通过 cookie 自动携带 session
- `api-server` 根据 token 查会话，再解析当前用户

这能让后续 `admin` 接入真实登录态时，不必再重新设计一条全新的身份传递链路。

## 会话存储设计

本次会话不会只放在内存里，而是落到 SQLite。

这样做的原因是：

- 服务重启后仍可保留短期会话
- 便于后续做会话失效和审计扩展
- 和当前 `SQLite + Drizzle ORM` 技术栈保持一致

新增表：`auth_sessions`

字段建议：

- `id`
- `user_id`
- `token_hash`
- `expires_at`
- `created_at`

这里不直接存明文 token，而是存 `token_hash`。客户端 cookie 保存原始 token，服务端只保存哈希结果。

## 密码设计

当前用户表此前没有密码字段，因此这一轮会为 `users` 增加：

- `password_hash`

本阶段仍然是开发期基础能力，所以使用的是轻量、可验证的哈希方案，而不是引入完整密码管理框架。

重点是：

- 服务端不再直接依赖前端 mock 密码
- 登录校验开始进入后端
- 后续如果要切换为更正式的密码策略，也有明确扩展点

## 接口设计

### `POST /api/auth/login`

请求体：

- `email`
- `password`

成功后：

- 返回当前 `session` 信息
- 通过 cookie 写入服务端 session token

### `POST /api/auth/logout`

行为：

- 删除当前 token 对应会话
- 清除 cookie

### `GET /api/auth/me`

行为：

- 从 cookie 读取 session token
- 解析当前用户
- 返回当前登录用户会话信息

## 与下一阶段的关系

`P5-2` 完成之后，后续就可以自然衔接：

- `P5-3`：`admin` 登录页接真实 `/api/auth/login`
- `P5-4`：基于当前用户做接口级权限守卫
- `P5-5`：前端页面和菜单按真实权限显示

换句话说，这一轮是在给“真实后台登录态”打地基。
