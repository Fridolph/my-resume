# P5-1：角色与权限模型落地记录

## 背景

`P5` 的首个子任务并不是直接去写登录接口，而是先把角色与权限规则收口到统一源。

如果不先做这一步，后面无论是：

- `api-server` 接真实鉴权
- `admin` 做真实登录态
- 接口守卫做权限判断
- 页面菜单按权限显示

都会面临“规则定义散落在多个地方”的问题。

因此，这一轮 `P5-1` 的目标，是先让角色与权限模型正式稳定下来，并同步到共享层、数据库层和管理后台演示链路里。

## 本次改动

### 1. 在 `packages/types` 建立统一角色与权限源

本次将角色与权限集合收口到了 `packages/types/src/index.ts`。

新增内容包括：

- `roleKeys`
- `permissionKeys`
- `rolePermissions`
- `getRolePermissions()`

同时将角色集合扩展为：

- `super-admin`
- `admin`
- `editor`
- `translator`
- `viewer`

并补充了 `site.read` 权限，用于区分“查看站点设置”和“修改站点设置”。

这一步完成之后，角色与权限规则不再需要在多个项目中重复维护。

### 2. 数据库用户种子切到统一权限派生

本次改造了 `packages/database/src/users.ts`。

主要变化有两点：

第一，用户 seed 改为通过 `getRolePermissions()` 派生权限，而不是手写一份本地角色映射。

第二，`ensureUsersSeed()` 改为逐条插入并 `onConflictDoNothing()`，让新增默认账号可以补进已有数据库，而不是只有首轮空表初始化时才生效。

本次补充了两个默认账号：

- `root@fridolph.local` / `root123`
- `viewer@fridolph.local` / `viewer123`

这让后续验证不同权限层级更方便。

### 3. 后端用户 DTO 校验改为共享角色定义

本次 `apps/api-server/src/modules/users/users.schema.ts` 不再手写角色枚举，而是改为直接引用共享的 `roleKeys`。

这样可以避免后续角色模型扩展时，前后端出现 schema 不一致的问题。

### 4. 管理后台登录态改为引用共享权限映射

本次 `apps/admin/app/composables/useAuth.ts` 中的 mock 登录链路仍然保留，但它已经不再手写一份本地角色权限映射，而是直接通过 `getRolePermissions()` 生成当前会话权限。

这意味着当前虽然还没接真实鉴权服务，但角色与权限模型已经和共享层对齐了。

### 5. 管理后台用户页同步显示完整角色集合

本次 `apps/admin/app/pages/users/index.vue` 改为基于共享 `roleKeys` 生成角色筛选、创建表单和角色切换入口。

同时增加了角色中文标签映射，让：

- 超级管理员
- 管理员
- 编辑
- 翻译
- 查看者

都能在用户管理页中直观看到。

## 本次实现的价值

这次 `P5-1` 虽然没有直接交付真实登录接口，但它完成了一个非常关键的基础动作：

- 角色规则统一了
- 权限规则统一了
- 前后端与数据库开始引用同一份模型了

这会直接降低后续 `P5-2`、`P5-3`、`P5-4` 的实现复杂度。

换句话说，这一步是在为真正的认证和权限体系“铺轨”。

## 本地自测

本次已完成以下验证：

- `pnpm --dir apps/admin typecheck` 通过
- `pnpm --dir apps/api-server typecheck` 通过
- `pnpm --dir apps/web typecheck` 通过
- `GET http://127.0.0.1:3011/api/users` 已能返回 `super-admin`、`viewer` 等新角色数据
- `http://127.0.0.1:3002/login` 可正常访问，登录页演示账号中已能看到新增角色

其中还顺手验证了一点：即便数据库中存在历史用户记录，接口返回的权限集合也已经改为按当前角色模型动态派生，不再依赖旧的权限 JSON 内容。
