# P5-6：公开接口与后台接口边界设计

## 背景

到 `P5-5` 为止，后台页面和接口级权限已经初步跑通，但 API 路径本身仍然混合了两类语义：

- 一类给公开站点 `web` 使用
- 一类给后台 `admin` 使用

这会带来两个问题：

- 路径命名上无法一眼判断接口属于公开域还是后台域
- 某些读取接口虽然当前被 `web` 使用，但后台也直接复用同一条路由，边界不够清晰

因此，`P5-6` 需要把“公开消费”和“后台管理”两类接口，在路径与客户端层都正式分开。

## 目标

这一轮要完成三件事：

- `api-server` 按 `public` / `admin` 两个命名空间拆分路由
- `web` 只消费公开接口
- `admin` 只消费后台接口，并让后台读取接口也纳入权限控制

## 路由策略

### 公开接口

公开接口统一走：

- `/api/public/site-settings`
- `/api/public/resume`
- `/api/public/projects`
- `/api/public/translations`

这些接口面向 `web`，只承载公开展示所需的数据。

### 后台接口

后台接口统一走：

- `/api/admin/auth/*`
- `/api/admin/users`
- `/api/admin/translations`
- `/api/admin/resume`
- `/api/admin/projects`
- `/api/admin/site-settings`

这些接口面向 `admin`，承担后台读取、编辑、登录态和权限校验。

## 权限策略

本轮会把后台读取接口也收进权限边界：

- `GET /api/admin/users` -> `user.read`
- `GET /api/admin/translations` -> `translation.read`
- `GET /api/admin/resume` -> `resume.read`
- `GET /api/admin/projects` -> `project.read`
- `GET /api/admin/site-settings` -> `site.read`

这样之后，后台“能不能进入页面”和“能不能读到接口数据”会继续保持一致。

## SDK 边界

为了避免 `web` 误用后台写接口，本轮会新增单独的公开内容客户端。

也就是说：

- `web` 使用 `public content api client`
- `admin` 继续使用带鉴权与写操作能力的后台客户端

这样路径边界会进一步落到调用层。
