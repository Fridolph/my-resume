# P6-2：发布时间、更新人、审核人字段设计

## 背景

`P6-1` 已经完成统一状态流转规则，但当前内容工作流还缺少“谁做了什么”的基础追踪能力。

现在系统虽然知道：

- 内容当前处于什么状态
- 哪些流转是允许的
- 非法流转会被后端拒绝

但还不知道：

- 最近是谁更新了这条内容
- 最终是谁把内容审核并发布出去
- 内容是什么时间真正进入发布态的

这会导致后台虽然已经开始具备工作流雏形，但还没有形成最基础的审计可见性。

因此，`P6-2` 的目标，是为核心内容模块补齐三类基础审计字段：

- `updatedBy`
- `reviewedBy`
- `publishedAt`

## 设计目标

本轮重点完成：

- 为文案、简历、项目补齐审计字段
- 让字段在数据库中持久化
- 让后台更新接口自动写入这些字段
- 让后台页面可以直接看到这些字段

本轮仍然不做：

- 详细历史版本
- 审核意见
- 多人审批链
- 版本对比

也就是说，`P6-2` 解决的是“基础审计字段”问题，而不是完整审计日志问题。

## 适用模块

本轮继续聚焦三类内容模块：

- `TranslationRecord`
- `ResumeDocument`
- `ProjectRecord`

## 字段设计

### `updatedBy`

表示最近一次把这条内容写入数据库的操作者。

设计为：

- `id`
- `name`
- `email`

使用对象而不是单一字符串，有两个好处：

- 后台展示更直观
- 后续如果要补用户详情页或变更追踪，不需要再回查一次用户表才能知道是谁

### `reviewedBy`

表示最近一次把内容推进到 `published` 的审核人 / 发布人。

当前阶段把“审核通过并发布”视为同一动作，因此：

- 当状态进入 `published` 时，写入当前操作者
- 当内容离开 `published` 时，先清空该字段

这样语义会保持一致：

- 只要当前不是发布态，就不保留“当前审核人”的概念

### `publishedAt`

表示内容最近一次进入 `published` 状态的时间。

当前规则：

- 进入 `published` 时写入当前时间
- 保持 `published` 不变时沿用原时间
- 离开 `published` 时清空

这样字段含义就保持为：

- 当前这次发布何时生效

## 存储策略

本轮使用 SQLite + Drizzle ORM，因此字段会直接加到对应表：

- `translations`
- `resume_documents`
- `projects`

其中：

- `updated_by`：`text`，存储 JSON 字符串
- `reviewed_by`：`text`，存储 JSON 字符串
- `published_at`：`text`，存储 ISO 时间字符串

这样做的原因是：

- 当前后台读取审计人信息时，直接拿到快照即可
- 不需要为了展示名字再联查 `users`
- 更适合当前阶段的轻量内容工作流

## 写入规则

### 普通编辑

当内容被保存，但并没有进入发布态时：

- `updatedBy` = 当前操作者
- `reviewedBy` = `null`
- `publishedAt` = `null`

### 审核发布

当内容从 `reviewing` 进入 `published` 时：

- `updatedBy` = 当前操作者
- `reviewedBy` = 当前操作者
- `publishedAt` = 当前时间

### 已发布内容继续保存

当内容已经是 `published`，并继续保存且状态保持 `published`：

- `updatedBy` = 当前操作者
- `reviewedBy` = 保留原值
- `publishedAt` = 保留原值

这样就能区分：

- 最近是谁改了内容
- 当前这次发布是谁批准的
- 当前发布何时生效

### 撤回或归档

当内容从 `published` 撤回到 `draft`，或转入 `archived`：

- `updatedBy` = 当前操作者
- `reviewedBy` = `null`
- `publishedAt` = `null`

## 接口策略

本轮会继续沿用 `P5-6` 的边界：

- `admin` 接口负责完整审计信息返回
- `public` 接口即使返回这些字段，也不会被前台使用

这样可以先保证一份统一结构在三端都兼容，不急着在本轮做专门的公开 DTO 裁剪。

## 后台页面策略

本轮后台页面会直接展示：

- 最近更新人
- 审核人
- 发布时间

其中：

- 文案页在列表卡片中展示
- 项目页在卡片与编辑区展示
- 简历页在状态区的审计信息卡片展示

这样这三个字段不再只是落库字段，而是真正可见、可验证的管理后台信息。

## 完成标准

`P6-2` 完成后，应满足：

- 三类核心内容具备 `updatedBy`、`reviewedBy`、`publishedAt`
- 字段写入与状态流转逻辑一致
- 后台接口返回这些字段
- 后台页面可以看到这些字段
- 数据库迁移可在本地成功应用
