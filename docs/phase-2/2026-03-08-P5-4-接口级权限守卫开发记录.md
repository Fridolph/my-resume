# P5-4：接口级权限守卫开发记录

## 背景

`P5-3` 已经完成了真实登录态接入，但仅有登录还不够。

如果所有后台接口只要“登录就能调”，那角色和权限模型仍然没有真正落到服务端执行层。`P5-4` 的任务，就是把统一的角色权限规则正式落到 NestJS 接口层。

## 本次实现

### 1. 新增 auth cookie 工具

本次先把读取登录 cookie 的逻辑从 `AuthController` 中抽了出来，放入公共 auth 工具中。

这样做之后：

- 控制器和守卫可以共用同一套 cookie 读取逻辑
- 后续如果 cookie 名称或策略调整，只需要改一处

### 2. 新增权限元数据装饰器

本次新增了三类基础装饰器：

- `@Public()`
- `@RequirePermissions(...)`
- `@CurrentUser()`

其中这一轮最核心的是：

- `@Public()`：显式声明公开接口
- `@RequirePermissions(...)`：声明当前接口所需权限

这让后端权限控制开始从“硬编码 if 判断”升级为“可组合的声明式权限规则”。

### 3. 建立 `ApiAuthGuard`

本次新增 `ApiAuthGuard`，统一负责：

- 读取 session cookie
- 解析当前用户
- 检查接口是否公开
- 校验当前用户是否具备所需权限

这一步是整个 `P5-4` 的核心，因为后续所有模块都可以复用这一层守卫能力。

### 4. 给关键接口挂上权限控制

本次已经开始给关键接口接入权限控制。

#### 用户模块

- 用户列表 -> `user.read`
- 创建用户 -> `user.write`
- 更新用户 -> `user.write`

#### 翻译模块

- 更新翻译 -> `translation.write`

#### 简历模块

- 更新简历 -> `resume.write`

#### 项目模块

- 创建项目 -> `project.write`
- 更新项目 -> `project.write`
- 删除项目 -> `project.write`

#### 站点设置模块

- 更新站点设置 -> `site.write`

#### 鉴权模块

- `login` 保持公开
- `logout`、`me` 改为登录后才能调用

### 5. 调整 `viewer` 只读权限

为了让后续“只读账号访问后台列表页”的场景更完整，本次还把 `viewer` 角色补充了 `user.read`。

这样它就可以：

- 查看用户列表
- 但不能修改用户

这更符合只读角色的定位，也方便后续前端页面权限收口时进行验证。

## 本次实现的价值

这一轮的价值在于：角色与权限模型第一次真正开始“执行”。

从现在开始：

- 不同角色不仅在前端显示不同
- 后端接口也会真正按权限放行或拒绝

这意味着权限系统从“配置”走到了“运行态”。

## 本地自测

本次已完成以下验证：

- `pnpm --dir apps/api-server typecheck` 通过
- `pnpm --dir apps/admin typecheck` 通过
- `pnpm --dir apps/web typecheck` 通过
- `pnpm --dir packages/database typecheck` 通过
- 未登录访问 `GET /api/users` 返回 `401`
- `super-admin` 登录后访问 `GET /api/users` 返回 `200`
- `viewer` 登录后访问 `GET /api/users` 返回 `200`，符合只读角色预期
- `viewer` 登录后访问 `POST /api/users` 返回 `403`，并提示缺少 `user.write` 权限
- `GET /api/projects` 仍保持公开可访问，不影响当前 `web` 站点链路

这次还额外验证了一点：当前后端守卫已经可以稳定地区分三种状态——公开接口、未登录、已登录但权限不足。
