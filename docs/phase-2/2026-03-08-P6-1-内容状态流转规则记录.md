# P6-1：内容状态流转规则记录

## 背景

`P6` 的第一步，不是直接上版本系统，而是先把内容状态流转规则统一下来。

原因很直接：

- 如果状态机本身没有被正式定义
- 如果前端和后端对“允许切换什么状态”理解不同
- 如果接口没有能力拒绝非法流转

那么后续继续做审核人、发布时间、版本快照时，就会不断返工。

所以这一轮先把“状态流转规则”做成共享能力。

## 本次实现

### 1. 在共享层定义统一状态机

本次把 `PublishStatus` 的统一流转规则收口到了 `packages/types/src/index.ts`。

新增内容包括：

- `publishStatusTransitionMap`
- `publishStatusLabels`
- `getAllowedPublishStatusTransitions()`
- `canTransitionPublishStatus()`

这意味着从这一轮开始：

- 后端服务
- 后台页面

都可以直接复用同一份状态机规则。

### 2. 后端在服务层拒绝非法状态流转

本次把状态流转校验加到了三个服务层：

- `apps/api-server/src/modules/translations/translations.service.ts`
- `apps/api-server/src/modules/resume/resume.service.ts`
- `apps/api-server/src/modules/projects/projects.service.ts`

当请求尝试执行非法流转时，接口会直接返回 `400`。

例如：

- 已发布项目不能直接切回审核中
- 已发布文案不能直接切回审核中
- 已归档简历不能直接直接发布

这一步让状态流转开始真正受后端约束，而不只是前端约定。

### 3. 后台抽出统一内容工作流辅助 composable

本次新增：

- `apps/admin/app/composables/useContentWorkflow.ts`

这个 composable 负责提供：

- 状态标签
- 状态颜色
- 当前状态可选择的目标状态集合
- 当前状态的主流转动作

这样之后，后台页面就不需要各自维护一套状态判断逻辑。

### 4. 文案页改成主流转动作 + 受控状态选择

本次文案页主要做了两件事：

- 列表按钮不再简单地“发布 / 转草稿”二选一
- 编辑区状态选择器不再展示全部状态，而是只展示当前允许的合法流转目标

同时，状态徽标也切换为统一中文标签和统一颜色。

### 5. 简历页与项目页改成受控状态切换

本次简历页与项目页的状态选择器都改为只展示：

- 当前状态
- 当前状态允许切换到的目标状态

并且在状态保存失败时，页面会自动回滚到原状态，避免出现：

- UI 看起来已经切换成功
- 但后端实际拒绝了这次流转

这种前后端不一致的问题。

## 本次实现的价值

这一轮的价值，不只是“多了几个 if 判断”。

真正的变化在于：

- 状态机第一次被正式定义为共享规则
- 前后端第一次开始严格使用同一套状态流转边界
- 后续版本系统、审核字段、发布时间字段有了稳定基础

换句话说，`P6-1` 是内容工作流真正开始成形的起点。

### 接口流转验证

#### 项目

已发布项目尝试执行非法流转：

- `published -> reviewing` -> `400`

接口返回明确错误信息：

- `项目状态不允许从 已发布 直接流转到 审核中。`

#### 文案

已发布文案尝试执行非法流转：

- `published -> reviewing` -> `400`

接口返回明确错误信息：

- `文案状态不允许从 已发布 直接流转到 审核中。`

#### 简历

本次对简历做了完整流转验证：

- `published -> draft` -> `200`
- `draft -> reviewing` -> `200`
- `reviewing -> archived` -> `200`
- `archived -> published` -> `400`
- `archived -> draft` -> `200`

最终已把简历状态恢复到：

- `draft`

这样不会把联调数据长期停留在不合适的中间状态。

### 页面验证

页面仍然可以正常访问：

- `http://127.0.0.1:3000/` -> `200`
- `http://127.0.0.1:3002/login` -> `200`

后台页面的状态徽标、状态下拉和主流转动作均已切换到新的规则体系。

## 一个阶段性结论

做到这里，项目的内容工作流已经不再只是“有个状态字段”。

它开始具备：

- 统一状态机
- 后端强校验
- 前端受控交互
- 异常回滚能力

也就是说，`P6` 已经不再停留在概念阶段，而是进入真正可扩展的实现阶段。
