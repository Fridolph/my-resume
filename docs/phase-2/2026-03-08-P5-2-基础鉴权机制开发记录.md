# P5-2：基础鉴权机制开发记录

## 背景

`P5-1` 已经把角色和权限模型统一到了共享层，但那还只是“权限规则统一”，并不代表后端已经知道“当前是谁在发请求”。

所以这一轮 `P5-2` 的核心任务，是让 `api-server` 先拥有正式的身份识别能力。

## 本次实现

### 1. 为用户表补上密码哈希字段

本次为 `users` 表新增了 `password_hash` 字段，并在数据库迁移中落库。

这样做之后，登录校验开始进入后端，而不再依赖前端 mock 账号逻辑。

### 2. 新增 `auth_sessions` 表

为了支持正式登录态，这次新增了 `auth_sessions` 表，用来保存：

- session id
- user id
- token hash
- 过期时间
- 创建时间

会话信息落到 SQLite 后，后续就可以继续扩展：

- 权限守卫
- 会话失效
- 多端登录控制
- 登录审计

### 3. 在数据库层补齐 auth 能力

本次在 `@repo/database` 中新增了 auth 相关能力，包括：

- 密码哈希与校验
- 用户认证
- 创建会话
- 按 token 解析当前用户
- 登出时撤销会话
- 清理过期会话

这让认证逻辑不需要直接写死在 NestJS 控制器中，而是和当前数据库层结构保持一致。

### 4. 建立 `AuthModule`

本次在 `apps/api-server` 中新增了：

- `AuthModule`
- `AuthService`
- `AuthController`
- `auth.schema`

并提供三个基础接口：

- `POST /api/auth/login`
- `POST /api/auth/logout`
- `GET /api/auth/me`

### 5. 采用 Cookie Session 作为当前阶段方案

这次没有直接引入 JWT，而是选择了：

- 服务端签发 session token
- 浏览器通过 `httpOnly cookie` 保存 token
- 后续请求由 `api-server` 解析 cookie 获取当前用户

这是更适合当前项目节奏的选择，因为下一步 `admin` 接入真实登录态时，只需要直接消费这套接口即可。

### 6. SDK 预留 auth client

本次还在 `packages/sdk` 中补充了 auth 相关 client 能力，并让基础请求默认支持 `credentials: 'include'`。

这样后续 `admin` 在 `P5-3` 接真实登录时，不需要再重新设计 SDK 请求基类。

## 本次实现的价值

这一轮最重要的价值，不是“用户已经能完整登录后台”，而是：

- `api-server` 已经开始具备正式登录态
- 后端第一次真正能识别当前用户
- 前端 mock 登录可以开始退出主链路
- 下一阶段接真实登录态已经有可复用接口

## 本地自测

本次已完成以下验证：

- `pnpm --dir apps/api-server typecheck` 通过
- `pnpm --dir apps/admin typecheck` 通过
- `pnpm --dir apps/web typecheck` 通过
- `pnpm --dir packages/database typecheck` 通过
- `POST http://127.0.0.1:3011/api/auth/login` 可返回 `200 OK`，并写入 `platform_admin_session` cookie
- `GET http://127.0.0.1:3011/api/auth/me` 在带 cookie 情况下可返回当前 `super-admin` 会话
- `POST http://127.0.0.1:3011/api/auth/logout` 可清除当前 cookie 与服务端会话
- 登出后再次请求 `GET /api/auth/me` 返回 `401`，符合预期

## 排障记录

这次联调中还遇到了一次典型的历史数据库问题：

- 新增的 `0005_auth_sessions.sql` 已加入仓库后，当前本地 SQLite 仍缺失 `auth_sessions` 表
- 导致 `revokeExpiredSessions()` 第一次执行时报 `no such table: auth_sessions`

最终定位结果是：当前本地数据库属于持续演进环境，迁移记录与真实表结构出现了短暂不一致。

本次处理方式是：

- 保留正式迁移文件到仓库
- 同时把当前本地数据库补齐到最新结构
- 再重新执行 auth 链路验证

这也提醒我们，后续进入 `P7` 时，需要把数据库迁移验证纳入更明确的启动与检查流程。
