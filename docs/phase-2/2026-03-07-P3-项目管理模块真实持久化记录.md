# P3：项目管理模块真实持久化记录

## 背景

在 `P3` 已完成用户管理、`i18n` 文案管理和简历管理之后，第二阶段已经基本跑通了后台核心内容的主干链路。接下来继续按顺序推进时，最自然的一步就是项目管理模块。

项目模块的重要性在于，它不仅是后台的核心内容模块之一，也直接对应未来 `P4` 前台项目列表页和项目详情页的真实数据来源。如果后台项目数据仍然停留在本地 `useState`，那么前台项目页后续即使要接正式 API，也没有稳定的数据上游。

因此，本次 `P3` 的目标很明确：把后台项目管理从本地状态正式迁移到 `NestJS + SQLite + Drizzle ORM` 这条统一主链路上。

## 本次目标

本次项目模块迁移的目标是：

- 建立 `projects` 表结构
- 建立共享数据库仓储函数
- 在 `apps/api-server` 中增加 `ProjectsModule`
- 提供项目列表、创建、更新、删除接口
- 在 `packages/sdk` 中补充项目 API Client
- 将 `apps/admin` 的项目管理页切换到真实 API

这一步完成后，后台项目列表、项目编辑、项目排序、标签维护和删除操作都将进入正式持久化链路。

## 为什么本次选择“单表 + JSON 字段”模型

当前项目数据的结构包含两类内容：

- 主字段：`slug`、`status`、`sortOrder`、`cover`、`externalUrl`、`updatedAt`
- 结构化字段：`tags`、多语言 `locales`

本次没有把标签、多语言内容和项目基础信息拆成多张子表，而是先采用单表 + JSON 字段的方式承载：

- `tags` 存为 JSON 字符串
- `locales` 存为 JSON 字符串

这样做的原因和简历模块一致：当前阶段的优先目标是快速建立正式数据链路，而不是过早把建模复杂度做高。等到后续要做项目详情扩展、版本管理或更复杂的公开查询时，再继续拆表也完全来得及。

因此，本次方案是当前阶段最符合成本与收益平衡的一种实现。

## 本次实现

### 1. 建立 `projects` 表结构

本次在 `packages/database` 中新增了项目表 schema，并补充了对应迁移文件。

当前表结构包含：

- `id`
- `slug`
- `status`
- `sort_order`
- `cover`
- `external_url`
- `tags`
- `locales`
- `updated_at`

其中 `slug` 被设置为唯一索引，用于保证每个项目的公开路由标识唯一。

### 2. 建立项目仓储与默认种子

本次在共享数据库包中新增了项目仓储能力，主要包括：

- 默认项目种子写入
- 项目列表读取
- 创建项目
- 更新项目
- 删除项目

默认种子延续了后台现有演示内容，包括：

- `resume-platform`
- `design-system-draft`
- `i18n-content-hub`

这让项目页迁移后仍能保持原有的演示体验，同时又切换到了真实数据库层。

### 3. 增加排序规范化逻辑

这次项目模块迁移里，还额外补了一项非常重要的收口：排序规范化。

原因是项目数据允许维护 `sortOrder`，如果只允许前端直接写入，而后端不做统一收口，就可能出现以下问题：

- 两个项目出现相同的 `sortOrder`
- 删除项目后排序断层
- 手动修改排序后列表顺序不稳定

因此，本次在数据库仓储层增加了排序规范化逻辑：

- 在项目创建后重新整理排序
- 在项目更新后重新整理排序
- 在项目删除后重新整理排序
- 若读取时发现已有重复排序值，也会自动修正

这保证了 `projects` 表在任何时刻都能维持稳定、连续、唯一的排序值。

### 4. 在 `apps/api-server` 中建立 `ProjectsModule`

本次新增了：

- `ProjectsModule`
- `ProjectsController`
- `ProjectsService`
- 项目请求校验 schema

并补充了以下正式接口：

- `GET /api/projects`
- `POST /api/projects`
- `PUT /api/projects/:projectId`
- `DELETE /api/projects/:projectId`

这意味着后台项目管理已经正式进入 NestJS 模块体系，而不再只是页面本地逻辑。

### 5. 在 `packages/sdk` 中增加项目 API Client

为了保持前端访问 API 的方式一致，本次在 `packages/sdk` 中补充了项目模块客户端能力，并继续沿用统一平台客户端的扩展方式。

当前前端已经可以通过统一 SDK 完成：

- 项目列表查询
- 项目创建
- 项目更新
- 项目删除

这也为后续 `P4` 的前台项目页接入正式 API 提前铺好了通路。

### 6. 将 `apps/admin` 项目管理页切换为真实 API

本次已经将后台项目页从本地 `useState` 驱动，迁移为通过 API Server 驱动。

当前页面已经完成以下变化：

- 页面加载时通过真实接口读取项目列表
- 新建项目通过 `POST /api/projects` 持久化
- 编辑项目通过 `PUT /api/projects/:projectId` 持久化
- 删除项目通过 `DELETE /api/projects/:projectId` 持久化
- 项目排序调整会同步回真实接口
- 多语言编辑、标签维护、状态切换仍保留当前交互体验

这说明本轮改造依然遵守了既定策略：优先替换数据源，不重写现有后台页面结构。
