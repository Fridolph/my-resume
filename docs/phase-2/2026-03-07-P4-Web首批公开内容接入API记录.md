# P4：Web 首批公开内容接入 API 记录

## 背景

随着 `P3` 四个核心内容模块和统一响应结构全部完成，第二阶段已经不再缺后台数据能力。这个时候进入 `P4`，最自然的工作就是：让 `web` 公开站点开始正式消费 `api-server` 的真实内容。

在进入本轮之前，`web` 已经具备了两类能力：

- 站点设置已经通过 `api-server` 提供
- 首页、简历页、项目页仍主要依赖 `packages/sdk` 中的 mock 内容 DTO

这意味着前台虽然已经有了不错的展示结构，但真正的公开内容仍然没有和后台管理形成完整闭环。

因此，本次 `P4` 的第一小步，不是一次性替换整个站点，而是优先把最成熟、收益最高的三类页面切到真实 API：

- 简历页
- 项目列表页
- 项目详情页

## 本次目标

本次首批替换的目标是：

- 将 `web` 简历页改为读取 `GET /api/resume`
- 将 `web` 项目列表页改为读取 `GET /api/projects`
- 将 `web` 项目详情页改为从真实项目列表中解析项目详情
- 保持现有页面 UI 结构不大改，只替换内容来源
- 完成本地联调验证，确认 `web` 已正式消费后台真实内容

这样做的目的是先把后台和前台真正串起来，而不是继续停留在结构演示层。

## 为什么这一轮先不动首页

这次没有先处理首页，而是先处理简历页和项目页，原因很简单：

- 简历页已经在 `P3` 完成后台真实持久化
- 项目模块也已经在 `P3` 完成后台真实持久化
- 首页当前仍混合了承担站点导览、阶段说明和入口聚合的职责

换句话说，简历和项目内容已经有成熟稳定的数据源，而首页还更偏向“站点入口编排”。

因此，先替换简历和项目页，能最快拿到真正的前后台闭环成果，同时避免首页在这个节点上承担过多结构改造风险。

## 本次实现

### 1. 改造 `useWebContent`

本次没有直接大改页面，而是优先改造 `apps/web/app/composables/useWebContent.ts`，把真实 API 接入集中收口在这一层。

这样做的价值在于：

- 页面仍然保持原有的 `useAsyncData` 消费习惯
- 页面层几乎不需要知道底层是 mock 还是 API
- 后续继续扩展首页和更多公开内容时，也能沿用同一思路

当前这层已经开始通过 `usePlatformApiClient()` 调用正式 API。

### 2. 简历页切换到 `api-server`

本次 `web` 简历页已经改为通过 `GET /api/resume` 读取真实简历文档。

然后在 `useWebContent` 中将后台返回的 `ResumeDocument` 转换为当前页面仍然需要的展示 DTO：

- `intro`
- `sections`

这意味着：

- 当前简历页展示的基础信息、教育经历、工作经历、技能分组、联系方式
- 已经不再来自 SDK mock
- 而是来自后台真实简历文档

这一步非常关键，因为它标志着“后台维护简历 -> 前台展示简历”这条链路终于闭环了。

### 3. 项目列表页切换到 `api-server`

本次 `web` 项目列表页已经改为通过 `GET /api/projects` 读取真实项目列表。

在前台展示层，本次增加了一条公开内容约束：

- 仅展示 `published` 状态的项目

这让前台项目页和后台内容管理之间开始出现真实的内容边界，而不再只是“后台有什么，前台就全展示什么”。

同时，项目列表页仍然保留了原有卡片结构，只是把数据来源替换为了 API。

### 4. 项目详情页切换到真实数据

本次 `web` 项目详情页没有单独新增一个详情接口，而是先基于 `GET /api/projects` 返回的真实项目列表，在前台按 `slug` 查找对应的已发布项目，再生成当前页面需要的详情 DTO。

这是一个很适合当前阶段的过渡方案，因为它具备几个好处：

- 不需要立刻新增新的后端详情接口
- 可以快速验证前台项目详情已经摆脱 mock 数据
- 后续如果要做真正的详情接口，也只需要替换这层映射逻辑

因此，这一轮实现虽然没有增加新的详情路由接口，但已经完成了“前台项目详情来自真实后台数据”的目标。

### 5. 保持页面 UI 结构稳定

本次改造最重要的一点，是尽量不去重做 `web` 页面本身。

当前：

- `resume.vue`
- `projects/index.vue`
- `projects/[slug].vue`

都仍然保持原有的展示结构和 SEO 设置方式。

变化主要集中在数据获取层。这种做法的优势是：

- 风险更低
- 回归更容易
- 更适合当前阶段逐步推进

## 当前结果

本次替换完成之后，`P4` 已经正式启动，并且拿到了第一批真实成果：

- `web` 简历页已接入真实 API
- `web` 项目列表页已接入真实 API
- `web` 项目详情页已接入真实 API

这让当前仓库第一次真正具备了“后台维护内容，前台消费内容”的正式特征。

## 这次工作的价值

本次 `P4` 首批替换的价值主要有三点。

第一，前后台真正开始打通了。

到这一步为止，后台不再只是管理端演示，而是真正影响前台公开站点展示结果。

第二，公开内容边界开始出现了。

前台项目列表只展示 `published` 内容，这为后续做更完整的发布流与审核流打下了基础。

第三，替换路径已经明确了。

当前 `web` 已经证明：

- 页面不需要重做
- 只要在数据获取层收口
- 就能逐步把 mock 内容替换为真实 API

这会让后续首页和更多公开内容替换更顺利。

## 下一步

在 `P4` 里，下一步建议继续按价值顺序推进：

- 首页接入真实公开内容
- 多语言公开文案开始接入后台翻译结果
- 项目详情能力继续从“列表映射”升级为真正的详情接口（如有需要）

其中下一项最优先建议的是：首页接入真实公开内容。因为它是整个公开站点的入口，一旦替换完成，`web` 的主要页面就基本都完成正式 API 化了。
