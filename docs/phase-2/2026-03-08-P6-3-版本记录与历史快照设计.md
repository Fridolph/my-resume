# P6-3：版本记录与历史快照设计

## 背景

`P6-1` 已经为内容模块建立了统一状态流转规则，`P6-2` 又补齐了 `updatedBy`、`reviewedBy` 与 `publishedAt` 三类基础审计字段。

做到这里，系统已经知道：

- 内容当前处于什么状态
- 哪些状态流转是允许的
- 最近一次是谁更新了内容
- 当前这次发布是谁完成的
- 当前发布是什么时间生效的

但是系统仍然缺少“版本维度”。

也就是说，后台虽然知道“这条内容现在长什么样”，但还不知道：

- 这条内容过去长什么样
- 一次更新之后形成了第几个版本
- 某次发布之前和发布之后的内容是否发生了变化
- 后续如果要做回滚，应该回到哪一个历史快照

因此，`P6-3` 的目标，是先把“历史快照”落地，为后续 `P6-4` 的回滚策略提供数据基础。

## 本轮目标

本轮聚焦“版本快照能力”的第一版落地，只做最小可用闭环：

- 为文案、简历、项目建立统一历史快照表
- 每次创建或更新内容时自动记录快照
- 每条快照都带上版本号、状态、操作者和时间
- 后台可以查看对应内容的历史版本列表

本轮暂时不做：

- 版本 Diff 视图
- 一键回滚按钮
- 跨版本字段级比较
- 审核意见与审批备注
- 发布批次管理

也就是说，`P6-3` 先解决“历史留痕”和“可查看”，暂时不解决“历史恢复”。

## 适用范围

本轮继续覆盖三类核心内容：

- `TranslationRecord`
- `ResumeDocument`
- `ProjectRecord`

为了避免每个模块都维护一套独立历史逻辑，本轮采用统一快照模型。

## 为什么选择统一快照表

当前有两种常见做法：

### 方案一：每个模块独立一张历史表

例如：

- `translation_versions`
- `resume_versions`
- `project_versions`

优点：

- 表结构直观
- 每个模块可以独立扩展字段

缺点：

- 三套版本逻辑要重复维护
- 后续想做统一历史中心时不方便聚合
- Admin 端也要为每类模块维护不同的历史查询模型

### 方案二：统一内容版本表

例如用一张 `content_versions` 表，记录：

- 模块类型
- 内容实体 ID
- 版本号
- 当前状态
- 完整快照
- 操作者
- 创建时间

优点：

- 版本逻辑统一
- 后续更容易做统一历史中心或审计视图
- Admin 与 API 端更容易复用一套查询模型

缺点：

- 快照字段需要 JSON 化存储
- 模块特有字段不会单独拆列

结合当前阶段目标，本轮优先选择 **统一内容版本表**。原因是：

- 当前阶段更需要稳定一致的能力，而不是模块级定制
- SQLite + Drizzle 已经在主表中使用 JSON 字符串承载复杂结构
- 后续 `P6-4` 的回滚策略也更适合站在统一快照模型之上实现

## 数据结构设计

本轮新增统一版本实体，核心字段如下：

- `id`：版本记录 ID
- `moduleType`：模块类型，取值为 `translation`、`resume`、`project`
- `entityId`：主内容 ID
- `version`：从 `1` 开始递增的版本号
- `status`：该次快照对应的内容状态
- `changeType`：本次快照的形成原因
- `snapshot`：完整内容快照
- `createdBy`：本次快照创建人
- `createdAt`：本次快照创建时间

其中：

### `version`

同一个 `moduleType + entityId` 下单独递增。

例如：

- 某条文案的第 1、2、3 版
- 当前简历文档的第 1、2、3 版
- 某个项目的第 1、2、3 版

不同实体之间互不影响。

### `snapshot`

本轮直接保存完整内容快照，而不是只保存 Diff。

这样做的原因是：

- 回滚时不需要再从多个 patch 重放
- 后台查看版本时可以直接展示快照元信息
- 当前数据量较小，完整快照更简单稳定

### `changeType`

本轮建议统一收口为三类：

- `seed`：初始化种子生成的首版快照
- `create`：新增内容时生成的版本
- `update`：编辑内容或状态变化时生成的版本

这里先不单独拆“状态变化”和“普通字段变化”，因为两者本质上都属于一次新的内容版本。后续若确实有展示需求，再细化分类即可。

## 写入时机

本轮采用“**每次成功写入主表后立即生成快照**”的策略。

具体规则如下：

### 文案

- 初始化种子数据时，为每条文案建立首版快照
- 每次更新文案时，新增一个版本

### 简历

- 首次种子文档落库时建立首版快照
- 每次保存简历时，新增一个版本

### 项目

- 初始化种子项目时，为每个项目建立首版快照
- 新建项目时，写入 `create` 版本
- 更新项目时，新增一个版本

## 查询策略

后台历史查询采用按实体查询，不做全局历史列表：

- 文案：`GET /api/admin/translations/:translationId/versions`
- 简历：`GET /api/admin/resume/versions`
- 项目：`GET /api/admin/projects/:projectId/versions`

返回结果按版本号倒序排列，便于后台优先看到最近一次修改。

## 后台展示策略

本轮 Admin 端只做“查看历史”，不做回滚操作。

### 文案页

在每条文案卡片中提供“查看版本”入口，按需加载该条文案的历史记录。

### 简历页

在简历管理页增加“版本历史”卡片，展示最近若干次版本。

### 项目页

在项目编辑区域增加“版本历史”卡片，展示当前编辑项目的历史记录。

后台展示以元信息为主：

- 版本号
- 状态
- 变更类型
- 创建人
- 创建时间

本轮不直接在列表中展开完整 JSON 内容，以免影响管理页可读性。

## 与 `P6-4` 的衔接关系

`P6-3` 落地后，`P6-4` 的回滚策略就有了数据基础。

届时回滚的核心思路会变成：

1. 读取某条历史快照
2. 将该快照恢复为当前主表内容
3. 同时再生成一条新的“恢复版本”快照

因此，`P6-3` 本质上是在为 `P6-4` 搭建“可恢复的数据底座”。

## 完成标准

`P6-3` 完成后，应满足：

- 三类核心内容具备统一历史快照表
- 成功写入主表后会自动生成新版本
- 初始化数据拥有首版快照
- 后台接口可以按实体读取版本列表
- Admin 页面可以查看版本历史
- 本地类型检查和联调通过
