# P3：i18n 文案管理模块真实持久化记录

## 背景

在 `P3` 的第一个子任务里，用户管理模块已经完成了从 Admin 页面到 API Server，再到 SQLite + Drizzle ORM 的正式落盘链路。这说明第二阶段的核心目标已经不再是“验证架构能否工作”，而是开始把后台真正需要长期维护的内容模块，逐个接入统一技术主线。

在这个阶段里，文案管理是最值得优先推进的第二个模块。原因很直接：它既是后台高频编辑内容，也是后续 Web 公开站点接入后台真实内容的关键前置条件。如果文案管理仍停留在本地状态或 mock 数据层，那么后面的简历、项目和前台 i18n 接入都会缺少统一内容来源。

## 本次目标

本次 `P3` 文案模块迁移的目标是：

- 建立 `translations` 表结构
- 建立共享数据库仓储函数
- 在 `apps/api-server` 中增加 `TranslationsModule`
- 提供文案列表与更新接口
- 在 `packages/sdk` 中补充文案 API Client
- 将 `apps/admin` 的文案管理页切换到真实 API

这一步完成后，后台文案模块就正式进入可持久化、可联调、可落盘的状态。

## 本次实现

### 1. 建立 `translations` 表结构

本次在 `packages/database` 中新增了翻译表 schema，并补充了对应迁移文件。

当前翻译表包含以下核心字段：

- `id`
- `namespace`
- `key`
- `locale`
- `value`
- `status`
- `missing`
- `updated_at`

其中 `namespace + key + locale` 被设置为唯一索引，用来保证同一命名空间下的同一个文案键，在同一种语言里只存在一条正式记录。这一点对后续做后台批量编辑、导入导出或发布流都很重要。

### 2. 建立翻译仓储与默认种子

本次在共享数据库包中新增了翻译仓储能力，主要包括：

- 默认文案种子写入
- 文案列表读取
- 单条文案更新

为了和当前后台演示阶段保持一致，默认种子沿用了现有页面里已经验证过的一组数据，包括：

- `common.nav.home`
- `resume.hero.title`
- `project.list.title`
- `seo.home.description`

同时，`missing` 字段不再交给前端单独判断，而是由数据库层在写入时根据 `value` 是否为空自动计算。这样可以让前端页面始终拿到明确状态，减少前后端对“缺失文案”的重复判断逻辑。

### 3. 在 `apps/api-server` 中建立 `TranslationsModule`

本次新增了：

- `TranslationsModule`
- `TranslationsController`
- `TranslationsService`
- 文案请求校验 schema

并补充了以下正式接口：

- `GET /api/translations`
- `PUT /api/translations/:translationId`

这意味着后台文案管理已经正式进入 NestJS 模块体系，而不是继续停留在页面本地状态里。

### 4. 在 `packages/sdk` 中增加文案 API Client

为了保持前端访问 API 的方式一致，本次在 `packages/sdk` 中补充了文案模块请求能力，并让平台 API Client 继续沿用统一入口。

当前前端已经可以通过统一 SDK 完成：

- 文案列表查询
- 单条文案更新
- 发布状态切换所需的接口调用

这样做的价值在于，Admin 页面不需要散落多份 `fetch` 细节，后续无论是扩展简历模块还是项目模块，都可以继续按同样结构前进。

### 5. 将 `apps/admin` 文案管理页切换为真实 API

本次已经将后台文案页从本地 `useState` 驱动，迁移为通过 API Server 驱动。

当前页面已经完成以下变化：

- 页面加载时通过真实接口读取文案列表
- 编辑文案时通过 `PUT /api/translations/:translationId` 持久化
- 发布 / 草稿切换通过正式接口更新
- 页面仍保留 namespace、locale、状态和关键字筛选能力
- 语言覆盖率统计基于真实数据集计算

这说明这一轮迁移仍然遵守了我们前面验证成功的原则：优先替换数据源，而不是重写整页 UI。

## 当前结果

P3 的第二个核心模块已经完成，当前仓库已经拥有一条新的正式业务链路：

- Admin 文案页通过 SDK 请求 API Server
- API Server 的 `TranslationsModule` 处理列表与更新
- Drizzle ORM 将数据落盘到 SQLite
- 文案状态和缺失状态会和数据库保持一致

这意味着第二阶段已经从“用户与配置模块接入”继续向“内容模块正式持久化”迈进了一步。

## 这次迁移的价值

文案模块的重要性不只在于它本身，还在于它为后续两个方向打好了基础。

第一，它为后台接下来的内容模块迁移继续验证了统一模板：

- 数据库先建表
- API Server 建模块
- SDK 补客户端
- Admin 页面切真实接口
- SQLite 落盘验证

第二，它为 Web 前台后续接入真实多语言内容提供了正式入口。等到公开页面开始从后台拉取文案时，就不需要再回头重做后台数据层。

## 下一步

在 `P3` 里，用户管理和文案管理都已经完成。接下来建议继续按既定顺序推进：

- 简历模块表结构与接口
- 项目模块表结构与接口
- 统一响应结构与错误处理收口
- Admin 页面剩余模块切换为 API 驱动

其中下一项最优先建议的是：简历模块。因为它同时连接 Admin 编辑后台与 Web 展示前台，是后续三端打通的关键桥梁。
