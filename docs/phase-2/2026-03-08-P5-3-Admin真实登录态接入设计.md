# P5-3：Admin 真实登录态接入设计

## 背景

`P5-2` 已经在 `api-server` 建立了基础鉴权机制：

- `POST /api/auth/login`
- `POST /api/auth/logout`
- `GET /api/auth/me`

但当前 `admin` 端仍然使用本地 mock 登录逻辑。这样会造成一个明显断层：

- 后端已经具备正式登录态
- 管理后台却还没有真正使用它

因此，`P5-3` 的目标就是把 `admin` 登录、登出、页面守卫与会话恢复全部切到真实 auth API。

## 设计目标

这一阶段要解决四件事：

- 登录页改为调用 `/api/auth/login`
- 页面初始化时通过 `/api/auth/me` 恢复会话
- 退出登录改为调用 `/api/auth/logout`
- `auth` / `guest` 中间件改为基于真实会话判断

## 会话策略

当前仍采用 `api-server` 的 cookie session 作为唯一真实登录态来源。

但为了让 `admin` 页面渲染和路由守卫更顺畅，本次会在前端保留一份轻量 session state 作为镜像缓存。它的作用是：

- 减少页面内部重复判断成本
- 让导航和页面组件直接读取当前用户信息
- 配合 middleware 快速判断当前路由是否应当放行

真正的会话来源仍然是：

- `platform_admin_session` cookie
- `GET /api/auth/me`

## SSR 与跨端 cookie 处理

由于 `admin` 与 `api-server` 运行在不同端口：

- `admin`: `localhost:3002`
- `api-server`: `localhost:3011`

因此本次需要确保两件事：

- 浏览器端请求 auth API 时带上 `credentials: 'include'`
- SSR 阶段由 `admin` 服务端把当前请求的 cookie 继续转发给 `api-server`

只有这样，登录后的页面刷新和服务端路由守卫才能正确识别当前用户。

## 中间件策略

### `auth` middleware

进入受保护页面前：

- 先调用 `refreshSession()`
- 如果仍然没有登录态，则跳转 `/login`

### `guest` middleware

进入登录页前：

- 先调用 `refreshSession()`
- 如果已经登录，则直接跳转后台首页

## 本阶段结果预期

完成 `P5-3` 后，`admin` 将具备：

- 真实登录
- 真实登出
- 刷新后仍可识别当前用户
- 路由守卫不再依赖 mock 登录态

这一步完成之后，下一步就可以进入：

- `P5-4`：接口级权限守卫
