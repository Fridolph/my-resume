# P7-3：环境变量与多端启动文档记录

## 背景

进入 `P7-3` 之前，第二阶段的三端项目其实已经可以正常联调：

- `apps/web` 可以读取公开接口
- `apps/admin` 可以完成登录和后台管理
- `apps/api-server` 可以作为统一后端运行
- `SQLite + Drizzle ORM` 已经成为正式数据方案

但这一阶段还有一个很典型的“工程后半程问题”：

> 功能都在，链路也能跑，但真正让别人接手时，仍然会不断问“这个变量要怎么配”“这个命令应该在哪跑”“这三个服务谁先起”。

这类问题不属于核心业务功能，却非常影响项目的可交付性。

所以 `P7-3` 的目标，是把第二阶段已经形成的三端统一架构，真正整理成一套新人可以直接照着操作的本地开发说明，并让代码配置与文档同步对齐。

## 本次实现

这次 `P7-3` 的工作分成两部分：

- 配置层收口
- 文档层收口

### 1. 为 Web / Admin 收口更清晰的环境变量命名

在这一轮之前：

- `apps/web` 和 `apps/admin` 都主要依赖同一个变量名：`NUXT_PUBLIC_API_BASE_URL`

这在“单个应用自己运行”时问题不大，但在三端并行的 monorepo 中，这个变量名其实不够清晰，因为：

- `web` 要的是公开接口地址
- `admin` 要的是后台管理接口地址

为了让变量语义更明确，这次分别引入了：

- `NUXT_PUBLIC_PUBLIC_API_BASE_URL`
- `NUXT_PUBLIC_ADMIN_API_BASE_URL`

同时为了不打断当前已有本地环境，这次还保留了对旧变量的兼容 fallback。

也就是说，现在代码层已经进入一个更稳的状态：

- 新文档推荐使用新变量名
- 旧本地环境仍然可以继续跑

### 2. 补齐统一环境变量示例文件

这次新增了：

- `.env.example`

它集中说明了当前第二阶段本地开发真正关键的变量：

- `PORT`
- `REPO_DATABASE_PATH`
- `NUXT_PUBLIC_PUBLIC_API_BASE_URL`
- `NUXT_PUBLIC_ADMIN_API_BASE_URL`

这样后续接手者至少不需要再从多个 `nuxt.config.ts` 和 `main.ts` 里手工找变量来源。

### 3. 让 Drizzle 配置与运行时数据库路径保持一致

这轮还顺手修正了一个很有价值的细节：

- `apps/api-server/drizzle.config.ts`

之前 `drizzle-kit` 默认固定读取：

- `data/platform.sqlite`

但运行时数据库其实已经支持：

- `REPO_DATABASE_PATH`

如果这两者不一致，就会带来一个隐患：

- 应用运行时连的是一个数据库
- `drizzle-kit` 工具链看的却是另一个数据库

这次已经把 `drizzle.config.ts` 对齐为优先读取 `REPO_DATABASE_PATH`，从而让：

- 运行时数据库
- Studio / status / 其他工具命令

真正指向同一份数据库文件。

### 4. 为根目录补齐数据库脚本入口

为了让根目录操作更顺手，这次在 `package.json` 中新增了：

- `db:studio`
- `db:status`

这样现在从仓库根目录就可以统一执行：

- `pnpm db:generate`
- `pnpm db:migrate`
- `pnpm db:studio`
- `pnpm db:status`

这会显著降低“命令到底在哪个目录执行”的心智负担。

### 5. 重写根目录与三端 README

这次把原先遗留的模板说明和旧项目说明系统性替换掉了。

当前已经重写：

- `README.md`
- `apps/web/README.md`
- `apps/admin/README.md`
- `apps/api-server/README.md`

这些文档现在会明确说明：

- 当前三端架构
- 默认端口
- 关键环境变量
- 本地启动方式
- 数据库命令
- 联调检查与常见排查项

这部分虽然不直接增加业务能力，但它会直接影响后续协作效率。

- `pnpm --dir apps/web typecheck`
- `pnpm --dir apps/admin typecheck`

### 数据库命令验证

以下命令已通过：

- `pnpm db:status`

输出结果为：

- `Everything's fine 🐶🔥`

这说明当前：

- 根目录新脚本可用
- `apps/api-server/drizzle.config.ts` 可正常解析
- 数据库路径与 Drizzle 配置已经保持一致

## 本次收口后的结果

做到这里，第二阶段的本地开发说明已经比之前清晰很多。

当前仓库已经具备：

- 一份统一的根目录项目说明
- 三端分别独立的 README
- 一份统一的 `.env.example`
- 更语义化的 Web / Admin API 地址变量命名
- 根目录统一数据库命令入口

从现在开始，新开发者只要按顺序执行：

1. `pnpm install`
2. `cp .env.example .env`
3. `pnpm db:migrate`
4. `pnpm dev:api`
5. `pnpm dev:web`
6. `pnpm dev:admin`

就能比较顺畅地完成本地初始化与联调。

## 一个阶段结论

`P7-3` 解决的不是“功能能不能用”，而是“项目能不能被稳定地继续开发”。

这一步做完以后，第二阶段已经不再只是一个“作者自己能跑通”的项目，而开始具备更明确的工程交付边界：

- 环境变量边界更清楚
- 三端启动方式更清楚
- 数据库命令入口更统一
- README 不再误导后续接手者

这为接下来的 `P7-4` 提供了很好的基础，因为部署方案和上线检查项，必须建立在本地环境说明足够清晰的前提之上。
