# P6-5：多模块统一发布策略设计

## 背景

到 `P6-4` 为止，内容系统已经具备：

- `P6-1`：统一状态流转规则
- `P6-2`：基础审计字段
- `P6-3`：历史快照能力
- `P6-4`：历史恢复能力

但是当前公开站点仍然采用“模块各自发布”的模式。

也就是说：

- 文案模块可以单独进入 `published`
- 简历模块可以单独进入 `published`
- 项目模块也可以单独进入 `published`
- 前台读取时再分别按 `status === published` 过滤

这种模式在早期开发阶段足够灵活，但随着内容工作流逐渐成熟，会暴露一个新的问题：

- 某次更新里，项目已经发布了
- 但配套文案还没发布
- 或者简历已经恢复到旧版本
- 但首页和项目列表展示的是另一批发布时间点的数据

这样前台就会出现“跨模块版本错位”。

因此，`P6-5` 的目标，是把公开站点从“模块各自发布”推进到“统一发布快照”模式，让一次公开发布能够代表一组彼此一致的公开内容状态。

## 本轮目标

本轮聚焦“统一公开发布策略”的第一版落地，完成以下能力：

- 建立统一发布批次模型
- 支持后台创建一次新的公开发布版本
- 发布时记录当前已发布的文案、简历、项目版本引用
- Web 端优先读取“当前激活的公开发布版本”
- 后台可以查看并激活不同公开发布批次

本轮暂时不做：

- 复杂发布审批流
- 定时发布
- 灰度发布
- 按语言分批发布
- 部分模块选择性发布
- 发布对比视图

也就是说，`P6-5` 先解决“同一时刻前台读取哪一组公开内容”，暂不解决更复杂的内容运营能力。

## 核心问题

### 当前问题不是“有没有 published”，而是“published 属于哪一批”

目前每个模块都有自己的 `published` 状态，但这些 `published` 并没有一个统一的时间边界。

例如：

- 上午 10 点发布了简历
- 上午 11 点发布了首页文案
- 下午 2 点发布了项目列表

那么前台当前看到的其实是：

- 10 点的简历
- 11 点的文案
- 2 点的项目

这在技术上是可用的，但在内容治理上并不稳定。

所以 `P6-5` 的关键，不是再给单个模块加更多状态，而是要引入“公开发布批次”这一层。

## 设计原则

### 1. 模块发布状态保留，统一发布批次额外引入

本轮不会移除模块自己的 `published` 状态。

因为模块内工作流仍然需要它来表达：

- 这条内容是否已经通过模块审核
- 这条内容是否有资格进入公开发布池

因此新的统一发布模型是：

- 模块自己的 `published`：表示“有资格被公开发布”
- 统一发布批次：表示“前台当前正在消费哪一组发布内容”

### 2. 统一发布批次只引用版本，不复制主内容

为了避免重复存储整份内容，本轮统一发布批次不再复制一份完整内容 JSON，而是记录三类模块当前采用的版本引用。

例如：

- `resumeVersionId`
- `translationVersionIds[]`
- `projectVersionIds[]`

这样做的好处是：

- 数据结构更轻
- 与现有版本系统直接衔接
- 后续切换激活批次时不需要重新生成一套内容副本

### 3. Web 只读取“激活中的统一发布批次”

本轮引入统一发布批次后，公开站点不再直接读取“当前主表里谁是 published”，而是优先读取：

- 当前激活的发布批次
- 再根据该批次引用的版本 ID 去解析公开内容

这一步很关键，因为它把前台的数据来源从“当前数据库最新 published 记录”切换成了“当前公开发布快照”。

### 4. 发布批次是一种显式动作，不是自动跟随模块变化

本轮不会在某个模块一进入 `published` 后，就自动让前台立刻采用它。

而是要求：

- 模块先独立完成内容编辑与审核
- 管理后台显式执行一次“创建并激活发布批次”
- 前台才切换到新的统一公开内容

这样能让前台始终保持批次一致性。

## 数据结构设计

本轮新增统一发布实体，建议命名为：

- `content_release_snapshots`

核心字段如下：

- `id`
- `name`
- `status`：`draft` | `active` | `archived`
- `resumeVersionId`
- `translationVersionIds`
- `projectVersionIds`
- `createdBy`
- `activatedBy`
- `activatedAt`
- `createdAt`
- `updatedAt`

其中：

### `draft`

表示一个已经生成、但尚未作为线上公开版本启用的发布批次。

### `active`

表示当前公开站点正在消费的发布批次。

同一时刻只允许存在一个 `active` 批次。

### `archived`

表示历史发布批次，保留查询能力，但不再生效。

## 首版策略

为了尽快形成闭环，本轮采用“创建即激活”的简化方案：

- 后台点击“创建公开发布”
- 系统自动抓取当前所有 `published` 状态的可公开版本
- 生成一个新的发布批次
- 立即将它设为 `active`
- 旧的 `active` 批次自动转为 `archived`

这意味着本轮不需要单独做一套复杂的发布草稿编辑器。

## 版本引用规则

### 简历

- 取当前简历模块最近一条 `status = published` 的版本
- 若不存在，则本次统一发布失败

### 文案

- 取所有 `status = published` 的文案记录对应的最新版本
- 这些版本构成当前批次的文案快照集合

### 项目

- 取所有 `status = published` 的项目对应的最新版本
- 这些版本构成当前批次的项目快照集合

## 接口设计

本轮新增一组发布批次接口，建议挂在：

- `GET /api/admin/releases`
- `POST /api/admin/releases/publish`
- `POST /api/admin/releases/:releaseId/activate`
- `GET /api/public/release`

其中：

### `GET /api/admin/releases`

用于后台查看历史发布批次。

### `POST /api/admin/releases/publish`

用于创建并激活一个新的统一公开发布批次。

### `POST /api/admin/releases/:releaseId/activate`

用于将某个历史发布批次重新设为当前生效版本。

这相当于“站点级恢复”，但恢复的是公开内容批次，而不是单条模块内容。

### `GET /api/public/release`

用于前台获取当前激活的公开发布批次信息与引用内容。

## Web 接入策略

本轮 Web 接入遵循“尽量少动页面”的原则。

因此：

- `web` 不需要一次性重写所有页面
- 先在 API Server 公共查询层中，把原本分散的公开数据，统一切换为从激活发布批次解析
- Web 继续调用原有 `public` 查询接口

也就是说，本轮主要改 API Server 的公共内容来源，不强迫 Web 页面大改。

## 后台交互策略

后台新增一个“统一发布中心”页面，第一版先提供三类能力：

- 查看当前激活批次
- 查看历史发布批次
- 创建并激活新批次

页面重点展示：

- 批次名称
- 状态
- 创建人
- 激活人
- 创建时间
- 激活时间
- 所含简历 / 文案 / 项目数量

## 边界与约束

### 1. 若没有已发布简历，则不能创建批次

因为公开站点的简历页是主链路内容，本轮要求必须有可发布简历版本。

### 2. 文案和项目允许为空集合，但要显式记录数量

这样首版方案仍然足够灵活。

### 3. 只允许一个激活批次

创建或激活新批次时：

- 当前 `active` 批次自动变为 `archived`

### 4. 模块后续继续编辑，不影响当前公开站点

这是本轮最核心的收益之一。

也就是说：

- 后台继续修改某条已发布内容
- 即使主表状态仍然是 `published`
- 只要没有重新创建统一发布批次
- 前台仍然读取旧批次引用的版本

这样前台内容就真正稳定下来。

## 完成标准

`P6-5` 完成后，应满足：

- 系统具备统一发布批次模型
- 后台可以创建并激活新的公开发布批次
- 历史批次可以重新激活
- Web 公开内容开始优先读取激活中的批次
- 前后台联调、类型检查与公开页面访问通过
