## 本次实现

这次 `P7-2` 并不是“零代码文档整理”，而是在联调过程中发现并修复了两类真实前端问题。

### 1. 为 Admin 补齐统一时间格式化，消除 hydration mismatch

在第一次使用 Playwright 验证后台页面时，`/translations`、`/projects`、`/releases` 等页面都出现了明显的 hydration mismatch。

问题根因很明确：

- 服务端和客户端都在渲染时间字段
- 但页面直接使用了 `new Date(value).toLocaleString()`
- 这会受到运行环境默认 locale / timezone 影响
- 结果就是 SSR 输出和 CSR 输出不一致

这类问题在页面上不一定立刻“挂掉”，但它意味着：

- 控制台会持续报 hydration mismatch
- 页面后续联调噪音很大
- 真实 SSR 稳定性存在隐患

这次做法是：

- 新增 `apps/admin/app/composables/useDateTimeFormatter.ts`
- 统一使用固定 `zh-CN + Asia/Shanghai + h23` 的时间格式
- 替换后台多个页面与工作流 composable 里的直接 `toLocaleString()`

调整后再次验证，后台关键页面的 hydration mismatch 已消失。

### 2. 修复 Web 在渲染期写 cookie / localStorage 导致的服务不稳定问题

在继续验证 Web 时，又暴露了第二个更关键的问题。

重启 `apps/web` 后，日志里出现了：

- `Cannot append headers after they are sent to the client`

同时页面还出现了几个异常表现：

- 有时根路径和简历页直接访问会失败
- 某些页面在 Playwright 里会读取到旧会话里的前端状态
- 服务在无 TTY 后台启动时不稳定，容易让问题更加难判断

继续排查后，根因落在两个 composable：

- `apps/web/app/composables/useWebLocale.ts`
- `apps/web/app/composables/useWebTheme.ts`

之前这里在 composable 初始化阶段就直接做了这些副作用：

- 写 cookie
- 读取并覆盖 localStorage
- 改写 `colorMode.preference`

对于 Nuxt SSR 来说，这类“渲染期间直接写状态”的模式很危险，特别是配合 `useAsyncData` 的缓存与服务端渲染时，更容易触发 header 已发送后再写 cookie 的问题。

这次修复方式是：

- 保留 cookie 作为初始状态来源
- 移除 composable 初始化阶段的直接写入副作用
- 把本地存储恢复与 `document` / `colorMode` 同步移动到 `onMounted()` 中执行
- 只在显式 `setLocale()` / `setThemePreference()` 时写 cookie 与 localStorage

这样处理后：

- Web 服务恢复稳定
- `/`、`/resume`、`/projects`、`/projects/[slug]` 可以稳定返回 `200`
- 重新打开全新浏览器会话时，页面内容与 SSR 输出保持一致

## 本次排查经验

这次 `P7-2` 很有价值的地方，在于它再次提醒我们：

### 1. 前端“能显示”不代表 SSR 稳定

后台页面最初从视觉上看几乎是正常的，但 Playwright 一跑，就把 hydration mismatch 暴露出来了。

如果不在 `P7-2` 收口阶段处理，后续越往后拖，这类噪音越难分辨。

### 2. Nuxt composable 里要谨慎做渲染期副作用

`useCookie()`、`localStorage`、`colorMode.preference` 这些动作，如果在 composable 初始化阶段直接混用，很容易把 SSR 和客户端行为搅乱。

在 Nuxt 场景下，更稳妥的原则是：

- 渲染阶段只读取初始状态
- 副作用移动到 `onMounted()` 或显式 action
- cookie / localStorage 写入只在用户动作或明确同步点发生

### 3. 浏览器验证要尽量使用“全新会话”复查

这次 Web 页面最开始看起来像是“服务端和客户端逻辑不一致”，但其中一部分其实和旧浏览器会话缓存叠加在一起，让现象更混乱。

因此在联调收口阶段，推荐固定增加一个动作：

- 用全新浏览器会话再直开关键页面一次

这样更容易区分：

- 是代码真的有问题
- 还是旧会话缓存干扰判断
