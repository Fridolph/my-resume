# P7：测试、部署与环境收口总设计

## 背景

到 `P6-5` 为止，第二阶段的业务主链路已经基本闭环：

- `apps/web` 已接入真实公开内容
- `apps/admin` 已具备真实管理能力
- `apps/api-server` 已具备认证、权限、内容工作流、版本与统一发布能力
- 数据层已经统一到 `SQLite + Drizzle ORM`

这意味着当前项目已经从“架构搭建期”进入“可验证、可交付、可收口”的阶段。

因此，`P7` 的目标不再是新增业务能力，而是为第二阶段收尾，完成：

- 质量保障
- 环境收口
- 启动与验证流程收口
- 部署方案整理
- 阶段总结沉淀

## P7 总目标

让当前三端项目从“本地可开发”升级为“可验证、可维护、可部署”的稳定工程阶段成果。

## P7 拆分原则

这一阶段不适合再按单一大任务推进，而应该拆成一组边界清晰、能逐项验收的收口任务。

因此，`P7` 拆为以下子任务：

- `P7-1`：API Server 关键链路测试补齐
- `P7-2`：Web / Admin 关键流程验证收口
- `P7-3`：环境变量与多端启动文档收口
- `P7-4`：部署方案与上线前检查项整理
- `P7-5`：第二阶段总结文档与交付清单整理

## P7-1：API Server 关键链路测试补齐

### 目标

为后端最关键的工作流能力建立自动化验证基础，优先覆盖“高价值、易回归”的核心链路。

### 本轮重点

优先测试以下能力：

- 认证与权限基础链路
- 内容状态机合法 / 非法流转
- 版本快照生成
- 历史版本恢复
- 统一发布批次创建与激活
- 公开接口读取激活批次内容

### 建议范围

先以服务层 / 数据层集成为主，不急着做大而全的 e2e。

### 完成标准

- 至少覆盖内容工作流最关键路径
- 新增测试能在本地稳定执行
- 能拦住明显回归问题

## P7-2：Web / Admin 关键流程验证收口

### 目标

把三端联调流程从“人工零散验证”收口成一套明确的关键流程清单。

### 本轮重点

建议先聚焦以下流程：

- Admin 登录
- 文案编辑与发布
- 简历状态流转与恢复
- 项目恢复与排序稳定性
- 统一发布中心创建 / 激活批次
- Web 首页、简历页、项目列表页对公开内容的读取验证

### 完成标准

- 形成一份明确的联调清单
- 可以重复执行
- 能作为后续部署前冒烟检查基础

## P7-3：环境变量与多端启动文档收口

### 目标

统一当前三端的运行方式与环境变量约定，降低后续维护成本。

### 本轮重点

整理并明确：

- `api-server` 端口与基础环境变量
- `admin` 端口与 API 地址变量
- `web` 端口与公开 API 地址变量
- SQLite 数据文件位置
- Drizzle migration / studio / status 常用命令
- 三端本地启动顺序与推荐命令

### 完成标准

- 新人可按文档独立跑起三端
- 环境变量来源清晰
- 常用命令统一可查

## P7-4：部署方案与上线前检查项整理

### 目标

为后续部署或上线准备一套基础方案，而不是等到上线时再临时拼装。

### 本轮重点

建议整理：

- 三端部署形态建议
- SQLite 部署注意事项
- 数据文件持久化策略
- Admin / Web / API 的域名与反向代理关系
- 上线前必须检查项
- 数据备份与回滚建议

### 完成标准

- 有一份明确的部署建议文档
- 上线前检查项可执行
- 后续迁移到正式环境时有基础蓝图

## P7-5：第二阶段总结文档与交付清单整理

### 目标

把第二阶段从“开发完成”升级为“可交付总结”。

### 本轮重点

形成以下沉淀：

- 第二阶段完成了什么
- 当前三端架构是什么
- 数据工作流到什么程度
- 当前仍有哪些未覆盖项
- 第三阶段可从哪里继续推进
- 对外或团队内部可复用的交付清单

### 完成标准

- 第二阶段有完整总结
- 当前仓库状态与能力边界清晰
- 后续阶段规划可以无缝衔接

## 推荐执行顺序

建议严格按以下顺序推进：

1. 先做 `P7-1`，补自动化测试底座
2. 再做 `P7-2`，收口三端关键联调流程
3. 然后做 `P7-3`，整理环境变量与启动方式
4. 接着做 `P7-4`，形成部署方案与上线清单
5. 最后做 `P7-5`，补阶段总结与交付清单

## 每个子任务的标准流程

后续 `P7-*` 的推进继续沿用当前协作标准：

1. 先出详细设计文档
2. 再编码实现
3. 完成本地自测 / 联调
4. 写技术记录博客到 `docs/phase-2/`
5. 更新 `docs/phase-2/里程碑规划.md`
6. 按“代码提交 / 文档提交”分别提交并推送

## 当前建议下一步

`P7` 拆分完成后，建议直接从：

- `P7-1`：API Server 关键链路测试补齐

开始推进。
