# P5-4：接口级权限守卫设计

## 背景

到 `P5-3` 为止，后台已经完成了真实登录态接入，但当前 `api-server` 仍然主要依赖“是否已登录”来决定是否放行。

这还不够，因为一个正式的管理系统不仅要知道“你是谁”，还要知道“你能做什么”。

因此，`P5-4` 的目标是把后端接口从“有登录态”升级为“按权限放行”。

## 设计目标

这一阶段主要完成三件事：

- 建立统一的 NestJS 权限守卫基础设施
- 为关键后台接口声明所需权限
- 在不破坏现有 `web` 公开内容链路的前提下，先保护敏感接口与写接口

## 基础策略

本次采取的是“增量收口”策略，而不是一次性把所有接口都强制鉴权。

原因是当前公开站点 `web` 仍然在消费以下公开接口：

- `GET /api/projects`
- `GET /api/resume`
- `GET /api/site-settings`
- `GET /api/translations`

如果在 `P5-4` 就直接把这些读取接口全部改为必须登录，会立刻破坏公开站点。

因此，本轮策略是：

- 公开读取接口继续保持公开
- 敏感后台接口先接权限守卫
- 写接口优先进入权限控制
- 用户管理接口作为第一批严格受保护接口

## 设计组件

### `@Public()`

用于显式声明公开接口。

作用：

- 与权限守卫配合使用
- 明确标记哪些接口不需要登录态
- 为后续 `P5-6` 的接口边界收口提供基础

### `@RequirePermissions(...)`

用于声明接口所需权限。

例如：

- `@RequirePermissions('user.read')`
- `@RequirePermissions('project.write')`

### `ApiAuthGuard`

统一完成三件事：

- 从 cookie 读取当前 session token
- 调用 `AuthService` 解析当前用户
- 根据 metadata 判断是否需要权限校验

### `@CurrentUser()`

用于在控制器中拿到当前登录用户。

这一轮主要先把它作为后续扩展点保留下来，便于 `P5-5`、`P5-6` 继续复用。

## 首批纳入权限控制的接口

### 用户接口

全部纳入保护：

- `GET /api/users` -> `user.read`
- `POST /api/users` -> `user.write`
- `PUT /api/users/:userId` -> `user.write`

### 文案写接口

- `PUT /api/translations/:translationId` -> `translation.write`

### 简历写接口

- `PUT /api/resume` -> `resume.write`

### 项目写接口

- `POST /api/projects` -> `project.write`
- `PUT /api/projects/:projectId` -> `project.write`
- `DELETE /api/projects/:projectId` -> `project.write`

### 站点设置写接口

- `PUT /api/site-settings` -> `site.write`

### Auth 接口

- `POST /api/auth/login` -> 公开
- `POST /api/auth/logout` -> 仅登录用户可调用
- `GET /api/auth/me` -> 仅登录用户可调用

## 为什么这轮先不保护公开 GET 接口

因为公开站点仍依赖这些读取接口工作。

如果现在就把：

- `GET /api/projects`
- `GET /api/resume`
- `GET /api/site-settings`
- `GET /api/translations`

改成必须登录，那么 `web` 的公开展示会立即中断。

因此，这一轮先做“写接口 + 敏感接口”的保护，是更合理的阶段性策略。真正的公开/后台接口边界，会在 `P5-6` 再彻底收口。

## 下一步

`P5-4` 完成之后，下一步就是：

- `P5-5`：Admin 页面接入真实权限判断

届时前端导航、按钮显隐、页面访问控制，就会和这一轮的后端权限守卫真正形成前后闭环。
