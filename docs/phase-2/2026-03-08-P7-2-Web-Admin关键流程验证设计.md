# P7-2：Web / Admin 关键流程验证设计

## 背景

到 `P7-1` 为止，第二阶段的后端关键链路已经具备自动化测试基础，`apps/api-server` 可以通过集成测试验证认证、工作流、版本恢复和统一发布批次。

但第二阶段的交付对象并不只有 API Server。

当前项目已经形成三端联动架构：

- `apps/web`：Nuxt 4 展示端
- `apps/admin`：Nuxt 4 管理后台前端
- `apps/api-server`：NestJS API Server
- `packages/database`：`SQLite + Drizzle ORM`

这意味着第二阶段收口时，除了后端自动化测试，还必须把前端关键流程整理成一套可重复执行的联调验证清单。否则项目虽然“能打开页面”，但仍然存在以下风险：

- 管理后台登录态、权限跳转和模块入口可能在后续改动后悄悄失效
- Web 公开页面虽然可渲染，但未必真的与当前激活发布批次保持一致
- 第二阶段结束后，新加入项目的开发者缺少一份可直接照着执行的联调手册

因此，`P7-2` 的目标不是继续堆业务模块，而是把当前已经落地的 Web / Admin 主流程正式收口为一套“可以重复执行、可以复查、可以交接”的验证方案。

## 本轮目标

本轮聚焦两类工作：

- 梳理 Web / Admin 当前必须验证的关键流程
- 基于本地真实服务完成一轮可记录的冒烟联调

本轮暂时不做：

- 浏览器级自动化测试文件沉淀
- 组件级单元测试补齐
- 全模块像素级 UI 回归
- 性能测试与压测
- 部署平台联调

也就是说，`P7-2` 解决的是“前端主流程可重复验证”，而不是一次性把所有前端测试体系都补完。

## 验证范围

## Admin 侧

本轮把 Admin 的关键流程收口为以下七类：

### 1. 登录页与真实会话建立

验证：

- 未登录用户可以访问 `/login`
- 使用演示账号可以成功登录
- 登录成功后能够跳转到后台首页
- 登录成功后页面可以显示当前账号名称、邮箱和角色信息

### 2. 鉴权拦截与登录跳转

验证：

- 未登录访问 `/`、`/users`、`/translations`、`/resume`、`/projects`、`/settings`、`/releases` 时，会被跳转到 `/login`
- 登录态建立后再次访问后台页面，不再发生错误跳转

### 3. 权限边界与页面可见性

验证：

- 当前账号只能看到有权限访问的模块卡片与入口
- 当账号缺少某模块读取权限时，会跳转到 `/unauthorized`
- 页面内写操作按钮会根据写权限决定是否展示

### 4. 文案管理主流程

验证：

- `/translations` 页面可以正常加载
- 文案列表能返回 namespace、locale、状态等基础信息
- 当前公开站点依赖的文案已经进入统一发布闭环

### 5. 简历管理主流程

验证：

- `/resume` 页面可以正常加载
- 简历状态与版本信息能正常显示
- 已发布简历可以继续参与统一发布批次

### 6. 项目管理主流程

验证：

- `/projects` 页面可以正常加载
- 项目列表、排序信息和状态信息可以展示
- 已发布项目可以进入统一发布批次

### 7. 统一发布中心

验证：

- `/releases` 页面可以正常加载
- 当前激活批次信息可以展示
- 历史批次列表可以展示
- 激活批次切换后，公开站点读取结果随之更新

## Web 侧

本轮把 Web 的关键流程收口为以下五类：

### 1. 首页可用性

验证：

- `/` 可以正常返回 `200`
- 首页能加载站点介绍、卡片和统计信息
- 页面不出现明显报错态

### 2. 简历页可用性

验证：

- `/resume` 可以正常返回 `200`
- 简历内容来自公开接口，不依赖本地 mock
- 页面在无报错情况下渲染主要区块

### 3. 项目列表页可用性

验证：

- `/projects` 可以正常返回 `200`
- 项目列表能展示真实项目数据
- 页面在无报错情况下渲染列表区块

### 4. 项目详情页可用性

验证：

- `/projects/[slug]` 可以正常返回 `200`
- 已发布项目详情可以正常打开
- 页面不出现公开内容读取失败提示

### 5. 公开内容与激活批次一致性

验证：

- `apps/web` 的公开内容读取统一指向 `api-server` 的 `/api/public/*`
- 当前首页、简历页、项目页读取的数据与 `/api/public/release` 返回的激活批次保持一致
- 如果后台切换激活批次，Web 的公开数据应随之切换

## 设计原则

### 1. 先验证“主链路可用”，再考虑自动化规模化

当前阶段的重点不是补齐一整套前端测试框架，而是先明确：

- 三端服务是否能同时启动
- 后台登录与权限边界是否可用
- 前台公开页面是否真的消费统一发布结果

所以本轮优先采用：

- 本地真实环境冒烟联调
- HTTP 请求校验
- 浏览器页面人工 / 半自动验证
- 结构化记录验证步骤与结果

### 2. 验证清单必须能够被新人重复执行

本轮产出的验证方案，不应该只服务当前开发过程，还应该服务后续交接。

因此验证文档必须明确：

- 服务启动方式
- 访问地址
- 演示账号
- 验证步骤
- 预期结果
- 失败时优先排查方向

### 3. 优先验证“发布闭环”，而不是零散页面能否打开

第二阶段最有价值的结果，并不是后台页多了几个路由，而是：

- 后台可管理内容
- 后端可维护版本与发布状态
- 前台可消费当前生效批次

所以 `P7-2` 的关键，不只是“页面返回 200”，而是“Web / Admin / API 三端闭环是否成立”。

## 执行方案

## 本地联调端口

本轮联调约定使用：

- `api-server`：`http://127.0.0.1:3011`
- `web`：`http://localhost:3000`
- `admin`：`http://localhost:3002`

其中：

- `apps/web` 默认读取 `http://localhost:3011/api/public`
- `apps/admin` 默认读取 `http://localhost:3011/api/admin`

## 演示账号

优先使用：

- `root@fridolph.local / root123`

它对应 `super-admin`，可以覆盖本轮绝大多数后台流程验证。

## 验证方式

本轮采用“分层验证”：

### 第一层：服务健康检查

验证：

- 三端端口是否都在监听
- 公开接口 `/api/public/release` 是否返回 `200`
- 管理接口未登录访问是否返回 `401` 或前端跳转

### 第二层：页面级冒烟验证

验证：

- `web` 首页、简历页、项目页、项目详情页可打开
- `admin` 登录页、首页、文案页、简历页、项目页、发布页可打开

### 第三层：发布闭环验证

验证：

- Admin 端统一发布中心存在激活批次
- Web 端公开内容可正常读取
- 公开接口返回的版本信息与后台页面展示保持一致

- 发布闭环验证结果有书面记录
- 下一位开发者可以按文档重复执行验证
