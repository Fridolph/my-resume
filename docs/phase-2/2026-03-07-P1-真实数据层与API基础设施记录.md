# P1：真实数据层与 API 基础设施记录

## 背景

第二阶段的第一步不是继续扩页面，而是把第一阶段已经稳定下来的结构，开始接到真实数据层上。

P1 的任务很明确：先选定数据库方案、选定数据访问层，并用一个足够清晰的样板模块，把“数据库 → 共享仓储 → API 路由 → 后台页面”这一条链路真正跑通。

## 本次技术选择

当前 P1 采用以下方案：

- 数据库：SQLite
- 数据访问层：Node.js 内建 `node:sqlite` + Repository
- 样板模块：站点配置

之所以这样选，是因为它足够轻量、启动成本低、适合当前 Monorepo 的第二阶段起步。同时，站点配置模块结构简单、跨前后台价值高，非常适合作为第一个真实数据闭环样板。

## 实现过程中的一次调整

P1 一开始尝试过 `SQLite + Drizzle ORM + better-sqlite3`。这个组合在抽象层面没有问题，但当前本地环境里 `better-sqlite3` 的原生绑定没有稳定跑起来，进一步引出了运行时初始化异常。

为了让第二阶段先把“真实数据链路”跑通，而不是把时间消耗在原生依赖兼容上，P1 最终回退到 Node.js 内建的 `node:sqlite`，并在 `packages/database` 中补上一层 Repository。这个调整让数据库依赖更轻，也让 Monorepo 在当前阶段更容易稳定开发。

## 本次实现

### 1. 建立共享数据库包

P1 新增了 `packages/database`，用于集中管理：

- 数据库连接与初始化
- 行结构类型定义
- 站点配置仓储函数
- 默认种子与首条数据落盘逻辑

这一步的意义在于，数据库能力不再散落在应用里，而是开始成为 Monorepo 的共享基础设施。

### 2. 建立本地数据库文件约定

P1 将本地数据库文件约定为：

- `data/platform.sqlite`

同时在 `.gitignore` 中忽略该文件，避免把本地运行数据直接提交到仓库。

### 3. 建立首个真实 API 路由

P1 在 `apps/admin/server/routes/api` 中新增了站点配置 API：

- `GET /api/site-settings`
- `PUT /api/site-settings`

并在 `apps/web/server/routes/api` 中新增了只读样板：

- `GET /api/site-settings`

这说明第二阶段已经开始进入真正的服务端 API 组织阶段，而不再只是前端本地状态模拟。

### 4. 将站点配置页切换为真实读写样板

P1 把后台 `settings` 页面切换为通过真实 API 获取和保存数据。这意味着当前后台已经出现了第一个真正的“数据库 → API → 页面”闭环样板。

虽然目前只打通了站点配置模块，但这个样板已经足够说明第二阶段的基础设施方向是可行的。

## 本地自测结果

### 1. 类型检查

执行通过：

- `pnpm --dir apps/admin typecheck`
- `pnpm --dir apps/web typecheck`

### 2. 本地服务与 API 验证

已验证：

- 后台站点设置页可以通过真实 API 读取配置
- 后台站点设置页可以通过真实 API 保存配置
- Web 端可以读取同一份站点配置数据

### 3. 运行时说明

当前开发环境会看到 `node:sqlite` 的 experimental warning。这是 Node.js 对该内建模块的现阶段提示，不影响 P1 已完成的本地开发与联调验证。
