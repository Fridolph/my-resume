# P1：真实数据层与 API 基础设施记录

> 可以参考下，之前AI这里用了`node:sqlite` + Repository，不是我预期的技术栈
> 发现了赶紧纠正回来，所以说大家使用AI时每个阶段还是得仔细看，别一把梭

## 背景

第二阶段最初的目标，是先让项目脱离纯 mock 数据，开始进入真实数据链路。这个判断本身没有问题，因为如果没有真实数据读写，前台和后台就仍然停留在结构性重构阶段，无法验证真正的业务可行性。

因此，P1 当时采用了一个过渡性思路：先用当前仓库最容易落地的方式，把“数据库 → API → 页面”这条链路跑通，再根据实践结果决定下一步是否要继续放大。

现在回头看，这次实践是必要的，但它更适合作为一次过渡验证，而不是第二阶段的最终技术路线。

## 当时的过渡方案

P1 当时采用的实际实现如下。

- 数据库：SQLite
- 数据访问层：Node.js 内建 `node:sqlite` + Repository
- API 入口：Nuxt 内部 server routes
- 样板模块：站点设置

这套方案最大的价值，不在于它本身会成为最终答案，而在于它帮助我们快速确认了几件关键事实。

- Monorepo 内部可以稳定共享一份本地 SQLite 数据文件。
- 站点设置适合作为第一条真实业务样板链路。
- 后台写入和前台读取可以在同一份数据基础上打通。
- 第二阶段确实已经到了需要独立后端的阶段。

## 这次过渡方案解决了什么

P1 完成后，项目首次具备了真实数据能力的最小闭环。

### 1. 验证了本地数据库落盘能力

P1 建立了共享数据库包和本地数据库文件约定，让项目第一次真正拥有了可持久化的数据层，而不再只是本地状态模拟。

### 2. 验证了首个真实业务样板

站点设置模块成功跑通了读写闭环。这意味着第二阶段后续所有内容模块，都可以参考这条链路继续抽象和扩展。

### 3. 暴露了系统真正的下一步问题

P1 做完以后，问题已经不再是“能不能接数据库”，而是“谁来作为整个系统的正式后端入口”。

这正是第二阶段从局部实现进入系统架构调整的分界点。

## 过渡方案的问题在哪里

P1 的问题不在于它不能工作，而在于它不适合作为后续长期架构继续扩展。

### 1. 后端职责分散在前端应用内部

当 API 放在 `apps/web` 和 `apps/admin` 内部时，系统虽然能跑，但业务入口是分散的。随着用户、权限、文案、简历、项目和发布流逐渐增加，业务逻辑会开始散落在两个 Nuxt 应用中。

这会让后续维护成本迅速升高，也会让权限、鉴权、日志、校验和错误响应难以统一。

### 2. 数据层方案与长期目标不一致

第二阶段的长期目标已经明确为 `SQLite + Drizzle ORM`。但 P1 因为环境兼容问题，临时切换到了 `node:sqlite + Repository`。这让 P1 能够快速完成验证，却也意味着如果继续在这条线上扩展更多模块，后续回到 Drizzle 的改造成本会越来越高。

### 3. Web 与 Admin 缺少统一 API Server

前台和后台未来都需要围绕同一个后端服务工作。P1 的做法更像是局部 BFF 验证，而不是正式的系统级 API 组织方案。

对于这个项目来说，第二阶段真正需要的已经不是“局部接口”，而是“统一后端”。

## 为什么要转向 NestJS

P1 过后，我们需要的不只是一个能返回 JSON 的接口层，而是一个可持续承载业务增长的后端。

选择 NestJS，核心原因有四点。

### 1. 它适合三端协同项目

当前项目已经不是单前端应用，而是公开展示端、管理后台前端和后端服务三端协作结构。NestJS 更适合作为中间的统一业务层。

### 2. 它适合模块化扩展

后续会接入的能力包括：

- 用户管理
- 权限控制
- i18n 文案管理
- 简历管理
- 项目管理
- 发布流和版本能力

NestJS 对这些领域模块的组织会明显比继续扩展 Nuxt server routes 更稳定。

### 3. 它更利于统一治理

DTO、参数校验、异常处理、守卫、拦截器和依赖注入，都是第二阶段后续会持续需要的能力。NestJS 在这些方面更适合作为长期基础设施。

### 4. 它能把前后台真正串起来

当 Web 和 Admin 都只通过 `apps/api-server` 访问数据时，系统的边界会变得清晰。后续无论是权限、缓存、日志还是发布策略，都可以在同一层统一实现。

## 为什么仍然坚持 SQLite + Drizzle ORM

虽然 P1 曾经因为环境问题退到了 `node:sqlite`，但这并不改变第二阶段的长期数据库方向。

我们仍然坚持 `SQLite + Drizzle ORM`，原因是：

- SQLite 轻量，适合当前阶段快速迭代
- Drizzle ORM 类型清晰，适合 TypeScript Monorepo
- Drizzle Kit 能帮助数据库迁移和结构演进保持有序

也就是说，P1 的临时实现是为了验证链路，而第二阶段后续的正式落地必须回到统一的数据层标准。

## 对 P1 的重新定位

现在可以明确给 P1 一个更准确的定义。

> P1 不是第二阶段的最终后端基础设施，而是第二阶段开始前的一次必要过渡验证。

它完成了验证任务，也帮助我们更早看清了正式架构必须具备的条件。

因此，P1 不会被删除，它会作为阶段记录保留下来，但后续不会继续沿着这套实现直接扩大范围。

## 新阶段的正式方向

从当前开始，第二阶段正式切换到以下架构。

- `apps/web`：Nuxt 4 展示端
- `apps/admin`：Nuxt 4 管理后台前端
- `apps/api-server`：NestJS API Server
- `packages/database`：Drizzle ORM 数据层
- `packages/sdk`：统一 API Client
- 数据库：SQLite

接下来的第一步，不再是继续扩展旧 P1，而是启动新的正式版 P1：重建 API Server 与数据库基座。

## 这次记录的价值

这篇记录不只是为了说明“P1 做了什么”，更是为了保留一次真实的架构演进判断。

在重构过程中，阶段性的过渡实现很常见。关键不在于它是否永久存在，而在于团队能不能及时识别它的边界，并在正确的时间收敛到长期方案上。

P1 的价值，恰恰就在这里。
