# P7-4：部署方案与上线前检查项记录

## 背景

在 `P7-4` 开始之前，第二阶段已经完成了三件关键事情：

- `P7-1`：后端关键链路具备自动化测试基础
- `P7-2`：Web / Admin 关键流程完成一轮真实联调
- `P7-3`：环境变量、启动命令和 README 已经完成收口

这意味着当前项目已经不缺“开发说明”，而开始缺“上线说明”。

如果没有这一步，后续真正要部署时，团队通常会临时遇到一连串问题：

- 线上域名应该配在哪里
- Web 的 `robots.txt` 和 `sitemap.xml` 是否真的反映线上内容
- API 的 CORS 到底应该开放到什么程度
- SQLite 数据文件放在哪才算持久化
- 上线前应该检查哪些点，才不至于刚发版就踩基础坑

所以 `P7-4` 的目标，是在正式上线前，把这些“迟早要补的工程问题”提前收口掉。

## 本次实现

这一轮 `P7-4` 不只是出文档，也顺手补了几个会真实影响上线质量的配置点。

### 1. 为 Web 补齐可部署的站点公开变量

在本轮之前，`apps/web` 虽然已经有站点级 SEO 配置，但默认值还是偏静态，部署时如果不改代码就不够自然。

这次已经在：

- `apps/web/nuxt.config.ts`

补齐以下变量支持：

- `NUXT_PUBLIC_SITE_URL`
- `NUXT_PUBLIC_SITE_NAME`
- `NUXT_PUBLIC_SITE_DESCRIPTION`
- `NUXT_PUBLIC_DEFAULT_LOCALE`

这意味着当前 Web 的线上站点域名、标题和默认 SEO 描述都可以通过环境变量覆盖，而不需要再改代码。

这一步很重要，因为它直接关系到：

- canonical 是否正确
- OG 图默认地址是否正确
- `robots.txt` 与 `sitemap.xml` 中的域名是否正确

### 2. 为 API Server 补齐 CORS 环境变量收口能力

在本轮之前，`apps/api-server` 的 CORS 仍然是：

- `origin: true`

这对本地开发很方便，但线上环境更合理的做法应该是：

- 明确允许哪些来源访问后台和公开接口

这次已经在：

- `apps/api-server/src/main.ts`

补齐：

- `CORS_ORIGIN`

支持多个来源通过逗号分隔传入，例如：

- `https://web.example.com,https://admin.example.com`

这样当前项目就进入了一个更健康的部署状态：

- 本地仍可保持默认宽松
- 线上可以收口为明确白名单

### 3. 修复 Sitemap 仍读取旧 mock 的部署隐患

这次最有价值的一处收口，其实是 sitemap。

之前：

- Web 主页面已经切到真实公开 API
- 但 `apps/web/server/routes/sitemap.xml.ts` 仍然读旧 SDK mock 数据

这类问题平时开发时不一定立刻暴露，但它对正式部署其实很危险，因为会导致：

- 站点地图与线上真实公开内容不一致
- 搜索引擎抓取到的 URL 集合可能不是当前发布版本

这次已经把 sitemap 数据源切换为：

- 通过 `createPublicContentApiClient()` 读取真实 `/api/public/projects`
- 并仅输出 `published` 项目路由

这意味着现在：

- Web 页面主链路和 sitemap 的数据来源一致了
- SEO 辅助能力终于跟真实公开内容对齐了

这轮还新增了两份部署侧文档：

其中：

- 检查清单负责把上线前要核对的动作变成可执行条目

另外，这次还把部署相关变量一并补进：

这样部署信息不再只散落在单篇博客里，而是已经进入仓库的主说明入口。

## 推荐部署拓扑

这次 `P7-4` 形成的推荐方案，是当前阶段最稳妥、最贴近现有架构的三服务分离部署：

- `web`：独立 Nuxt 服务
- `admin`：独立 Nuxt 服务
- `api-server`：独立 NestJS 服务
- `SQLite`：挂载在 `api-server` 所在机器的持久化目录

推荐域名关系：

- `example.com` 或 `web.example.com` -> `apps/web`
- `admin.example.com` -> `apps/admin`
- `api.example.com` -> `apps/api-server`

这样有几个明显好处：

- Web 与 Admin 的缓存策略可以分开
- API 的 CORS 可以按域名显式收口
- SQLite 只需要跟随 API Server 部署与备份
- 三端后续迁移到更正式的基础设施也更容易分步演进
