# P3：统一响应结构与错误处理收口记录

## 背景

在 `P3` 的四个核心内容模块全部完成之后，后台已经具备了完整的真实业务链路：

- 用户管理
- i18n 文案管理
- 简历管理
- 项目管理

但在这一阶段结束前，还有一项非常关键的收口工作必须完成：统一响应结构与错误处理。

如果这一步不做，当前仓库虽然“能用”，但还存在几个明显问题：

- 不同接口直接返回裸数据，响应结构不统一
- SDK 只能假设“接口返回的就是业务数据本身”
- 校验失败、业务异常和未知异常的错误输出格式不一致
- 后续 `P4` 当前台开始正式接入 API 时，前后端会继续在响应格式上产生额外适配成本

因此，本次工作不是新增某个业务模块，而是对整个 `api-server` 做一次基础协议层收口。

## 本次目标

本次收口的目标是：

- 为 `api-server` 建立统一成功响应结构
- 为 `api-server` 建立统一错误响应结构
- 为参数校验、业务异常和未知异常提供一致的错误输出
- 调整 `packages/sdk`，让前端客户端自动解包统一响应结构
- 回归验证现有核心模块接口，确保不影响 Admin / Web 的既有调用

这一步完成后，当前 Monorepo 的前后端交互协议就会稳定下来，能为 `P4` 和后续鉴权阶段提供更可靠的基础。

## 本次实现

### 1. 建立统一成功响应结构

本次新增了统一成功响应模型，当前 API 成功返回统一为：

- `success`
- `data`
- `meta`

其中：

- `success` 固定为 `true`
- `data` 承载业务数据
- `meta` 记录 `path` 和 `timestamp`

例如原来 `GET /api/users` 直接返回数组，现在会统一返回：

- `success: true`
- `data: UserRecord[]`
- `meta: { path, timestamp }`

这让后续所有接口都具备一致的协议外壳，而不是每个模块各自返回不同风格的结构。

### 2. 增加全局响应拦截器

本次在 `apps/api-server` 中新增了全局响应拦截器，用于自动包装成功响应。

这样做的价值在于：

- 控制器无需重复手写返回格式
- 每个模块只关注业务数据本身
- 协议层规则由全局统一处理

当前的控制器仍然保持轻量，只返回业务对象，统一结构由拦截器自动补齐。

### 3. 建立统一错误响应结构

本次新增了统一错误响应模型，当前错误返回统一为：

- `success`
- `error`
- `meta`

其中 `error` 统一包含：

- `code`
- `message`
- `statusCode`
- `details`

当前错误返回已经可以区分以下场景：

- 请求参数校验失败
- `HttpException` 类型业务异常
- 未知内部异常

这样做的意义是，前端以后不需要再针对不同接口解析不同错误格式，统一按 `error.message`、`error.code` 和 `error.details` 处理即可。

### 4. 增加全局异常过滤器

本次在 `apps/api-server` 中新增了全局异常过滤器，对异常做统一归一化输出。

当前它会处理：

- `ZodError`
- `HttpException`
- 未知异常

例如：

- `ZodError` 会被统一映射为 `VALIDATION_ERROR`
- `NotFoundException` 会被统一映射为 `NOT_FOUND`
- 未知异常会被统一映射为 `INTERNAL_SERVER_ERROR`

这让错误语义变得更清晰，也为后续前端做统一错误提示和错误埋点提供了条件。

### 5. 收口 ValidationPipe 的错误工厂

除了全局异常过滤器之外，本次还收口了 `ValidationPipe` 的 `exceptionFactory`。

这样做之后，NestJS 参数校验失败时，也会输出统一的错误结构，不会和自定义校验逻辑分裂成两套表现形式。

当前参数校验失败时，会统一返回：

- `message: 请求参数校验失败`
- `validationErrors` 作为明细

### 6. 调整 `packages/sdk` 自动解包响应

由于 API 成功返回已经从“裸数据”切换为统一结构，因此本次同步调整了 `packages/sdk`。

当前 SDK 已新增统一请求函数，用于：

- 自动解析 `ApiResponse<T>`
- 成功时返回 `payload.data`
- 失败时抛出 `PlatformApiError`

并且 `PlatformApiError` 现在会保留：

- `message`
- `code`
- `statusCode`
- `details`

这意味着 Admin / Web 侧继续调用 SDK 时，拿到的仍然是业务数据本身，而不是需要在每个页面手动写 `response.data.data` 的重复解包逻辑。

## 本地自测结果

本次已完成以下验证。

### 1. 类型检查通过

执行通过：

- `pnpm --dir apps/api-server typecheck`
- `pnpm --dir apps/admin typecheck`
- `pnpm --dir apps/web typecheck`

### 2. 成功响应结构验证通过

已验证：

- `GET /api/health`
- `GET /api/users`

当前成功响应均已返回统一结构：

- `success: true`
- `data`
- `meta`

### 3. 错误响应结构验证通过

已验证：

- 向 `POST /api/users` 发送非法参数
- 向不存在的翻译记录执行 `PUT /api/translations/not-found`

当前错误响应均已返回统一结构：

- `success: false`
- `error.code`
- `error.message`
- `error.statusCode`
- `error.details`
- `meta`

并且已经验证到：

- 参数校验错误会返回 `VALIDATION_ERROR`
- 业务不存在错误会返回 `NOT_FOUND`

### 4. SDK 解包与抛错验证通过

已通过脚本验证：

- `createUsersApiClient().listUsers()` 可以正确拿到业务数据数组
- `createUsersApiClient().createUser(...)` 在参数错误时会抛出 `PlatformApiError`
- `PlatformApiError` 中能够拿到 `message`、`code` 和 `statusCode`

这说明前端调用层已经成功适配统一协议，而不是被新的响应结构破坏。

## 当前结果

这次收口完成之后，`api-server` 已经从“多个功能模块并列可用”升级为“具备统一接口协议的正式服务层”。

当前仓库已经具备以下能力：

- 成功响应结构统一
- 错误响应结构统一
- 参数校验输出统一
- 业务异常输出统一
- SDK 自动解包成功响应
- SDK 自动抛出结构化错误对象

这意味着后续无论是 Web 公开内容接入，还是后台继续补权限和鉴权能力，都不需要再回头重构接口协议层。

## 这次收口的价值

这次工作的价值，主要体现在三个层面。

第一，协议稳定了。

后续任何新模块，只要接入 `api-server`，都能天然获得一致的成功/失败响应格式，而不需要再单独定义每个模块自己的规则。

第二，前端调用更稳定了。

Admin / Web 以后只需要通过 SDK 获取业务数据，并在异常时读取统一错误对象，不再需要在页面里重复兼容多种错误格式。

第三，`P3` 现在已经真正具备阶段闭环意义。

因为到这一刻为止，`P3` 不只是做完了四个内容模块，还把协议层和异常层也收干净了。这样进入 `P4` 时，前台页面接 API 就会顺畅很多。

## 下一步

随着统一响应结构与错误处理完成，`P3` 已基本进入完成态。接下来建议按计划推进下一阶段：

- 继续让 Web 公开内容全面接入后台数据
- 优先推进首页、简历页、项目列表页和项目详情页的正式 API 替换

如果要继续按价值排序推进，下一步最优先建议是：启动 `P4`，先让 Web 的简历页和项目页接入当前已经完成的正式 API。
