# P6-1：内容状态流转规则设计

## 背景

到 `P5` 阶段结束时，项目已经具备：

- 后台真实登录态
- 接口级权限控制
- 页面级权限收口
- 公开接口与后台接口边界分区

但内容本身的“生命周期”仍然比较松散。

虽然 `translation`、`resume`、`project` 这些模块已经有 `status` 字段，但当前状态更多只是一个展示字段，还没有形成真正统一的工作流规则。

这会带来几个问题：

- 后台可以任意切换状态，缺少明确的流转约束
- 各模块对状态的理解可能逐步分叉
- 后端没有统一校验，状态流转逻辑只能依赖前端约定
- 后续如果要增加审核人、发布时间、版本快照，基础状态机不清晰会让扩展变得更困难

因此，`P6-1` 的目标是先把“状态流转规则”这层基础设施做扎实。

## 设计目标

这一轮只解决一件核心事情：

- 为文案、简历、项目建立统一、可复用、可校验的状态流转规则

同时明确本轮边界：

- 先不引入 `publishedAt`、`reviewedBy`、`updatedBy`
- 先不做版本快照和回滚
- 先不做多人协作审批

也就是说，`P6-1` 是后续内容工作流能力的“状态机基础层”。

## 适用模块

本轮统一接入以下三个内容模块：

- 翻译文案 `TranslationRecord`
- 简历文档 `ResumeDocument`
- 项目内容 `ProjectRecord`

这三个模块都已经存在 `PublishStatus`，因此适合先统一收口。

## 状态定义

继续沿用已有四种状态：

- `draft`：草稿
- `reviewing`：审核中
- `published`：已发布
- `archived`：已归档

## 统一状态流转规则

本轮约定统一状态机如下：

- `draft` -> `reviewing`
- `draft` -> `archived`
- `reviewing` -> `draft`
- `reviewing` -> `published`
- `reviewing` -> `archived`
- `published` -> `draft`
- `published` -> `archived`
- `archived` -> `draft`

### 规则说明

#### `draft`

草稿只能继续：

- 提交审核
- 或直接归档

不允许从草稿直接发布。

这样做的目的，是先明确“审核”这一环节在流程上的存在感。

#### `reviewing`

审核中可以：

- 驳回为草稿
- 批准发布
- 或转入归档

这让“审核中”成为真正的中间状态，而不是名义状态。

#### `published`

已发布可以：

- 撤回为草稿
- 或归档

不允许从已发布直接回到审核中。

如果内容需要重新走审阅，应该先回到草稿，再重新提交审核。

#### `archived`

归档内容只能：

- 恢复为草稿

不允许从归档直接发布。

这能避免“归档内容”绕过编辑与审阅流程，直接重新上线。

## 实现策略

### 1. 共享状态机定义放到 `@repo/types`

本轮会把状态机规则放到共享层，而不是写在某个页面里。

这样做的好处是：

- `api-server` 可以直接复用
- `admin` 可以直接复用
- 后续测试也可以基于同一份规则做断言

共享层会提供：

- 状态流转映射表
- 状态中文标签
- 判断是否允许流转的工具函数
- 获取当前状态可流转目标状态的工具函数

### 2. 后端做强校验

真正的业务边界必须在后端。

因此，本轮会在以下服务层做状态流转校验：

- `translations.service.ts`
- `resume.service.ts`
- `projects.service.ts`

如果请求尝试执行非法流转，后端返回 `400`。

例如：

- `published -> reviewing` 拒绝
- `archived -> published` 拒绝
- `draft -> published` 拒绝

### 3. 后台页面改为“受控状态切换”

本轮不会再让后台编辑区展示所有状态选项，而是改为：

- 当前状态
- 当前状态允许切换到的目标状态

这样前端就不再鼓励用户尝试非法流转。

此外：

- 文案列表页会展示“主流转动作”按钮，例如“提交审核”“批准发布”“撤回草稿”
- 简历页和项目页的状态选择器只展示合法目标状态
- 状态更新失败时，前端会回滚到原状态，避免 UI 和后端状态不一致

## 当前不做的事情

本轮明确不做以下扩展：

- 记录谁提交审核
- 记录谁批准发布
- 记录发布时间
- 保存历史版本快照
- 对公开站点做“最后已发布版本”读取能力

这些内容会在后续 `P6-2` 到 `P6-5` 继续建设。

## 完成标准

`P6-1` 完成后，应满足：

- 三个核心内容模块使用同一套状态流转规则
- 后端能够拒绝非法状态流转
- 后台 UI 不再暴露非法目标状态
- 状态切换失败时，页面不会停留在错误的本地状态

## 下一步

`P6-1` 完成后，下一步建议进入：

- `P6-2`：增加发布时间、更新人、审核人字段
