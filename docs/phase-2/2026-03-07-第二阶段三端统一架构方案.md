# 第二阶段三端统一架构方案

## 背景

第一阶段完成后，项目已经具备了 Monorepo 基础、前台展示端、管理后台前端，以及一套共享类型和 UI 组件的基础结构。第二阶段原本计划沿着“先接真实数据、再逐步打通业务”的方向推进，这个方向本身没有问题，但在 P1 的实践中，我们也暴露出了更深一层的架构边界问题。

P1 过渡方案使用了 Nuxt 内部的 server routes 和临时数据访问层，成功验证了“真实数据链路可以跑通”，但它没有解决项目长期演进最关键的问题：谁来作为系统唯一的后端入口。随着用户管理、权限控制、文案管理、简历管理、项目管理和发布流逐步进入第二阶段，前台和后台继续各自承载 server route，最终会让业务边界分散在两个前端应用里。

因此，第二阶段现在需要完成一次明确的架构收敛：把系统整理为前台 Web、后台 Admin、后端 API 三端协同的统一方案，并以 `SQLite + Drizzle ORM + NestJS` 作为长期技术栈。

## 架构目标

新的第二阶段架构需要解决四个问题。

- 让前台、后台、后端三端职责清晰，不再混用边界。
- 让数据库访问方式统一，避免过渡方案和最终方案并存太久。
- 让 Web 与 Admin 都通过同一套 API 与同一份业务规则交互。
- 为后续用户、权限、i18n 文案、内容发布流和部署做准备。

## 最终技术栈

第二阶段统一后的核心技术选型如下。

- 前台展示端：Nuxt 4 + TypeScript + Nuxt UI
- 管理后台前端：Nuxt 4 + TypeScript + Nuxt UI
- 后端服务：NestJS
- 数据库：SQLite
- ORM：Drizzle ORM
- Migration：Drizzle Kit
- 共享契约：`packages/types`
- 前端请求封装：`packages/sdk`
- 数据库层：`packages/database`

这套组合的意义很明确。Nuxt 继续承担页面展示和交互层，NestJS 承担统一 API 和业务编排，Drizzle 负责数据模型、查询和迁移管理。整个 Monorepo 的角色边界会比过渡方案更稳定，也更适合后续扩展。

## 三端职责划分

### `apps/web`

`apps/web` 是公开展示站，面向访客访问。

它负责以下内容：

- 页面渲染与路由组织
- SEO、静态化和内容展示体验
- 调用 API Server 获取公开内容
- 展示来自后台管理的站点配置、简历、项目和公开文案

`apps/web` 不再承担核心业务 API 的持久化职责。即使保留少量 Nuxt server route，也只应该承担 BFF、缓存代理或 SEO 辅助处理，不再成为系统主后端。

### `apps/admin`

`apps/admin` 是管理后台前端，面向内容管理者和管理员。

它负责以下内容：

- 用户管理界面
- i18n 文案管理界面
- 简历、项目、站点设置等内容管理界面
- 表单交互、状态反馈、权限页面控制
- 调用 API Server 完成增删改查

`apps/admin` 不再直接维护最终数据写入逻辑。它只承担“管理界面”和“前端交互容器”的职责。

### `apps/api-server`

`apps/api-server` 是第二阶段新增的统一后端服务，使用 NestJS 实现。

它负责以下内容：

- 对外提供统一 REST API
- 集中处理鉴权、权限、业务校验和错误响应
- 组织领域模块，例如用户、文案、简历、项目、站点设置
- 统一接入 Drizzle ORM 和数据库
- 成为 Web 与 Admin 的唯一业务数据入口

这一步是第二阶段最重要的架构调整。项目从这里开始，真正拥有完整意义上的“前端 + 后端 + 管理后台”三端协作结构。

## Monorepo 推荐目录结构

```txt
apps/
  web/          # Nuxt 4 公开展示端
  admin/        # Nuxt 4 管理后台前端
  api/          # NestJS API Server

packages/
  database/     # Drizzle schema、db client、migrations、repositories
  sdk/          # Web/Admin 调用 API 的统一 SDK
  types/        # DTO、实体类型、共享契约
  ui/           # 共用 UI 组件
  content-schema/
```

在这个结构下，业务职责的依赖方向应该保持单向。

- `web` 和 `admin` 依赖 `sdk`、`types`、`ui`
- `api` 依赖 `database`、`types`
- `database` 不依赖具体应用
- `types` 作为共享契约层，不承载运行时业务逻辑

## 数据流设计

统一后的数据流如下。

```txt
Admin / Web
   ↓
packages/sdk
   ↓
apps/api-server (NestJS)
   ↓
packages/database (Drizzle ORM)
   ↓
SQLite
```

这个数据流有几个直接收益。

- 前后台共享同一份业务入口，不再各自定义 API。
- 业务校验与权限控制集中在 API Server。
- 数据访问策略统一在 Drizzle 和数据库包里。
- 前端应用可以更专注于体验与页面结构。

## 数据库与 ORM 设计原则

第二阶段后续全部以 `SQLite + Drizzle ORM` 为唯一标准，不再继续扩展 `node:sqlite + 手写 Repository` 作为正式方案。

`packages/database` 后续应承载以下内容：

- `schema/`：按领域拆分 Drizzle schema
- `client/`：统一数据库连接
- `migrations/`：数据库迁移文件
- `repositories/`：领域仓储层
- `seeds/`：初始数据脚本

建议优先建立这些核心表或领域结构。

- `users`
- `site_settings`
- `translation_entries`
- `resume_documents`
- `project_records`
- 后续扩展 `audit_logs`、`publish_versions`

## API Server 模块划分建议

NestJS 不需要一开始就做得过重，但模块边界要从第一天开始清晰。

第一批建议模块如下。

- `AppModule`
- `HealthModule`
- `DatabaseModule`
- `SiteSettingsModule`
- `UsersModule`
- `TranslationsModule`
- `ResumeModule`
- `ProjectsModule`
- 后续增加 `AuthModule`、`PermissionsModule`、`PublishModule`

其中，`SiteSettingsModule` 适合作为迁移后的第一个真实样板模块，因为它已经在 P1 的过渡实践里完成了业务验证。

## SDK 层职责

`packages/sdk` 后续不应继续只是 mock 数据容器，而应该升级为真正的 API Client 层。

它负责：

- 统一封装 `web` 和 `admin` 的请求方法
- 统一处理 API 地址、请求头、错误结构
- 统一输出前端可消费的数据契约
- 降低 Web/Admin 对具体请求细节的耦合

这一步完成后，前端两端的“数据来源”会从本地 mock 和局部 server route，正式切换为统一 API 服务。

## 环境与运行约定

为了让三端协作更稳定，建议第二阶段开始明确环境变量边界。

### `apps/api-server`

- `PORT`
- `DATABASE_URL`
- `JWT_SECRET` 或后续认证相关配置
- `CORS_ORIGIN`

### `apps/web`

- `NUXT_PUBLIC_API_BASE_URL`
- `NUXT_PUBLIC_SITE_URL`

### `apps/admin`

- `NUXT_PUBLIC_API_BASE_URL`
- `NUXT_PUBLIC_ADMIN_APP_NAME`

这样处理之后，Web 和 Admin 不需要再依赖各自内部的 server route 作为主数据源，而是显式地面向 API Server 工作。

## 迁移策略

当前仓库里已经存在一个 P1 过渡实现。这个实现不需要被视为失败，它完成了两个非常重要的验证。

- 验证了 Monorepo 里真实数据文件可以落盘和共享。
- 验证了“站点配置”可以作为第一条业务闭环样板。

但是，它也清楚地暴露出长期架构问题：如果继续基于 Nuxt server routes 追加更多模块，后续会让业务逻辑分散在 Web 和 Admin 两个应用内，最终提高重构成本。

因此，推荐迁移策略如下。

1. 保留 P1 文档与阶段记录，作为过渡验证的技术资产。
2. 停止继续扩展 Nuxt 内部 API 为长期方案。
3. 新建 `apps/api-server`，建立 NestJS 基础骨架。
4. 将 `packages/database` 重构到 Drizzle ORM 版本。
5. 先迁移 `site settings` 模块到 NestJS。
6. 再让 `admin` 与 `web` 改为通过 SDK 调用新 API。

这条路径能最大程度复用已完成的业务认知，同时把技术栈拉回统一方向。

## 为什么选择 NestJS

选择 NestJS，不是因为它“更流行”，而是因为它更贴合这个项目现在的阶段需求。

第一，项目已经不再只是单一展示站，而是包含公开前台、管理后台、用户管理、文案管理和内容工作流的多端系统。这样的系统需要一个独立且可维护的后端组织方式。

第二，NestJS 天然适合模块化拆分。后续无论是用户、翻译、简历、项目、权限还是发布流，都可以在统一的模块约束下继续扩展。

第三，NestJS 在 DTO、验证、守卫、拦截器、异常处理和依赖注入方面更适合中后台系统。相比继续把核心业务塞进 Nuxt server routes，它更容易形成长期可维护结构。

第四，NestJS 与 Drizzle 的组合足够轻，又能保持清晰的数据层边界，非常适合当前阶段的 SQLite 起步方案。

## 为什么坚持 SQLite + Drizzle ORM

既然第二阶段已经明确要进入真实数据库开发，就应该尽快把数据库访问方式统一，不要让项目长期处在多套数据层方案并存的状态。

选择 `SQLite + Drizzle ORM` 的原因主要有三点。

- SQLite 足够轻，适合当前阶段快速落地和迭代。
- Drizzle ORM 类型边界清晰，适合 TypeScript Monorepo。
- Drizzle 兼顾 schema、查询和 migration，能让数据库演进更有秩序。

未来如果需要切换数据库，也可以在保留 Drizzle 模型层的前提下逐步演进，而不必从零重写全部数据访问逻辑。

## 第二阶段后续开发总原则

新的阶段推进建议遵循以下原则。

- 先统一后端入口，再扩展业务模块。
- 先打通一个完整样板，再横向复制到其他领域。
- 先保证三端边界稳定，再进入发布流、权限和部署。
- 文档与开发同步推进，每个里程碑都沉淀复盘。

## 下一步

第二阶段接下来的第一个正式目标，不再是继续扩展当前过渡版 P1，而是启动新的三端统一架构实施计划。

第一步建议直接开始以下工作。

- 创建 `apps/api-server`
- 接入 NestJS
- 重建 `packages/database` 为 Drizzle ORM 版本
- 迁移 `site settings` 成为第一条正式业务链路
- 让 `admin` 和 `web` 改为调用 API Server

完成这一步之后，第二阶段才算真正进入长期有效的技术主线。
