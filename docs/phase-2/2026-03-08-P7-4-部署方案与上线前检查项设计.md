# P7-4：部署方案与上线前检查项设计

## 背景

到 `P7-3` 为止，第二阶段已经补齐：

- API Server 关键链路测试基础
- Web / Admin 关键流程联调验证
- 环境变量与多端启动文档

这意味着当前项目已经不再卡在“本地怎么跑”，而是进入另一个更接近交付的问题：

- 这三端未来应该怎么部署
- 域名和反向代理关系应该怎么设计
- SQLite 这种本地文件数据库上线时要注意什么
- 上线前到底要检查哪些点，才能避免基础性事故

因此，`P7-4` 的目标，不是今天就把项目真正发到某个平台上，而是先整理出一套明确的部署蓝图和上线前检查清单，让后续正式部署时不再临时拍脑袋。

## 本轮目标

本轮聚焦四类成果：

- 梳理三端推荐部署拓扑
- 收口线上环境变量建议
- 补一份可执行的上线前检查清单
- 顺手修复会影响部署真实性的配置问题

## 当前部署问题

### 1. Web 域名配置仍偏静态

当前 `apps/web` 虽然已经有：

- `runtimeConfig.public.siteUrl`

但默认值仍是写死的示例域名。为了让部署更自然，应该允许通过环境变量直接覆盖：

- `NUXT_PUBLIC_SITE_URL`

必要时还应允许同时覆盖：

- `NUXT_PUBLIC_SITE_NAME`
- `NUXT_PUBLIC_SITE_DESCRIPTION`
- `NUXT_PUBLIC_DEFAULT_LOCALE`

### 2. API CORS 还缺少线上边界

当前 `apps/api-server` 默认：

- `origin: true`

这对本地开发很方便，但线上部署时更合理的做法是允许通过环境变量收口指定来源，例如：

- `CORS_ORIGIN=https://web.example.com,https://admin.example.com`

这样可以让本地开发保持宽松，线上再收口为明确白名单。

### 3. Sitemap 不能继续依赖旧 mock 数据

当前 Web 页面主链路已经切到公开 API，但 `sitemap.xml` 仍然存在使用旧 SDK mock 的风险。

如果 sitemap 不读取真实公开项目数据，就会带来两个问题：

- 站点地图和真实线上内容不一致
- 上线后搜索引擎看到的 URL 集合可能是旧的演示数据

所以 `P7-4` 需要把 sitemap 的动态项目来源切到真实公开 API。

## 推荐部署拓扑

当前阶段更推荐采用“三服务分离”的简单部署方式：

- `web`：一个 Nuxt 服务
- `admin`：一个 Nuxt 服务
- `api-server`：一个 NestJS 服务
- `SQLite`：挂载在 `api-server` 所在主机的持久化目录

推荐域名关系：

- `web.example.com` -> `apps/web`
- `admin.example.com` -> `apps/admin`
- `api.example.com` -> `apps/api-server`

若使用统一根域，也可以采用：

- `example.com` -> `web`
- `admin.example.com` -> `admin`
- `api.example.com` -> `api-server`

## SQLite 部署原则

当前阶段继续使用 `SQLite + Drizzle ORM`，部署时必须注意：

- 数据库文件必须放在持久化目录
- 不能把数据库文件仅放在临时构建目录
- 发布前要先做备份
- migration 必须在服务启动前或部署过程中明确执行

推荐原则：

- `REPO_DATABASE_PATH` 指向明确的持久化路径
- 上线前先备份旧库
- 先执行 migration，再切换服务版本

## 上线前检查范围

### Web

- 首页、简历页、项目列表页、项目详情页可访问
- `robots.txt` 可访问
- `sitemap.xml` 可访问
- canonical 与 OG 图使用真实线上域名
- 公开页面读取的是正式 API，而不是本地地址

### Admin

- `/login` 正常打开
- 登录、退出登录正常
- 权限受限页面正确跳转
- 关键管理页可打开
- Admin 的 API 地址指向正式 `/api/admin`

### API Server

- `/api/health` 正常返回
- `/api/public/release` 正常返回
- 登录接口可用
- 关键管理接口在已登录状态下可用
- CORS 来源符合线上部署关系

### Database

- 目标数据库路径正确
- migration 已执行
- 备份已完成
- 发布批次存在有效激活版本
