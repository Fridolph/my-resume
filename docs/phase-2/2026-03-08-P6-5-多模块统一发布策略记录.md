# P6-5：多模块统一发布策略记录

## 背景

在 `P6-4` 完成之后，内容系统已经具备：

- 状态流转
- 审计字段
- 历史快照
- 版本恢复

但公开站点仍然存在一个阶段性问题：

> 前台读到的是“各模块各自当前 published 的内容”，而不是“一次统一发布出来的内容集合”。

这会导致前台在某些时候出现版本错位：

- 简历是这次新发布的
- 项目还是上一次的
- 文案已经被单独改过一轮

也就是说，模块内部工作流已经很完整了，但站点级发布仍然是分散的。

所以这一轮 `P6-5` 的目标，就是把公开站点的数据读取方式，从“模块各自发布”升级为“统一发布批次”。

## 本次实现

### 1. 新增统一发布批次模型

本次先在共享类型里新增了统一发布结构：

- `ContentReleaseStatus`
- `ContentReleaseRecord`
- `PublicReleaseSnapshot`

这样系统开始拥有“站点级发布批次”这个概念，而不再只有模块级 `published` 状态。

### 2. 数据库新增 `content_releases` 表

本次新增：

- `packages/database/src/schema/content-releases.ts`
- `packages/database/src/content-releases.ts`
- `packages/database/drizzle/0008_content_releases.sql`

当前这张表主要记录：

- 批次 ID
- 批次名称
- 当前状态
- 简历版本引用
- 文案版本引用集合
- 项目版本引用集合
- 创建人 / 激活人
- 创建时间 / 激活时间 / 更新时间

这意味着统一发布批次并不复制整份内容，而是引用已经存在的内容版本快照。

### 3. 发布批次支持“创建并激活”与“重新激活历史批次”

本次在数据层和 API 层都建立了两类核心动作：

- 创建并激活新的统一发布批次
- 将历史批次重新激活

当前规则是：

- 同一时刻只允许一个 `active` 批次
- 新批次激活时，旧的 `active` 自动转成 `archived`
- 历史批次重新激活时，当前 `active` 同样会被归档

这一步让统一发布中心不仅能“发布一次”，还能“切换回旧站点批次”。

### 4. 公开接口改为优先读取激活批次

本次最重要的链路改造，是把公开接口切换到了统一发布批次之上。

当前：

- `GET /api/public/resume`
- `GET /api/public/projects`
- `GET /api/public/translations`

都已经不再直接读取主表当前值，而是优先解析：

- 当前激活发布批次引用的版本快照

同时新增：

- `GET /api/public/release`

用于查看当前公开站点到底在消费哪一个发布批次。

这一步完成后，Web 就真正开始站在“统一发布快照”之上运行，而不是零散地从各模块当前状态拼装内容。

### 5. 管理后台新增“统一发布中心”页面

本次新增页面：

- `apps/admin/app/pages/releases/index.vue`

同时把导航入口接入：

- `apps/admin/app/composables/useAdminAccess.ts`

页面当前支持：

- 查看当前激活批次
- 查看历史批次列表
- 创建并激活新批次
- 重新激活历史批次

本轮先采用最轻量的操作流，重点验证链路与数据模型闭环。

### 6. 保留一个兼容性兜底

本轮还补了一层很重要的兼容逻辑：

- 如果当前还没有激活发布批次
- 或者当前批次暂时无法建立
- 则公开接口会退回到当前公开内容的兼容读取模式

这样做的原因是：

- 项目已经处于渐进式重构中
- 数据库可能存在尚未进入统一发布中心的旧内容
- 本地开发和阶段切换期间，不能因为批次暂未建立就让前台直接不可用

因此当前策略是：

- 公开站点 **优先** 读取激活批次
- 没有批次时 **降级** 到兼容公开内容读取

这也符合 `P6-5` 设计里“优先读取激活批次”的要求。

## 本次实现的价值

这一轮最大的价值，是让公开站点终于有了“内容一致性边界”。

也就是说，现在系统不仅知道：

- 某条内容有没有发布

还开始知道：

- 前台当前到底应该消费哪一批统一发布出来的内容版本

这让内容治理从“模块级工作流”真正走向“站点级工作流”。

做到这里，第二阶段的内容工作流主链路已经基本闭环：

- 模块级状态流转

- 模块级历史版本
- 模块级恢复

- 内容治理
- 版本治理
