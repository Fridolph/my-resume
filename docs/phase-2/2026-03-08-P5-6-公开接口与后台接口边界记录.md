# P5-6：公开接口与后台接口边界记录

## 背景

在 `P5-5` 完成之后，后台页面和接口级权限已经开始对齐，但 API 仍然存在一个结构性问题：

- `web` 和 `admin` 都在使用同一批无语义区分的接口路径
- 某些读取接口虽然同时服务公开站点和后台，但路径本身并没有体现边界

这会让后续维护变得越来越模糊。

例如：

- 新人很难一眼判断某条接口是否应该公开暴露
- `web` 容易不小心复用后台能力
- 后台读取接口虽然页面受权限控制，但接口路径仍和公开接口混在一起

因此，`P5-6` 的核心不是再加一个守卫，而是把“公开消费”和“后台管理”的接口边界真正固化到路径和调用层。

## 本次实现

### 1. API 路径按 `public` / `admin` 分区

本次把 `api-server` 中原本混合的路由拆成了两类：

#### 公开接口

- `GET /api/public/site-settings`
- `GET /api/public/resume`
- `GET /api/public/projects`
- `GET /api/public/translations`

这一组接口只面向 `web`。

#### 后台接口

- `POST /api/admin/auth/login`
- `POST /api/admin/auth/logout`
- `GET /api/admin/auth/me`
- `GET /api/admin/users`
- `POST /api/admin/users`
- `PUT /api/admin/users/:userId`
- `GET /api/admin/translations`
- `PUT /api/admin/translations/:translationId`
- `GET /api/admin/resume`
- `PUT /api/admin/resume`
- `GET /api/admin/projects`
- `POST /api/admin/projects`
- `PUT /api/admin/projects/:projectId`
- `DELETE /api/admin/projects/:projectId`
- `GET /api/admin/site-settings`
- `PUT /api/admin/site-settings`

这一组接口只面向 `admin`。

同时，旧的混合路径开始失效，例如：

- `/api/projects`
- `/api/site-settings`
- `/api/auth/login`

现在都会返回 `404`，从路径层直接阻止旧边界继续扩散。

### 2. 后台读取接口正式纳入权限控制

这轮之前，某些后台读取接口仍然沿用公开读路由。

本次已经把后台读取接口统一切到 `/api/admin/*`，并继续挂上真实 `read` 权限：

- `GET /api/admin/users` -> `user.read`
- `GET /api/admin/translations` -> `translation.read`
- `GET /api/admin/resume` -> `resume.read`
- `GET /api/admin/projects` -> `project.read`
- `GET /api/admin/site-settings` -> `site.read`

这样后台页面、后台接口、角色权限三者终于开始使用同一套边界规则。

### 3. Web SDK 与 Admin SDK 开始分层

为了避免 `web` 继续拿着“后台客户端”访问公开接口，本次新增了公开内容客户端：

- `packages/sdk/src/public-content.ts`

现在：

- `apps/web` 使用公开内容客户端
- `apps/admin` 继续使用带鉴权与写能力的后台客户端

这一步很重要，因为边界不应该只停留在后端路由层，也应该体现在前端调用层。

### 4. Nuxt 运行时配置也同步分流

本次同时调整了两端的默认 API 基地址：

- `apps/web` -> `http://localhost:3011/api/public`
- `apps/admin` -> `http://localhost:3011/api/admin`

从这一轮开始：

- `web` 天然只会命中公开接口
- `admin` 天然只会命中后台接口

## 本次实现的价值

这一轮最大的价值是：接口边界终于具备“可见性”。

以前需要靠约定理解：

- 哪些接口给公开站点用
- 哪些接口给后台用
- 哪些读取接口其实不应该继续公开暴露

而现在这些边界会直接体现在：

- 路由路径
- SDK 客户端
- Nuxt 运行时配置
- 后台权限控制

这会让后面的开发更稳定，也能明显降低误用概率。

## 一个阶段性结论

这一轮做完后，第二阶段的认证、权限和接口边界已经基本闭环：

- 后台登录态已经是真实的
- 后端写接口和读接口都已经纳入权限控制
- 页面层已经按权限收口
- 路由层已经完成公开域和后台域分区

这也意味着下一步可以更放心地进入 `P6`，去做真正的内容工作流能力，而不是继续在边界问题上反复返工。
