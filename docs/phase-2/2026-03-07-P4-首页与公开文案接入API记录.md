# P4：首页与公开文案接入 API 记录

## 背景

`P4` 的前一轮已经完成了第一批公开内容替换：

- 简历页改为读取 `api-server`
- 项目列表页改为读取 `api-server`
- 项目详情页改为读取真实项目数据

但这还不算 `web` 端真正完成 API 化，因为首页仍然承担着站点入口、阶段说明和内容聚合的职责，公开文案也仍然主要依赖前端静态字典。

因此，这一轮 `P4` 的重点不再是继续扩展新页面，而是把首页和公开文案这两块真正打通，让前台首页也开始消费后台真实内容，并让后台翻译结果能够覆盖前端默认文案。

## 这一轮要解决的问题

这次收口主要处理两类问题。

第一类问题，是首页内容来源还不够正式。

虽然 `site-settings`、`resume`、`projects` 都已经进入统一 API 链路，但首页仍然在消费原来的结构化展示数据。这样会导致站点最重要的入口页，没有和后台形成完整闭环。

第二类问题，是公开文案的管理边界还不清晰。

如果所有公开文案都继续写死在 `web` 端，那么后台虽然已经有翻译模块，但它还无法真正影响前台展示结果。这会让“管理后台维护 i18n 文案”这件事停留在能力层，而没有进入实际使用层。

## 最终方案

这一轮采用的是“后台翻译优先，静态字典兜底”的方案。

也就是说：

- 后台已发布翻译优先作为公开站点文案来源
- 当前还未被后台覆盖的 key，仍然回落到 `web` 本地字典
- 页面结构不重写，主要在 `composables` 层替换数据源

这样做的价值很明确：

- 风险低
- 可逐步迁移
- SSR 兼容性更好
- 后台翻译可以逐步接管公开文案，而不是一次性推倒重做

## 本次实现

### 1. 新增公开翻译查询层

本次新增了 `apps/web/app/composables/usePublicTranslationsQuery.ts`。

这层负责在 `web` 端统一拉取后台翻译数据，并只保留 `published` 状态的记录作为公开文案源。这样可以确保后台未发布或草稿文案不会直接暴露到公开页面。

随后在 `apps/web/app/app.vue` 中，通过顶层 `await usePublicTranslationsQuery()`，让这批公开翻译在 SSR 阶段就进入 Nuxt state。这样页面在首屏渲染时就能直接读取后台翻译，而不是等客户端再补一次。

### 2. 重构 `useWebLocale`

本次没有引入新的 i18n 框架，而是继续沿用当前轻量方案，并在 `apps/web/app/composables/useWebLocale.ts` 中增加一层远端翻译映射。

当前 `t(key)` 的读取顺序已经改为：

1. 后台已发布翻译
2. 当前前端静态字典
3. 中文静态字典兜底
4. 最后回退为 key 本身

这意味着：

- `nav.*`
- `footer.*`
- `page.home.description`
- `home.intro.*`
- `home.stats.*`
- `home.features.*`

这些公开站点文案，已经具备“后台覆盖前端”的能力。

### 3. 首页改为聚合真实 API 内容

本次继续沿用“页面结构稳定，数据层替换”的策略，主要改造 `apps/web/app/composables/useWebContent.ts`。

`useHomePageContent()` 现在会并行读取：

- `site-settings`
- `projects`

然后组装当前首页需要的 DTO。

这样做之后，首页已经开始具备真实的后台内容特征：

- 站点标题与 SEO 来自后台站点设置
- 首页统计项来自真实项目数据与语言定义
- 首页入口卡片文案优先来自后台翻译模块

这说明首页不再只是一个静态入口页，而是开始承担正式内容聚合页的职责。

### 4. 扩展翻译种子，保证首轮联调可见

为了让当前阶段能够直接联调验证，本次还补充了 `packages/database/src/translations.ts` 中的公开站点翻译种子。

新增覆盖范围包括：

- 导航文案
- Footer 文案
- 首页描述
- 首页 intro / stats / features 相关文案

同时，这次继续使用逐条插入并 `onConflictDoNothing()` 的方式，确保：

- 新环境首次启动时可以自动拥有一批可见的公开翻译
- 已经存在的翻译记录不会被种子脚本直接覆盖
- 后续人工在后台维护过的文案仍然保持优先级

## 这次实现的关键判断

### 为什么不是直接把所有公开文案都迁到后台？

因为当前阶段更重要的是先建立稳定的可运行链路，而不是一次性追求“后台完全接管所有文案”。

如果一口气完全移除前端静态字典，那么：

- 首次启动体验会变差
- 未录入翻译的 key 会直接暴露出来
- 联调成本和排障成本都会上升

所以本次采用“后台优先、前端兜底”的方式，是更适合当前里程碑节奏的方案。

### 为什么依然不大改页面结构？

因为我们这一阶段的核心目标，是把数据源正式切换到统一 API，而不是重做视觉层。

当前首页、导航、Footer、简历页、项目页的结构都已经稳定，继续保持页面结构不动，把变化集中在 `composables` 层，能最大限度降低回归风险，也更利于后续继续替换更多公开内容。

## 本地自测结果

本次已完成以下验证。

### 1. 类型检查通过

执行通过：

- `pnpm --dir apps/web typecheck`
- `pnpm --dir apps/api-server typecheck`

### 2. API 翻译数据可读

已验证：

- `GET http://localhost:3011/api/translations` 可以正常返回翻译记录
- 返回结果中已包含 `footer.*`、`nav.*`、`home.*`、`page.home.description` 等公开 key
- 公开站点只消费 `published` 状态的翻译记录

### 3. 首页 SSR 已接入真实内容

已验证：

- `http://localhost:3000/` 可正常 SSR
- 页面标题已经读取后台 `site-settings`
- 首页描述已经支持后台翻译覆盖
- 首页统计项已读取真实项目数据并在前台渲染

### 4. 中英文公开文案覆盖生效

已验证：

- 中文首页导航、Footer、首页说明可正常读取公开翻译
- 英文页面在传入 `web-locale=en-US` 时，也可正常读取英文公开翻译
- 当后台翻译缺失时，前端静态字典仍然可正常兜底

## 当前结果

完成这一轮之后，`P4` 已经可以视为整体完成。

当前 `web` 端已经具备以下特征：

- 首页接入真实站点设置与项目聚合内容
- 简历页接入真实简历数据
- 项目列表与详情页接入真实项目数据
- 导航、Footer、首页公开文案开始接入后台翻译结果
- 公开站点形成“后台翻译优先，静态字典兜底”的稳定运行模式

这意味着第二阶段的公开站点，已经不再只是“结构展示”，而是正式进入了“消费后台真实内容”的阶段。

## 这一轮工作的价值

这次工作的价值主要有三点。

第一，前后台内容边界真正清晰了。

后台不只是维护数据，还已经能够实际影响前台首页、导航、Footer 和 SEO 描述的展示结果。

第二，公开 i18n 能力真正落地了。

翻译模块从“后台可编辑”变成了“前台可消费”，这让多语言能力第一次真正进入生产链路。

第三，后续继续扩展会更顺畅。

现在 `web` 端已经有了成熟的接入模式：

- 在 `app.vue` 预取公共配置和公开翻译
- 在 `composables` 层聚合真实 API 数据
- 在页面层尽量只负责展示

这个模式后续可以直接复用到更多公开内容模块。

## 下一步

`P4` 收口之后，第二阶段可以继续推进下一类更系统化的能力，例如：

- 认证与权限边界
- 发布流与审核流
- 内容版本与回滚能力
- 前后台接口分层与治理

从里程碑顺序上看，下一步更适合进入 `P5`。
