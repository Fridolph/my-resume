# P6-2：发布时间、更新人、审核人字段记录

## 背景

`P6-1` 让内容工作流第一次真正具备了状态机，但系统仍然缺少最基础的可追踪信息。

如果后台只能看到：

- 草稿
- 审核中
- 已发布
- 已归档

却看不到：

- 谁最后改了内容
- 谁真正执行了发布
- 内容是什么时候进入发布态

那么工作流仍然是“有流程、没痕迹”。

所以这一轮 `P6-2` 的目标，是先把最基础的审计字段补齐。

## 本次实现

### 1. 共享类型增加内容审计字段

本次在 `packages/types/src/index.ts` 中新增了：

- `ContentActorSummary`
- `ContentAuditFields`

并让以下记录类型统一继承这些字段：

- `TranslationRecord`
- `ResumeDocument`
- `ProjectRecord`

这意味着内容模块从这一轮开始，正式具备统一的审计结构。

### 2. 数据库增加三个持久化字段

本次为以下表补了字段：

- `translations`
- `resume_documents`
- `projects`

新增列：

- `updated_by`
- `reviewed_by`
- `published_at`

同时新增迁移文件：

- `packages/database/drizzle/0006_content_audit_fields.sql`

这样本地 SQLite 在服务启动后，就能把这三类字段真正持久化下来。

### 3. 数据层统一负责生成审计字段

本次新增：

- `packages/database/src/content-audit.ts`

这里统一封装了审计字段生成逻辑。

当前规则是：

- 每次保存都会更新 `updatedBy`
- 内容进入 `published` 时写入 `reviewedBy` 与 `publishedAt`
- 内容离开 `published` 时清空 `reviewedBy` 与 `publishedAt`
- 已发布内容继续保存且仍保持发布态时，沿用既有发布时间与审核人

这一步很关键，因为审计字段不应该由页面拼接，而应该由数据层根据“当前状态 + 下一状态 + 当前操作者”统一计算。

### 4. 后台接口接入当前用户并自动写审计字段

本次把 `@CurrentUser()` 接到了以下后台写接口：

- 文案更新
- 简历更新
- 项目创建 / 更新

也就是说，从这一轮开始：

- 接口不只知道改了什么
- 也知道是谁改的

### 5. 后台页面开始展示审计字段

本次同时把后台 UI 补上：

- 文案页：列表卡片展示更新人、审核人、发布时间
- 项目页：列表卡片与编辑区展示更新人、审核人、发布时间
- 简历页：状态区域增加审计信息卡片

这样本轮新增的字段可以直接在管理后台看到，而不只是留在数据库里。

## 本次实现的价值

这一轮最大的价值，是让内容工作流开始具备“谁”和“何时”的维度。

也就是说，现在系统不仅知道：

- 内容在哪个状态

还开始知道：

- 最近是谁改了它
- 谁让它进入发布态
- 当前发布是什么时间发生的

这为后续版本快照、历史记录、审计视图打下了很重要的基础。

做到这里，内容工作流开始从“状态管理”走向“基础审计”。
