# P6-3：版本记录与历史快照记录

## 背景

`P6-1` 解决了统一状态流转，`P6-2` 解决了更新时间、更新人、审核人和发布时间。

但是系统仍然只有“当前态”，缺少“历史态”。

如果后台只能看到：

- 当前这条内容是什么状态
- 最近是谁改了它
- 它什么时候发布过

却看不到：

- 它之前长什么样
- 一共改过多少次
- 后续要回滚时应该回到哪个版本

那么内容工作流就还不完整。

所以这一轮 `P6-3` 的目标，是先把版本记录与历史快照能力补齐，并且让后台可以直接查看这些历史版本。

## 本次实现

### 1. 共享类型补齐版本模型

本次在 `packages/types/src/index.ts` 中新增了：

- `ContentModuleType`
- `ContentVersionChangeType`
- `ContentVersionRecord`
- `TranslationVersionRecord`
- `ResumeVersionRecord`
- `ProjectVersionRecord`

这样前后端与 SDK 都开始使用统一版本结构，而不是各自维护一套历史数据类型。

### 2. 数据库新增统一版本表

本次新增：

- `packages/database/src/schema/content-versions.ts`
- `packages/database/src/content-versions.ts`
- `packages/database/drizzle/0007_content_versions.sql`

核心表为：

- `content_versions`

当前字段包括：

- `module_type`
- `entity_id`
- `version`
- `status`
- `change_type`
- `snapshot`
- `created_by`
- `created_at`

这里采用统一快照表，而不是为文案、简历、项目分别建三张历史表。这样可以让：

- 数据层只维护一套版本逻辑
- API 层只维护一套历史查询模式
- 后续回滚和统一审计视图更容易扩展

### 3. 内容写入后自动生成历史快照

本次把版本写入逻辑接到了三类内容的数据层：

- `packages/database/src/translations.ts`
- `packages/database/src/resume-documents.ts`
- `packages/database/src/projects.ts`

当前规则如下：

- 种子数据首次进入系统时，生成 `seed` 版本
- 新建项目时，生成 `create` 版本
- 每次更新内容时，生成 `update` 版本

同时还补了一层兼容逻辑：

- 如果数据库里已经有老数据，但还没有历史版本
- 那么首次读取这条内容时，也会自动补一条首版快照

这一步很重要，因为当前项目已经经历了多个阶段开发，数据库里存在早于 `P6-3` 的内容。如果不做这层兼容，历史列表就会出现“新内容有版本、老内容没版本”的断层。

### 4. API Server 增加版本查询接口

本次在 NestJS 中新增了三类版本读取接口：

- `GET /api/admin/translations/:translationId/versions`
- `GET /api/admin/resume/versions`
- `GET /api/admin/projects/:projectId/versions`

对应服务层也已经补齐：

- 文案版本查询
- 简历版本查询
- 项目版本查询

这样后台不需要直接碰数据库，就可以通过统一 API 获取历史快照。

### 5. SDK 补齐版本查询能力

本次在 `packages/sdk` 中补充了：

- `listTranslationVersions()`
- `listResumeVersions()`
- `listProjectVersions()`

这样 Admin 端继续保持通过 SDK 访问 API，而不是在页面中自行拼装请求。

### 6. Admin 页面开始支持查看版本历史

本次把版本历史接到了三个管理页面：

- 文案页：每条文案卡片可按需展开版本历史
- 简历页：增加“版本历史”卡片
- 项目页：编辑区增加“版本历史”卡片

后台当前展示的重点信息包括：

- 版本号
- 状态
- 版本来源类型（种子 / 创建 / 更新）
- 创建人
- 创建时间
- 当前语言下的关键信息摘要

这保证了版本能力不只是“落库成功”，而是真正进入管理后台可见范围。

## 一个兼容问题与本次修正

这次实现里有一个很典型的阶段性问题：

> 新功能从现在开始会自动写历史，但老数据并不会自动长出历史。

这类问题在渐进式重构中非常常见。

如果只考虑新写入流程，代码很容易实现，但真实项目里通常已经存在：

- 旧的种子数据
- 旧的测试数据
- 旧阶段保留下来的演示数据

所以本次除了在“写入时生成快照”之外，还额外补上了“首次读取时补首版快照”的兼容逻辑。这样现有库里的老数据也能平滑进入版本体系，不需要人工清库重建。

## 阶段结论

做到这里，内容系统已经不再只有“当前值”，而是开始拥有“历史值”。

这意味着接下来的 `P6-4` 可以不再从零设计回滚，而是直接建立在现有快照模型上：

- 读版本
- 选版本
- 恢复版本
- 再写一条恢复后的新版本

所以 `P6-3` 的本质，是为回滚能力先铺好数据底座。
