# P5-1：认证与角色权限模型设计

## 背景

第二阶段走到 `P5` 时，三端的内容链路已经基本打通：

- `web` 已经开始消费 `api-server` 的真实公开内容
- `admin` 已经具备用户、文案、简历、项目、站点设置等模块
- `api-server` 已经承接正式 API 与 SQLite + Drizzle ORM 数据层

但当前系统还缺少一个足够稳定的“身份与权限基础层”。

这会带来几个明显问题：

- `admin` 登录仍然偏演示态
- 角色定义不够完整，难以承接后续的权限控制
- 权限散落在前端与数据层的局部实现中，缺少统一源
- 后续如果进入真实鉴权、权限守卫、后台接口隔离，改动面会变大

因此，`P5` 的第一步不应该立刻冲进登录接口开发，而应该先把角色与权限模型正式定稿，并收口成共享规范。

## 设计目标

`P5-1` 的目标很明确：

- 定义统一的角色集合
- 定义统一的权限 key 集合
- 建立角色到权限的稳定映射
- 让前端、后端、数据库层引用同一份模型
- 为下一步真实鉴权和权限守卫打基础

这一阶段不直接解决“真实登录”，而是先解决“权限规则来源唯一化”。

## 角色模型

当前阶段采用轻量 RBAC 模型。

本次确定五类角色：

- `super-admin`
- `admin`
- `editor`
- `translator`
- `viewer`

### 角色定位

#### `super-admin`

拥有全部后台能力，用于系统级管理与高风险操作。

典型职责：

- 用户管理
- 角色调整
- 站点设置修改
- 所有内容模块管理
- 后续系统级策略配置

#### `admin`

作为常规后台管理员，拥有绝大多数运营与内容管理权限，但保留和 `super-admin` 的层级差异。

典型职责：

- 内容管理
- 翻译管理
- 项目与简历管理
- 常规站点设置维护
- 用户管理

#### `editor`

主要负责内容生产与内容维护。

典型职责：

- 简历编辑
- 项目编辑
- 查看站点设置
- 查看翻译内容

#### `translator`

聚焦 i18n 文案维护。

典型职责：

- 翻译列表查看
- 翻译内容编辑
- 查看后台基础信息

#### `viewer`

只读账号，用于演示、审阅或运营查看。

典型职责：

- 查看后台概览
- 查看用户、文案、简历、项目、站点设置
- 不具备写权限

## 权限模型

本次权限 key 仍然保持轻量、可读、可扩展的风格。

当前统一权限集合为：

- `dashboard.read`
- `user.read`
- `user.write`
- `translation.read`
- `translation.write`
- `resume.read`
- `resume.write`
- `project.read`
- `project.write`
- `site.read`
- `site.write`

## 角色与权限映射

### `super-admin`

拥有全部权限。

### `admin`

拥有绝大多数后台管理权限：

- 仪表盘查看
- 用户读写
- 翻译读写
- 简历读写
- 项目读写
- 站点设置读写

### `editor`

偏内容编辑：

- 仪表盘查看
- 翻译查看
- 简历读写
- 项目读写
- 站点设置只读

### `translator`

偏翻译协作：

- 仪表盘查看
- 翻译读写
- 站点设置只读

### `viewer`

偏审阅与查看：

- 仪表盘查看
- 用户查看
- 翻译查看
- 简历查看
- 项目查看
- 站点设置查看

## 为什么现在要加 `site.read`

此前权限模型只有 `site.write`，没有 `site.read`。

这会导致一个问题：很多角色其实应该能“查看站点设置”，但不应该能“修改站点设置”。如果缺少 `site.read`，就只能把“看”和“改”混在一起处理。

因此，这次补上 `site.read`，是为了让读写权限边界更清晰，也更符合后续页面级和接口级权限控制的需要。

## 共享模型落点

本次设计要求角色与权限模型必须收口到共享层，而不是继续散落在：

- `admin` composable
- `database` seed
- `api-server` schema

因此，共享模型落点定为：

- `packages/types`

这里将统一维护：

- `roleKeys`
- `permissionKeys`
- `rolePermissions`
- `getRolePermissions()`

这样后续三端都从同一个源读取模型。

## 本阶段实现范围

`P5-1` 本次会覆盖：

- 共享角色与权限常量
- 数据库用户种子同步到统一角色模型
- 后端用户 schema 改为引用共享角色定义
- 管理后台 mock 登录态改为引用共享权限映射
- 管理后台用户页改为展示完整角色集合

## 本阶段刻意不做的事

为了保证节奏稳定，这一阶段不会直接做下面这些内容：

- 不接入真实登录接口
- 不实现 token / session 服务端签发
- 不实现 NestJS 权限守卫
- 不做后台接口路径重构
- 不做最终权限矩阵 UI

这些内容会分别留到 `P5-2` 以后继续推进。

## 下一步

`P5-1` 完成后，下一步建议直接进入：

- `P5-2`：`api-server` 基础鉴权机制

到那一步时，这一轮已经定义好的角色与权限模型就可以直接作为真实认证与鉴权的底层规则来源。
