# P7-3：环境变量与多端启动文档设计

## 背景

到 `P7-2` 为止，第二阶段的三端架构已经可以本地联调：

- `apps/web`：Nuxt 4 展示端
- `apps/admin`：Nuxt 4 管理后台前端
- `apps/api-server`：NestJS API Server
- `packages/database`：`SQLite + Drizzle ORM`

并且当前已经完成了一轮真实联调，验证了：

- 后台登录与权限边界
- 公开站点关键页面
- 统一发布批次对公开内容的影响

但从“能联调”到“可交接、可复现、可维护”之间，还差最后一层非常关键的基础设施文档化工作：

- 哪些环境变量是三端真正依赖的
- 本地开发默认端口分别是什么
- 首次启动前需要做哪些准备
- 常用命令应该从根目录执行还是在子项目执行
- 数据库相关命令应该在哪个目录执行

如果这部分不整理清楚，项目虽然功能上已经进入第二阶段后半程，但对后续开发者来说仍然会有几个明显问题：

- 不知道哪些变量必须配置，哪些变量有默认值
- 不知道 Web / Admin 应该分别连哪个 API 前缀
- 不知道数据库命令应该在 `packages/database` 还是 `apps/api-server` 执行
- 不知道如何快速同时启动三端进行联调

因此，`P7-3` 的目标，就是把三端环境变量、启动命令和本地开发约定正式收口为统一文档，并尽量补足最小可用的示例文件。

## 本轮目标

本轮聚焦三类收口工作：

- 梳理三端环境变量边界与默认值
- 统一本地开发与联调启动命令
- 为后续接手者提供一份从安装到启动、从数据库到联调的最小操作手册

本轮暂时不做：

- Docker 化
- 一键脚本式进程编排
- CI 环境变量托管配置
- 生产环境密钥托管平台接入

也就是说，`P7-3` 解决的是“本地开发与联调说明清晰”，而不是“一次性补完部署平台自动化”。

## 当前问题盘点

结合当前仓库现状，`P7-3` 需要优先解决以下问题。

### 1. 三端环境变量分散且命名未集中说明

目前可以从代码中读出几类关键变量：

- `PORT`
- `REPO_DATABASE_PATH`
- `NUXT_PUBLIC_API_BASE_URL`

但这些变量目前主要散落在：

- `apps/api-server/src/main.ts`
- `packages/database/src/client.ts`
- `apps/web/nuxt.config.ts`
- `apps/admin/nuxt.config.ts`

这对熟悉代码的人问题不大，但对后续接手者并不友好。

### 2. `web` 与 `admin` 目前都使用同名公开变量

当前：

- `apps/web` 使用 `NUXT_PUBLIC_API_BASE_URL` 指向 `/api/public`
- `apps/admin` 使用 `NUXT_PUBLIC_API_BASE_URL` 指向 `/api/admin`

虽然单独开发时能工作，但从文档和认知层面看并不直观。

更清晰的方式应该是：

- `apps/web` 使用 `NUXT_PUBLIC_PUBLIC_API_BASE_URL`
- `apps/admin` 使用 `NUXT_PUBLIC_ADMIN_API_BASE_URL`

同时为了兼容当前已有本地习惯，可以保留旧变量作为 fallback，避免直接破坏现有环境。

### 3. README 仍残留模板内容

当前 `apps/web/README.md` 与 `apps/admin/README.md` 仍然大量保留 Nuxt UI Starter 模板内容，包括：

- 默认端口描述不准确
- 没有反映当前 monorepo 架构
- 没有说明 API Server 和数据库依赖
- 没有说明实际联调方式

这部分如果不替换，后续阅读者会被明显误导。

### 4. 根目录缺少统一启动与联调说明

目前根目录脚本已经有：

- `dev:web`
- `dev:admin`
- `dev:api`
- `db:generate`
- `db:migrate`

但还缺少：

- 三端端口说明
- 推荐启动顺序
- 数据库初始化顺序
- 常见问题排查

## 设计原则

### 1. 以“新人第一次拉仓库即可跑通”为标准

`P7-3` 的文档设计必须面向：

- 第一次接手这个项目的人
- 未来回来看文档的自己

因此文档必须尽量回答两个问题：

- 我该先执行什么
- 如果跑不起来，我应该先查什么

### 2. 变量命名尽量语义化，同时保留兼容

对于 Web / Admin 的 API 地址变量，本轮建议：

- 引入更语义化的新变量名
- 继续兼容旧变量名

这样既不打断当前环境，也能让后续文档更清晰。

### 3. 文档与代码要同步落地

本轮不只写博客，还要让代码配置和文档真正对齐，例如：

- Nuxt runtimeConfig 与环境变量命名对齐
- `.env.example` 或同类示例文件补上
- README 更新为当前项目真实说明

## 实施方案

## 代码层

本轮建议落地以下调整：

### 1. 收口 Web 环境变量命名

在 `apps/web/nuxt.config.ts` 中优先读取：

- `NUXT_PUBLIC_PUBLIC_API_BASE_URL`

同时兼容旧变量：

- `NUXT_PUBLIC_API_BASE_URL`

### 2. 收口 Admin 环境变量命名

在 `apps/admin/nuxt.config.ts` 中优先读取：

- `NUXT_PUBLIC_ADMIN_API_BASE_URL`

同时兼容旧变量：

- `NUXT_PUBLIC_API_BASE_URL`

### 3. 提供仓库级环境变量示例

建议在根目录提供：

- `.env.example`

用于说明：

- API 端口
- 数据库路径
- Web 公共 API 地址
- Admin 管理 API 地址

如果后续需要，也可以再细分到应用级示例文件，但 `P7-3` 先保证有一份统一入口示例即可。

### 4. 视情况补充根脚本

如果目前脚本已经足够覆盖主要场景，本轮可以不继续新增脚本；但文档里必须明确每个脚本的用途和执行位置。

## 文档层

本轮建议补齐以下文档：

- 根目录 `README.md`：重写为当前 monorepo 项目说明
- `apps/web/README.md`：改为 Web 真实说明
- `apps/admin/README.md`：改为 Admin 真实说明
- `docs/phase-2/2026-03-08-P7-3-环境变量与多端启动文档记录.md`
- 更新 `docs/phase-2/里程碑规划.md`

## 文档重点内容

文档里必须明确：

### 环境变量

- 每个变量属于哪一端
- 是否必填
- 默认值是什么
- 主要用途是什么

### 本地端口

明确约定：

- `api-server`：`3011`
- `web`：`3000`
- `admin`：`3002`

### 启动顺序

建议顺序：

1. `pnpm install`
2. 配置 `.env`
3. 执行数据库迁移
4. 启动 `api-server`
5. 启动 `web`
6. 启动 `admin`

### 数据库常用命令

包括：

- 生成 migration
- 执行 migration
- 打开 Drizzle Studio
- 查看状态

### 常见排查项

例如：

- 端口占用
- API 地址配置错误
- `better-sqlite3` 原生依赖问题
- `.env` 未生效

## 完成标准

`P7-3` 完成后，应满足：

- 三端关键环境变量有统一说明
- Web / Admin 的 API 地址变量命名更清晰
- 根目录与应用 README 不再保留模板误导信息
- 新开发者可以按文档完成数据库初始化与三端启动
- 本地命令至少再跑一轮验证，确认文档与现实一致
