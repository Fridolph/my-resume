# P6-4：版本回滚与恢复策略设计

## 背景

`P6-3` 已经完成统一历史快照能力，当前系统已经可以：

- 为文案、简历、项目持续生成版本快照
- 在后台查看每条内容的历史版本列表
- 为旧数据自动补首版快照

这意味着系统已经具备了“历史可追踪”的能力，但仍然缺少“历史可恢复”的能力。

如果后台只能查看历史，却不能把历史版本恢复为当前内容，那么版本系统仍然只完成了一半。

因此，`P6-4` 的目标，是在 `P6-3` 的版本快照基础上，为三类核心内容建立统一的回滚 / 恢复策略。

## 本轮目标

本轮聚焦“最小可用恢复闭环”，重点完成：

- 后端支持按历史版本恢复当前内容
- 恢复操作继续经过统一数据层，而不是绕过主表写入链路
- 恢复完成后自动生成一条新的 `restore` 版本记录
- 后台管理页可以直接点击某个历史版本执行恢复

本轮暂时不做：

- 版本 Diff 对比恢复
- 部分字段恢复
- 批量恢复多个模块
- 恢复审批流
- 恢复备注与审计原因
- 前台公开站点的“预览恢复”能力

也就是说，`P6-4` 先解决“能恢复”，不解决“高级恢复协作”。

## 适用范围

本轮继续覆盖三类核心内容：

- `TranslationRecord`
- `ResumeDocument`
- `ProjectRecord`

并继续沿用 `P6-3` 的统一版本表：

- `content_versions`

## 核心设计原则

### 1. 恢复是一种新的当前写入，而不是时间回退

恢复历史版本，不是把数据库“回到过去”，而是：

- 读取过去某个版本的快照
- 以该快照为蓝本，重新生成一份新的当前内容
- 将这次操作视为一次新的正式更新

这样做的好处是：

- 当前表永远只有一份最新结果
- 历史链路不会断裂
- 恢复后仍然保留恢复前的状态与数据痕迹

### 2. 恢复后必须再生成一个新版本

如果恢复操作只改主表、不生成新版本，那么版本链会中断。

所以本轮规定：

- 每次恢复成功后，必须写入一条新的 `restore` 版本

这样版本时间线就会清晰变成：

- v1：seed
- v2：update
- v3：update
- v4：restore

也就是说，恢复并不会“删除未来”，而是追加一条新的现实版本。

### 3. 审计字段以“当前恢复操作者”为准

这里有一个非常重要的语义选择。

如果恢复的是一条历史快照，那么审计字段到底应该：

- 完全照搬历史快照里的 `updatedBy` / `reviewedBy` / `publishedAt`
- 还是按“这次恢复是谁做的”重新计算

本轮选择第二种，也就是：

> 内容结构来自历史快照，但审计语义属于这次恢复操作本身。

因此恢复时：

- `updatedBy` 应该是当前执行恢复的人
- 如果恢复后的状态是 `published`，那么 `reviewedBy` 应该是当前恢复人
- 如果恢复后的状态是 `published`，那么 `publishedAt` 应该写为当前恢复时间

这样后台看到的语义才准确：

- 当前内容是由谁恢复出来的
- 当前这次发布是在什么时候重新生效的

而不是误以为“旧版本的发布时间一直延续到今天”。

### 4. 恢复仍然走统一模块写入函数

本轮不允许回滚操作直接在控制器里拼 SQL 或绕开数据层。

恢复必须复用现有数据写入链路，让这些能力继续生效：

- 数据结构归一
- 审计字段生成
- 版本快照生成
- 排序归一化（项目）
- 后续共享逻辑扩展

因此，恢复本质上是为现有写入函数增加一种新的 `changeType`：

- `restore`

## 数据模型调整

在 `P6-3` 的基础上，本轮会扩展：

- `ContentVersionChangeType`

新增取值：

- `restore`

这样版本来源就变成四类：

- `seed`
- `create`
- `update`
- `restore`

## 恢复策略

### 文案

恢复文案时：

- 读取目标版本的 `snapshot`
- 取其中的 `namespace`、`key`、`locale`、`value`、`status`
- 调用文案更新逻辑写回主表
- 审计字段按当前操作者重新计算
- 生成一条新的 `restore` 版本

### 简历

恢复简历时：

- 读取目标版本中的整份 `ResumeDocument` 快照
- 以该快照为基础恢复当前简历
- 审计字段按当前操作者重新计算
- 生成一条新的 `restore` 版本

### 项目

恢复项目时：

- 读取目标版本中的 `ProjectRecord` 快照
- 恢复项目正文、多语言、状态、封面、标签等字段
- 保留当前项目 ID，不允许恢复成另一条项目
- 排序继续通过现有项目更新链路处理
- 生成一条新的 `restore` 版本

## 接口设计

本轮为三类模块增加恢复接口：

- `POST /api/admin/translations/:translationId/versions/:versionId/restore`
- `POST /api/admin/resume/versions/:versionId/restore`
- `POST /api/admin/projects/:projectId/versions/:versionId/restore`

接口特点：

- 仍然受后台鉴权保护
- 恢复动作属于写操作，因此需要对应 `*.write` 权限
- 接口参数只需要路径参数，不额外要求请求体

## 安全边界与校验规则

本轮恢复时必须校验：

### 1. 版本必须存在

如果目标版本不存在，直接返回 `404`。

### 2. 版本必须属于当前实体

例如：

- `translationId = tr_1`
- 但传进来的 `versionId` 实际属于 `tr_2`

这种情况必须拒绝，避免跨实体误恢复。

### 3. 版本模块类型必须匹配当前接口

例如：

- 文案恢复接口不允许用项目版本 ID
- 简历恢复接口不允许用文案版本 ID

### 4. 删除实体不纳入本轮恢复范围

当前项目虽然支持删除项目，但本轮暂不支持“从历史版本恢复已删除项目”。

也就是说：

- 恢复是针对仍然存在的实体执行
- 删除后恢复实体本身，留待后续独立扩展

## 后台交互策略

### 文案页

在版本历史卡片中为每个版本增加“恢复此版本”按钮。

### 简历页

在版本历史列表中为每个版本增加“恢复此版本”按钮。

### 项目页

在项目版本历史列表中为每个版本增加“恢复此版本”按钮。

本轮交互先用最轻量方案：

- 点击即恢复
- 成功后刷新当前内容与版本列表
- 用 Toast 提示成功或失败

本轮先不额外引入确认弹窗。如果后续误操作风险增大，再在 UI 层增加二次确认。

## 与 `P6-5` 的关系

`P6-4` 解决的是“单条内容如何恢复”。

但后续多模块联动发布时，还会遇到另一个问题：

- 文案恢复了
- 项目恢复了
- 简历没恢复
- 前台公开内容是否会出现版本错位

这就是 `P6-5` 要解决的“多模块统一发布策略”问题。

所以 `P6-4` 完成后，下一步就可以从单条恢复能力，继续走向多模块协同内容治理。

## 完成标准

`P6-4` 完成后，应满足：

- 三类核心内容支持按历史版本恢复
- 恢复走统一数据层，不绕过既有写入链路
- 恢复后自动生成新的 `restore` 版本
- Admin 页面可直接操作恢复
- 本地类型检查、接口联调与页面链路验证通过
