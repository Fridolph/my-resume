# P7-1：API Server 关键链路测试记录

## 背景

进入 `P7` 之前，第二阶段的后端业务主链路其实已经比较完整，但仓库里仍然没有正式测试基础设施。

这意味着当前系统虽然：

- 能启动
- 能联调
- 能人工验证

但还缺少一层非常关键的保障：

- 改完代码后，能不能快速自动证明核心链路没有回归

所以这一轮 `P7-1` 的目标，不是继续堆业务功能，而是为 `apps/api-server` 补齐一套最小可用的关键链路自动化测试底座。

## 本次实现

### 1. 为数据库补齐测试库隔离能力

本次最先处理的不是测试文件，而是数据库隔离问题。

因为 `packages/database` 之前默认直接使用：

- `data/platform.sqlite`

如果测试直接操作这份主开发库，就会带来两个严重问题：

- 测试污染日常开发数据
- 测试结果依赖当前本地库状态，变得不稳定

所以本次先在：

- `packages/database/src/client.ts`

补充了：

- `REPO_DATABASE_PATH` 环境变量支持
- 按数据库路径缓存 SQLite 连接
- `closeDatabaseConnection()` 清理能力

这样测试就可以独立使用：

- `data/platform.test.sqlite`

而不会碰主库。

### 2. 建立 API Server 测试基础设施

本次在 `apps/api-server` 中新增了：

- `vitest`
- `@nestjs/testing`
- `supertest`

并补充：

- `apps/api-server/vitest.config.ts`
- `apps/api-server/test/helpers/test-app.ts`
- `apps/api-server/test/api-server.integration.test.ts`

同时在 `apps/api-server/package.json` 中增加：

- `test`
- `test:run`

这样 `apps/api-server` 从这一轮开始，正式具备可执行的自动化测试入口。

### 3. 建立测试态 Nest 应用工厂

本次新增：

- `apps/api-server/test/helpers/test-app.ts`

这里统一完成：

- 设置测试数据库路径
- 删除旧测试库
- 执行 migration
- 启动测试态 Nest 应用
- 关闭应用并清理测试库

这一步很重要，因为它保证了每次执行测试时，环境都是可控且可重复的，而不是依赖当前本地已有数据。

### 4. 落地一条覆盖关键链路的 API 集成测试

本次没有先写很多零散小测试，而是先补一条“真正高价值”的关键链路集成测试。

当前测试文件覆盖了以下场景：

#### 认证与受保护接口

验证：

- 未登录访问后台接口会返回未授权
- 正确账号密码可以登录
- 登录态可以在后续请求中持续生效

#### 内容状态流转

验证：

- 简历状态不允许 `draft -> published` 直接跳转
- 合法链路 `draft -> reviewing -> published` 可以成功执行
- 审计字段会随合法流转写入

#### 版本快照与恢复

验证：

- 简历版本列表能正确生成快照
- 恢复历史版本后会生成新的 `restore` 版本

#### 统一发布批次

验证：

- 在存在已发布简历的前提下，可以创建统一发布批次
- 新批次会成为 `active`
- 再次创建批次时，旧批次自动归档
- 可以重新激活历史批次

#### 公开接口读取激活批次

验证：

- `/api/public/release` 能读取当前激活批次
- `/api/public/resume` 能返回激活批次中的公开简历内容

这条测试链路基本覆盖了第二阶段最核心的后端工作流闭环。

## 本次排查与修正

这轮测试接入过程中，还顺手暴露并解决了几个很有价值的问题。

### 1. `vitest` 初始无法解析 workspace 别名

第一次跑测试时，`@repo/types` 和 `@repo/database` 无法被 `vitest` 正确解析。

本次通过：

- 在 `apps/api-server/vitest.config.ts` 中显式配置 alias

解决了这个问题。

### 2. 登录测试最开始没有正确复用 cookie

最初测试虽然登录成功，但后续请求没有正确带上登录态，导致受保护接口仍然返回 `401`。

本次改为：

- 使用 `supertest.agent()` 持久化 cookie

让测试流更贴近真实浏览器行为。

### 3. 测试回传了不该直接回传的审计字段

第一次做简历更新测试时，直接把 `GET /resume` 返回的整份对象原样提交回 `PUT /resume`。

但其中包含：

- `updatedBy: system@local`

而这不满足当前 schema 对 email 的校验要求，因此触发了 `VALIDATION_ERROR`。

这也提醒我们：

> 测试不应该无脑回传整份响应对象，而应该构造符合写接口契约的请求体。

所以本次又补了一个专门的 payload 归一辅助逻辑，只发送更新接口真正需要的业务字段。

### 4. 测试步骤之间的状态影响需要显式控制

在“版本恢复”测试之后，简历被恢复到了 `draft` 状态，导致后续“统一发布批次”测试没有可发布简历可用，从而失败。

这说明集成测试里的场景虽然串联，但状态依赖必须显式处理。

本次最终在统一发布测试开始前，显式把简历重新推进到：

- `reviewing -> published`

让测试链路可预测、可重复。

## 一个阶段结论

做到这里，`apps/api-server` 已经不再只有人工联调能力，而是开始具备正式的自动化回归基础。

虽然这还不是完整测试体系，但它已经完成了第二阶段收口最重要的一步：

- 把最关键、最容易回归的后端主链路变成可重复执行的自动化验证

这会显著降低后续继续推进 `P7-2`、`P7-3`、`P7-4`、`P7-5` 时的回归风险。
