# P3：简历管理模块真实持久化记录

## 背景

在 `P3` 已完成用户管理和 `i18n` 文案管理之后，第二阶段已经不再只是做单点模块演示，而是开始逐步把后台核心内容真正迁移到统一数据链路上。

接下来最适合推进的模块就是简历管理。原因很明确：它同时连接了后台编辑入口与前台展示目标，是后续 `P4` Web 公开内容全面接入后台数据之前，最关键的一条中间链路。如果简历内容仍停留在本地 `useState` 或 mock 数据里，那么前台简历页后续即使要接正式 API，也缺少稳定的数据来源。

## 本次目标

本次 `P3` 简历模块迁移的目标是：

- 建立简历文档表结构
- 建立共享数据库仓储函数
- 在 `apps/api-server` 中增加 `ResumeModule`
- 提供简历文档查询与更新接口
- 在 `packages/sdk` 中补充简历 API Client
- 将 `apps/admin` 的简历管理页切换到真实 API

这一步完成后，后台简历管理页就不再只是本地演示状态，而会正式进入 SQLite + Drizzle ORM 的持久化链路。

## 为什么本次选择“单文档 + 多语言 JSON”模型

这次没有把简历拆成很多张子表，而是先选择了一种更稳妥的正式落地方式：单文档 + 多语言 JSON。

当前 `ResumeDocument` 本身已经是一个结构化文档对象，包含：

- 发布状态
- 更新时间
- 多语言 `locales`
- 每种语言下的基础信息、教育经历、工作经历、技能分组、联系方式

在这个阶段，优先目标是打通后台真实读写链路，而不是过早追求拆表精细化。如果现在就把教育经历、工作经历、技能分组、联系方式全部拆成多张关系表，那么数据建模和更新逻辑会明显复杂化，也会拖慢 `P3` 的推进节奏。

因此，本次采取的策略是：

- 数据层先用单表保存整份简历文档
- 将多语言内容作为 JSON 存入 `locales`
- 后续在 `P6` 做版本流、发布流或审计追踪时，再按需要继续拆分

这个方案更符合当前阶段“先建立正式主链路，再逐步演进模型”的原则。

## 本次实现

### 1. 建立 `resume_documents` 表结构

本次在 `packages/database` 中新增了简历文档表 schema，并补充了对应迁移文件。

当前表结构包含：

- `id`
- `status`
- `locales`
- `updated_at`

其中：

- `status` 用于承载 `draft / reviewing / published / archived`
- `locales` 用于承载 `zh-CN` 和 `en-US` 的整份结构化简历内容

这意味着简历文档已经从页面级本地状态，进入数据库正式存储。

### 2. 建立简历仓储与默认种子

本次在共享数据库包中新增了简历仓储能力，主要包括：

- 默认简历文档种子写入
- 简历文档读取
- 简历文档更新

默认种子沿用了当前后台页面里已经整理好的双语简历内容，包括：

- 基础信息
- 教育经历
- 工作经历
- 技能分组
- 联系方式

这样可以保证迁移后的后台页面数据表现与迁移前保持一致，减少本轮改造引入的额外变量。

### 3. 在 `apps/api-server` 中建立 `ResumeModule`

本次新增了：

- `ResumeModule`
- `ResumeController`
- `ResumeService`
- 简历文档请求校验 schema

并补充了以下正式接口：

- `GET /api/resume`
- `PUT /api/resume`

这意味着后台简历管理已经正式进入 NestJS 模块体系，而不再只是在页面内部修改本地状态。

### 4. 在 `packages/sdk` 中增加简历 API Client

为了保持后台访问 API 的方式一致，本次在 `packages/sdk` 中补充了简历模块客户端能力，并继续沿用平台 API Client 的统一扩展方式。

当前前端已经可以通过统一 SDK 完成：

- 简历文档查询
- 简历文档更新

这也为后续 `P4` 的前台简历页接入真实 API 做好了接口基础。

### 5. 将 `apps/admin` 简历管理页切换为真实 API

本次已经将后台简历页从本地 `useState` 驱动，迁移为通过 API Server 驱动。

当前页面已经完成以下变化：

- 页面加载时通过真实接口读取简历文档
- 保存当前语言内容时通过 `PUT /api/resume` 持久化
- 发布状态切换通过正式接口更新
- 语言切换、结构化表单编辑、列表项增删仍保留原有交互体验
- 页面 SSR 数据已经来自真实接口返回的简历文档

这说明这一轮改造仍然遵守既定原则：优先替换数据源，而不是推倒重写现有后台页面。

## 本地联调中的问题与修复

这次联调过程中还出现了一个真实问题：后台简历页在切换到 API 版本后，首次访问返回了 `500`，错误信息为：

- `Element is missing end tag`

### 问题定位过程

定位过程分成了三步：

1. 先通过 `curl http://localhost:3003/resume` 发现页面返回的是 Vite 编译错误，而不是接口错误。
2. 再根据错误提示定位到 `apps/admin/app/pages/resume/index.vue` 的 `template v-else` 区域。
3. 继续逐段检查模板层级，最终确认是因为新增 `pending / error / success` 三态模板时，最外层容器缺少结束标签，导致 SFC 模板结构不完整。

### 修复方式

本次修复没有改动业务逻辑，只做了结构收口：

- 补齐外层 `div` 的结束标签
- 冷启动 `admin` 开发服务，清理旧编译缓存
- 再次验证页面是否恢复 `200 OK`

修复后，页面已经可以正常 SSR，且 `__NUXT_DATA__` 中能够看到来自真实接口的简历文档数据。

这类问题也提醒我们：在对现有大页面做“数据源迁移”时，模板结构问题往往比接口问题更容易被忽略，尤其是在加入 `pending / error / success` 三态渲染之后，更要注意最外层标签是否完整闭合。
