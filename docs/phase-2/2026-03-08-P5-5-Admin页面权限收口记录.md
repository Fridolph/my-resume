# P5-5：Admin 页面权限收口记录

## 背景

到 `P5-4` 为止，后端已经可以按权限放行接口，但前端页面层还没有完全收口。

这会带来两个问题：

- 用户虽然没有权限，但仍然可能看到不该看到的导航入口
- 页面虽然能访问，但读写边界不够清晰

因此，这一轮 `P5-5` 的重点，是让后台导航、首页入口和页面内操作，都和真实权限集合保持一致。

## 本次实现

### 1. 抽取统一页面访问配置

本次新增了 `apps/admin/app/composables/useAdminAccess.ts`。

这里统一定义了后台页面的：

- 标题
- 描述
- 路由
- 读取权限
- 写入权限

这样之后：

- 导航菜单
- 后台首页快速入口

都可以复用同一份配置，而不再各自维护一套权限判断逻辑。

### 2. 后台导航改为按权限动态显示

本次 `app.vue` 中的 `UNavigationMenu` 已切换为基于 `visibleRouteItems` 渲染。

这样不同角色登录后，顶部导航会自动收缩为“当前账号真正可访问的页面集合”。

### 3. 首页快速入口改为按权限动态显示

本次后台首页的快捷卡片不再写死，而是改为读取同一份页面访问配置，并过滤掉没有访问权限的页面。

这意味着后台首页开始真正成为“当前账号视角”的工作台，而不再是所有模块入口的固定列表。

### 4. 设置页拆分为只读与可编辑两层能力

本次重点把设置页作为页面级权限收口样板。

当前逻辑改为：

- 具备 `site.read` 就能进入页面
- 仅具备 `site.write` 才显示保存、新增、删除等写操作入口
- 只读账号进入页面时，会看到明确的只读提示
- 表单区域继续展示，但通过禁用状态保持“可看不可改”

这样设置页就成为当前后台中第一个较完整的“同一页面，不同权限层级，不同交互能力”的示例。

## 本次实现的价值

这一轮的价值在于：前端权限开始真正和后端权限对齐。

从现在开始：

- 能不能看到入口，不再只是 UI 设计问题
- 能不能进入页面，不再只是路由问题
- 能不能执行动作，也不再只是按钮显隐问题

而是三者都开始共同受真实权限集合驱动。

### 权限联调

#### `translator`

登录 `translator@fridolph.local` 后：首页导航与快捷入口仅保留：

- `/`
- `/translations`
- `/settings`

同时：

- 不再显示 `/users`
- 不再显示 `/resume`
- 不再显示 `/projects`

直接访问 `/users` 时，会被路由层拦到无权限页面。

#### `viewer`

登录 `viewer@fridolph.local` 后访问 `/settings`：

- 可以进入页面
- 可以看到只读提示
- 看不到“保存站点配置”按钮
- 看不到“新增社交链接”按钮
- 看不到“新增下载资源”按钮
- 看不到删除按钮

这说明设置页已经正确拆分为“可读”和“可写”两层能力。

#### `super-admin`

登录 `root@fridolph.local` 后：首页导航与快捷入口可看到全部后台模块：

- `/`
- `/users`
- `/translations`
- `/resume`
- `/projects`
- `/settings`

这说明页面权限收口没有影响高权限账号的完整操作链路。

## 过程中确认的一点

这一轮联调也再次证明了一个设计判断：

- 后端接口守卫负责真正的安全边界
- 前端页面权限负责减少错误入口和优化使用体验

两者缺一不可。

如果只有后端守卫，没有前端收口，页面仍然会显得“处处都能点，但点进去才发现不能用”；而现在这轮完成后，后台体验和权限模型已经开始对齐。
