# 第 05 章：简历 RAG 版本 5 的策略修正、配置外置与上下文构建拆分

## 背景

在版本 4 中，我们已经完成了三件很重要的事情：

- 会话历史不再依赖 `RESUME_RAG_HISTORY` 环境变量，而是改成本地 session store
- Prompt 不再硬编码在 JS 中，而是拆到了 `prompts/` 目录
- 增加了一个“重排后再去噪”的阶段

这让整个 demo 更像一个真正可演进的问答系统，而不只是单次脚本。

但在继续做代码阅读时，又暴露出一批新的问题。  
这一次的问题已经更偏“架构清理”和“实现质量提升”。

---

## 这轮分析暴露出的关键问题

### 1. `detectQuestionStrategy()` 存在死代码风险

之前的规则里：

- `experience` 分支已经匹配了“项目”
- 后面的 `project` 分支又单独匹配“项目”

这会导致：

- 一旦问题里含“项目”
- 很可能优先被 `experience` 命中
- `project` 分支几乎不会真正触发

这类问题并不是功能完全不可用，  
但它说明分类逻辑的优先级已经不够清晰。

版本 5 对此的改进方向是：

- 先判断是否是“纯项目类问题”
- 再判断是否是“经历/经验类问题”
- 避免把所有含“项目”的问题一律吞进 `experience`

### 2. `formatContext()` 不应把分数注入 Prompt

之前的实现中，进入 Prompt 的 context 里含有：

- `rawScore`
- `rerankScore`

这些字段对开发者调试很有用，  
但它们不一定应该出现在 LLM 的输入里。

原因有三个：

1. 这些分数更适合给人看，而不是给模型看
2. 模型可能被这些数值干扰注意力
3. 这属于“调试元数据”，而非真正的业务证据

因此版本 5 的明确改进是：

> **分数继续保留在日志里，但不再注入 Prompt context。**

### 3. `getKeywordHints()` 和 `scoreSectionBoost()` 仍然写死在代码里

之前虽然已经把 Prompt 做到了目录化，  
但还有两块明显的领域知识仍然散落在代码里：

- 关键词扩展词
- 不同 strategy 下的 section 权重

这会带来两个问题：

1. 改规则要改源码，不利于实验
2. 换一个简历场景时，迁移成本高

因此版本 5 继续把这两类内容外置为配置：

- `keyword-hints.json`
- `section-boost-config.json`

### 4. `runRAG()` 职责仍然偏重

版本 4 的 `runRAGv4()` 虽然已经比早期脚本清晰很多，  
但它还是同时负责：

- 向量化
- 检索
- 重排
- 去噪
- 组装 context
- 构建 prompt
- 调用 LLM

这在工程上已经不算差，但如果继续往下演进，会逐渐变成一个“什么都做”的函数。

所以版本 5 的设计选择是：

> **把“检索上下文构建”拆成单独的 `buildRAGContext()`，再由 `runRAGv5()` 负责真正的生成阶段。**

这样做的好处是：

- 可以单独测试“检索与上下文质量”
- 可以做“只检索不生成”的场景
- 更容易继续插入别的阶段（如语义 rerank、answer 评估）

### 5. 多轮对话在调用层继续进化为 Message 数组

版本 4 虽然已经把历史从环境变量搬到 session store，  
但 Prompt 仍然是“把历史拼成字符串，再塞进模板”。

这是可用的，但还不是最自然的对话调用方式。

因此版本 5 继续往前一步：

- System Prompt：用模板文件渲染
- History：转换成 `HumanMessage / AIMessage`
- Current Question：单独作为最新 `HumanMessage`

也就是说，版本 5 开始不再只是“拼一个大字符串 Prompt”，  
而是开始使用更贴近 LangChain / Chat 模型原生接口的方式。

---

## 版本 5 的核心改进方向

这次版本 5 的重点可以概括为 4 个词：

### 1. 策略修正

修正 `detectQuestionStrategy()` 的分支优先级，  
避免 `project` 分支被 `experience` 提前吞掉。

### 2. 配置外置

把这两类强业务规则从代码中拿出来：

- 关键词扩展词
- section 权重

后续实验时，可以只改配置，不改流程代码。

### 3. 上下文构建拆分

把“检索到生成”拆成两层：

- `buildRAGContext()`：只负责取证据
- `runRAGv5()`：负责组装消息并生成答案

### 4. Prompt 与多轮对话进一步规范化

版本 5 继续使用 Prompt 文件目录，  
但同时把历史消息交给 Chat 模型的 message 数组，而不是全都糅成一个字符串。

### 5. 重新校准 `scoreSectionBoost` 与去噪阈值

这一轮还有一个很关键的补充认知：

> **规则重排本质上只是在做“微调”，它无法完全对抗 base score 的强势优势。**

也就是说，如果某条 `skills` 片段在向量相似度上天然更高，  
仅仅给它一个 `-0.02` 的惩罚，很多时候并不足以把它压下去。

#### 为什么 `-0.02` 太弱

从实际分布看，当前这套数据里：

- `base score` 常见落在 `0.45 ~ 0.75`
- `keywordBoost` 单次命中是 `+0.015`
- `keywordBoost` 总上限是 `+0.09`
- section boost 最高大约是 `+0.12`

这意味着：

- `+0.015` 属于**很轻微的微调**
- `+0.09` 属于“足以影响接近分数排序”的幅度
- `-0.02` 对高分 `skills` 片段几乎构不成有效压制

因此版本 5 这次做了两项明确调整：

1. `experience` 策略下：
   - `skills` 从 `-0.02` 改为 `-0.08`
2. 增加一个“当前数据集经验阈值”：
   - `rawScoreNoiseThreshold = 0.50`

注意，这个 `0.50`：

- **不是通用真理**
- **不是跨模型、跨数据集都成立的固定阈值**
- 它只是当前这份简历数据、当前 embedding 模型、当前召回分布下的一个经验值

因此版本 5 的实现思路不是：

- “小于 0.50 一律删除”

而是：

- 把“小于 0.50 且又没有主题 hint、也不在优先 section”的片段，视为强噪声候选

也就是说，这个阈值是：

- **局部经验阈值**
- **结合业务规则使用**
- **不是独立生效的绝对裁决器**

#### 这次落到代码里的改动

版本 5 把这些规则继续外置到配置层：

- `section-boost-config.json`
- `rerank-config.json`

其中：

- `experience.skills.default = -0.08`
- `experience.__default = -0.05`
- `rerank-config.rawScoreNoiseThreshold = 0.50`
- `rerank-config.keywordBoostPerHit = 0.015`
- `rerank-config.keywordBoostMax = 0.09`

这样后续继续实验时，不需要再改源码，只需调整配置文件。

## 版本 5 的设计结果

### 新增模块职责

#### 1. `prompt-builder-v5.mjs`

职责：

- 读取 Prompt 模板文件
- 加载 strategy → template 的映射配置
- 把历史消息转换成 LangChain `Message` 数组
- 最终构建 `messages`

#### 2. `context-builder.mjs`

职责：

- 问题分类
- 问题向量化
- Milvus 检索
- 重排
- 去噪
- 生成“仅包含业务证据”的 context

注意：

- 这一层不负责调用大模型
- 只负责把“证据上下文”准备好

#### 3. `rag-pipeline-v5.mjs`

职责：

- 调用 `buildRAGContext()`
- 调用 `buildMessages()`
- 交给 LLM 生成回答

### 与版本 4 的关键差异

| 项目 | 版本 4 | 版本 5 |
|------|--------|--------|
| `strategy` 判断 | 存在潜在死代码 | 已修正判断优先级 |
| Prompt context | 含调试分数 | 不再注入分数 |
| 关键词扩展词 | 写死在代码 | 外置到 JSON |
| section boost | 写死在代码 | 外置到 JSON |
| `skills` 降权 | `-0.02` 偏保守 | 调整为 `-0.08` |
| raw 噪声阈值 | 无明确经验阈值 | 增加 `0.50` 的当前数据集经验阈值 |
| 检索/生成职责 | 仍偏集中 | 拆成 `buildRAGContext + runRAGv5` |
| 多轮对话输入 | 历史字符串拼接 | 使用 `Message` 数组 |

## 关于“噪声阈值”的进一步说明

版本 4 已经明确过：

> 不能简单依赖一个固定绝对阈值（如 `< 0.1`、`< 0.05`）来判断噪声。

版本 5 延续这个结论，并进一步强调：

- 去噪仍然是“相对排序 + 业务优先级 + hint 命中”的组合判断
- 版本 5 的重点不在于重新发明一套分数阈值
- 而在于让上下文构建更干净、更可解释、更容易实验

所以版本 5 没有把“阈值问题”简单暴力化，  
而是优先修正更关键的结构问题：

- strategy 错位
- Prompt 污染
- 配置散落
- runRAG 过重

但在这一轮补充里，版本 5 也进一步承认一个现实：

> **如果某条记录的 base score 已经明显偏低，那么只靠 boost 抬分通常意义不大。**

因此这次增加的 `0.50` 阈值，不是为了搞一个“万能规则”，  
而是为了在当前数据集上先挡掉一批明显边缘的低质量候选。

---

## 当前结论

这轮的结论可以沉淀为一句话：

> **版本 5 的重点，不是继续增加更多规则，而是把已有规则整理成“更清晰的分类逻辑、更干净的 Prompt 输入、更可配置的策略层、更可拆分的上下文构建过程”。**

再加上这轮补充后，可以更具体地理解为：

> **版本 5 既修正了结构问题，也开始把“权重到底有多大影响”这件事显式量化，让 `section boost / keyword boost / raw threshold` 的作用边界更容易被学习和调整。**
