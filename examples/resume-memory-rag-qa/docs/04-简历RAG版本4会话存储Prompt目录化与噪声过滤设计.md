# 第 04 章：简历 RAG 版本 4 的会话存储、Prompt 目录化与噪声过滤设计

## 背景

在第 03 章中，我们已经把 `ask 3` 重构成了管线化结构：

- `Retriever`
- `PromptBuilder`
- `RAGPipeline`
- `ask-resume-3-pipeline-ask.mjs`

这一步解决的是“结构可复用”的问题。  
但在继续往前推时，又出现了一批更偏工程化和产品化的问题。

这次的思考重点不再只是“怎么把回答做出来”，而是：

- 会话历史应该怎么存？
- Prompt 是否应该继续写死在代码里？
- 噪声片段怎么判断？
- 版本 4 的演进应该沿着什么方向走？

---

## 这轮提出的真实问题

这次重点提出了以下几个问题：

### 1. `RESUME_RAG_HISTORY` 不应该靠环境变量硬塞

当前 `ask 3` 里虽然支持：

- `RESUME_RAG_HISTORY`

但实际上并没有在 `.env` 中定义这个变量，而且从使用体验上看，它也不适合作为真正的“对话历史存储方式”。

问题在于：

- 每次 CLI 调用都是一个独立进程
- 历史要手动拼成 JSON 字符串
- 用户需要自己维护历史结构
- 这不是真正的“会话状态”

所以这里得到的结论是：

> **`RESUME_RAG_HISTORY` 可以保留为调试兜底能力，但不应该作为版本 4 的主方案。**

### 2. 会话历史不应该混进当前简历向量 collection

这里还进一步问到：

- 是不是要新建一个 collection？
- 还是继续往当前的 `resume_profile_chunks` 里写？

这个问题本质上是在问：  
“会话状态”是不是应该和“简历知识库”混在一起？

答案是否定的。

原因很明确：

- `resume_profile_chunks` 当前承载的是**简历知识块**
- 它服务的是“候选人资料检索”
- 会话历史承载的是**当前用户会话状态**
- 它服务的是“多轮上下文保持”

如果混进同一个 collection，会出现几个问题：

1. **语义污染**
   - 历史回答会被当成新的知识片段
   - 检索时可能把“AI 自己生成过的话”又召回来

2. **schema 语义混乱**
   - 简历块和会话块本来就不是同一类数据

3. **调试困难**
   - 你会越来越难判断，召回的是“原始事实”，还是“历史回答”

因此这轮的明确结论是：

> **版本 4 不把会话历史写进当前 `resume_profile_chunks`。**

### 3. 那要不要新建一个 Milvus collection 来存会话历史？

这里的答案是：

- **当前版本 4 先不新建 Milvus 会话 collection**
- **先用本地文件式 session store**

原因是：

#### 适合当前 demo 阶段

当前场景是：

- 本地 CLI 学习 demo
- 单人使用
- 重点还在“链路理解”和“能力抽象”

这时直接上“第二个向量 collection 来存对话历史”，会把问题复杂化。

#### 会话状态本质上更像运行时数据

当前会话历史更适合看成：

- 某个 `sessionId` 下的上下文缓存
- 运行时状态
- 短期记忆

这类数据更适合先用：

- 本地 JSON 文件
- SQLite
- Redis

而不是一开始就做语义向量化。

#### 只有在“要做历史语义检索”时，才考虑单独 history collection

例如未来如果真要做：

- 很长会话的语义压缩
- 跨 session 的历史经验召回
- “你上次问过什么”这类记忆型 Agent

那时再新建一个独立 collection 会更合理，例如：

- `resume_chat_history_chunks`

但这已经是下一阶段问题，不是当前版本 4 的首要任务。

---

## 关于“会话历史是否只存 answer”

这次还有一个很关键的问题：

> 当前用户对话结果是否只存 `answer` 部分？

从直觉上看，好像只存回答就够了。  
但真正落到多轮对话时，会发现这不够。

因为如果只存 answer，会丢掉：

- 用户原始问题
- 问题与回答之间的对应关系
- 后续追问的上下文锚点

例如：

第一轮：

- 用户：这个候选人有哪些 AI Agent 开发相关经验？
- 助手：回答了一段项目与技能总结

第二轮：

- 用户：那其中最能代表他的项目是哪一个？

如果只存上一轮 `answer`，而没有上一轮 `question`，  
系统对“那其中”指的是什么，会明显更模糊。

所以这轮的最终结论是：

> **版本 4 的 session store 至少要保存 question + answer 成对结构。**

这不是为了“存更多”，而是为了让多轮对话真正可用。

---

## 关于 `RESUME_RAG_FILTER` 和 `RESUME_RAG_PROMPT_TEMPLATE`

这两个变量在 `ask 3` 中虽然能工作，但这轮也明确暴露出一个问题：

- 如果 `.env` 里本来没有这些变量，就不应该在后续编码里不断往环境变量层面扩散

这轮得到的更稳妥结论是：

### 1. Prompt 应该目录化管理，而不是散在代码里

所以版本 4 采用：

- 单独的 `prompts/` 目录

例如：

- `resume_qa.md`
- `resume_experience_qa.md`
- `skill_eval.md`
- `job_match.md`

好处是：

- 模板更容易维护
- 更容易对照修改
- 更适合后续继续增加场景
- Prompt 不再淹没在 JS 逻辑里

### 2. 过滤条件和模板选择更适合走 CLI 参数或代码参数

相比不断新增环境变量，更适合用：

- `--template resume_experience_qa`
- `--filter 'section == "projects"'`
- `--session demo-1`

这样更符合“实验脚本”的使用方式，也更清楚。

这轮结论可以概括为：

> **版本 4 不再围绕新增环境变量扩展，而是转向“prompt 目录 + CLI 参数 + 本地 session store”的工程组织方式。**

## 如何判断召回片段是不是噪音？

这是这轮最重要、也最容易误解的问题之一。

提出的问题是：

> 回归了 8 条之后，如何判断哪条是噪音？  
> 能不能靠固定阈值，比如 `< 0.1` 或 `< 0.05`？

答案是：

> **不能依赖一个全局固定分数阈值。**

### 为什么不能简单用固定阈值

因为向量相似度分数的分布，会受到很多因素影响：

- embedding 模型不同
- 数据集内容不同
- chunk 粒度不同
- 问题写法不同
- metric type 不同

同样是 `0.50`：

- 在某个数据集里可能已经很相关
- 在另一个数据集里可能已经很弱

所以像：

- `< 0.1`
- `< 0.05`

这种阈值，对当前场景几乎没有实际参考意义。  
因为我们现在看到的分数本来就集中在 `0.45 ~ 0.75` 附近。

### 更合理的判断方式

版本 4 采用的是“相对 + 业务”的判断方式，而不是死阈值。

主要看三类信号：

#### 1. 相对分差

不是看一个片段“绝对分是多少”，  
而是看它与头部结果差了多少。

例如：

- 第一名 `0.74`
- 某条只有 `0.61`

如果这条既没有命中 AI / Agent 关键词，也不是当前问题优先 section，  
那它就更可能是噪声。

#### 2. 业务优先级

例如问题是：

- “AI Agent 开发相关经验”

那优先级应更偏向：

- `projects`
- `work_experience`
- `core_strengths`

如果某条虽然分数不算太低，但：

- 没命中任何相关 hint
- 也不是当前问题关心的业务 section

那它更像“语义沾边”，不一定是真正想要的证据。

#### 3. 主题命中情况

如果问题中明确包含：

- `AI`
- `Agent`
- `SSE`
- `工作流`
- `my-resume`

那一个候选片段是否真正命中这些主题，也是判断它是不是噪音的重要依据。

### 版本 4 的结论

所以版本 4 不使用：

- “小于某个固定值就是噪音”

而采用：

- 相对排序
- section 优先级
- keyword / hint 命中
- 必要时做降噪裁剪

这是一种更贴合当前 demo 阶段的方案。

## 版本 4 的演进方向

基于这轮思考，版本 4 决定做以下三件事：

### 1. 引入文件式 session store

作用：

- 不再依赖 `RESUME_RAG_HISTORY` 环境变量
- 用 `sessionId` 自动维护本地多轮对话历史
- 每次问答后自动把本轮结果写入本地 session 文件

### 2. 引入 `prompts/` 目录

作用：

- Prompt 从代码里拆出去
- 模板独立管理
- 后续更容易扩展多个业务场景

### 3. 引入显式噪声过滤逻辑

作用：

- 对重排后的结果再做一次“是否保留”的判断
- 让像“早期无关经历”“领域无关项目”这类片段不轻易进入最终 Prompt

---

## 当前结论

这轮可以沉淀成一句明确结论：

> **版本 4 的重点，不是继续堆更多规则，而是把会话历史、Prompt 模板和噪声过滤这三个能力工程化，让整个简历 RAG demo 更接近真正可持续演进的问答系统。**

---

## 下一步

基于这份结论，版本 4 将新增：

1. 文件式 session history store
2. `prompts/` 模板目录
3. 去噪后的 `ask-resume-4` 会话脚本

并继续保留：

- 原始召回结果
- 重排结果
- 去噪结果

用于后续学习和效果对比。
