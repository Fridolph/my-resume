# 第 03 章：简历 RAG 的管线化重构与可复用设计

## 背景

在第 02 章里，我们已经确认：

- `ask-resume-2-perf-ask.mjs` 的回答质量比初版更好了
- 通过“扩大候选召回 + 简单重排 + prompt 优先级约束”，已经能让回答更偏向项目与工作经历

例如同样的问题：

> 这个候选人有哪些 AI Agent 开发相关经验？

相比最初版本，现在已经能够把这些内容抬上来：

- 成都澳昇能源的 AI 辅助开发实践
- `GreenSketch` 的 AI 开发实践
- `my-resume` 的 AI 分析工作台

这说明优化方向是有效的。

---

## 这轮测试后的新结论

虽然 `ask-resume-2-perf-ask.mjs` 已经明显优于初版，但它仍然是一个“实验性脚本”，还不适合作为后续长期扩展的基础。

### 当前优点

- 召回结果更贴近“经验类问题”的业务预期
- 项目与工作经历不再完全被技能块压制
- 输出同时展示 `raw candidates` 与 `reranked candidates`，方便调试

### 当前仍然存在的局限

| 问题 | 表现 |
|------|------|
| `Prompt` 硬编码 | 换个问题场景就要改 prompt |
| 单轮问答 | 没有对话历史，无法自然追问 |
| 召回策略固定 | `TOP_K`、召回字段、排序逻辑都写死在脚本里 |
| 难复用 | 整个 `main()` 是一个大流程，不适合拆开组合 |
| 迭代成本高 | 每加一种新问答场景，就要再复制一个脚本 |

这意味着：

- `ask 2` 更像是“验证优化方向是否有效”
- 下一步需要进入“结构化重构”

---

## 为什么要做 RAG Pipeline

现在的核心问题，已经不再是某一个 prompt 怎么写，  
而是整个问答流程里，哪些部分是**稳定骨架**，哪些部分是**可变策略**。

更合理的做法是把流程拆成几层：

```txt
QueryBuilder   -> 决定问题类型 / 召回参数
Retriever      -> 去 Milvus 检索候选片段
Reranker       -> 根据业务规则做二次排序
PromptBuilder  -> 根据场景选择 Prompt 模板
LLM            -> 生成最终回答
```

这样做的本质是：

- 把“变化的部分”参数化
- 把“不变的部分”沉淀成框架

---

## 这次确定的重构原则

### 1. 召回层独立

把 Milvus 查询单独抽出来，形成一个 `Retriever`。

这样后续就可以灵活调整：

- `topK`
- `metricType`
- `filter`
- `outputFields`

而不需要改整个问答脚本。

### 2. Prompt 模板独立

把 Prompt 做成模板层，而不是写死在脚本里。

例如后续可以支持：

- `resume_qa`
- `resume_experience_qa`
- `skill_eval`
- `job_match`

这样就能按场景切换，而不是反复复制脚本。

### 3. 主流程独立

把“向量化 → 检索 → 重排 → 拼 context → 构建 prompt → 生成回答”  
抽象为统一的 `runRAG()`。

这样以后写新脚本时，只需要关心：

- 当前问题是什么
- 用哪个模板
- 用什么召回配置

而不是把整条链路重写一遍。

### 4. 允许保留策略层

虽然要做通用化，但也不能把所有“业务感知”都抹平。

例如这次已经证明有效的：

- 经验类问题优先 `projects / work_experience`
- 技能类问题优先 `skills / core_strengths`

这些仍然应保留，只是应该从“硬编码在 `main()`”  
升级为“管线中的可替换策略”。

---

## 这轮设计后的目标

下一步希望得到一版新的 `ask 3`：

- 不修改原有 `ask-resume.mjs`
- 不修改 `ask-resume-2-perf-ask.mjs`
- 新增一版更偏“框架化、可复用、可扩展”的问答脚本

它的目标不是立刻变成“最终生产版”，  
而是作为后续继续实验的基础骨架。

### 这版应该具备的能力

1. 支持用参数切换 Prompt 模板
2. 支持按问题类型自动选择问答策略
3. 支持调整召回数量与过滤条件
4. 支持后续接入多轮对话历史
5. 支持复用到“岗位匹配”“技能评估”等场景

---

## 从“脚本式问答”到“可组合问答”

这次最重要的认知变化是：

> 不是再写一个更长的脚本，而是开始把简历 RAG 的能力拆成可组合模块。

也就是说，后续再写新的问答场景时，不应该再是：

- 复制旧脚本
- 改 prompt
- 再调一次 topK

而应该变成：

- 复用同一个 `pipeline`
- 换一个 `template`
- 换一个 `retrieval option`
- 换一个 `question strategy`

---

## 当前结论

这轮优化后的明确结论可以写成一句话：

> **`ask 2` 证明了“重排 + 经验优先回答”是有效方向；下一步应把这套能力抽象成 RAG Pipeline，让 Prompt、召回策略和主流程解耦，从而为后续多场景复用打基础。**

---

## 下一步

基于这份结论，下一阶段将继续推进：

1. 新建 `Retriever`
2. 新建 `PromptBuilder`
3. 新建 `RAGPipeline`
4. 基于这些模块实现新的 `ask-resume-3` 脚本

后续继续在此基础上做：

- 多轮对话
- 混合召回
- 更细的业务重排
- 与 `my-resume` 正式项目接轨
