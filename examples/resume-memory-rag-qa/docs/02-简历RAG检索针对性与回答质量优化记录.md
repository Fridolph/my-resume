# 第 02 章：简历 RAG 的检索针对性与回答质量优化记录

## 背景

在 `resume-memory-rag-qa` 这套 demo 中，当前主流程已经基本跑通：

- markdown 简历可以被解析成语义记录
- 语义记录可以写入 Milvus
- 可以基于用户问题做向量检索
- 可以把召回片段交给大模型生成最终回答

但在真实提问测试中，一个新的核心问题暴露出来了：

- **能答，不代表答得对路**
- **能召回，不代表召回的就是用户最想看的证据**

这意味着，下一阶段的重点不再是“链路是否跑通”，而是：

- **如何提高检索的针对性**
- **如何让最终回答更贴近用户真实情况**
- **如何让模型优先引用更有代表性的项目 / 经历，而不是只抓到字面最像的问题片段**

## 当前暴露出来的核心问题

### 1. 回答“不是错”，但“不是最想要的”

从严格意义上说，这次回答并不算离谱：

- 它确实引用到了 AI Agent 相关技能
- 也引用到了核心竞争力
- 还带到了一些工作经历和项目经历

但它的问题在于：

- **没有优先回答我最想看的那部分经历**
- **没有把 `my-resume` 作为主叙事中心**
- **更像“泛化总结”，而不是“针对候选人核心 AI Agent 实战经历的回答”**

这对简历问答来说是一个很重要的分界线：

- 用户要的不是“看起来沾边的内容”
- 用户要的是“最能代表这个人的真实项目经验的内容”

### 2. 纯向量召回会优先命中字面最像的问题片段

当前问题是：

- 问题里有 `AI Agent 开发`
- 简历里恰好有一个技能小节标题就叫 `AI Agent 开发`

因此从纯向量召回角度看，这些记录会天然很占优势：

- `skills / AI Agent 开发 / skill_item`
- `core_strengths / AI 工程化实践 / strength_item`

这类结果很像“关键词最接近”的正确答案，  
但不一定是“业务上最重要”的正确答案。

也就是说，当前系统更像在回答：

- “简历中哪里最像 AI Agent 这几个字？”

而不是：

- “候选人最有代表性的 AI Agent 相关项目和实战经验是什么？”

### 3. 项目经验没有被优先排序

就这次问题而言，我更希望系统优先回答：

- `my-resume`
- 其他明确体现 AI 工作台 / 工作流 / SSE / Prompt / Agent 编排的项目

而不是优先回答：

- 技能概述
- 核心竞争力
- 泛化的“AI 辅助开发”

所以问题不是“有没有召回项目”，而是：

- **项目没有排到足够靠前**
- **回答阶段也没有优先拿项目作为主证据**

### 4. 当前 prompt 偏“忠实引用”，但缺少“回答优先级策略”

当前 prompt 更强调：

- 只能基于片段回答
- 不要脑补
- 证据不足要保守

这很好，但还不够。

因为它没有显式要求模型：

- 如果问题是在问“经验 / 做过什么 / 项目经历”
- 那就应优先引用 `projects` 与 `work_experience`
- `skills` 与 `core_strengths` 只能作为补充，不应喧宾夺主

于是模型就会自然地沿着“相似度最高的片段”去组织答案。

## 初步原因判断

基于这次对话和召回结果，可以先做一个比较稳妥的判断：

### 1. 这不是 Milvus 写坏了

当前召回结果说明：

- 向量检索是正常工作的
- 至少能召回相关语义块
- 不是“完全没命中”

所以这不是底层存储故障，而更像是：

- **召回排序问题**
- **业务权重问题**
- **回答策略问题**

### 2. 也不只是单纯的 `topK` 太小

把 `k` 调高，确实可能带来更多相关候选片段。  
但如果不做后续处理，也会带来两个副作用：

- 噪音增加
- 回答更发散

因此单纯增大 `topK` 并不是根治方案，它最多只是“补召回”。

### 3. 更像是“召回 + 重排 + 回答策略”三段都要优化

更合理的方向应该是：

1. **召回层**：先尽量别漏掉相关项目
2. **重排层**：让项目与工作经历在“经验类问题”中有更高权重
3. **回答层**：在生成答案时，显式要求优先引用项目与实战经历

---

## 当前阶段的优化目标

下一步不应只追求“检索更多”，而应追求：

### 1. 提高检索针对性

对于“AI Agent 开发相关经验”这种问题，希望系统更容易命中：

- `projects`
- `work_experience`
- 与问题语义更贴近的 `project_summary / project_detail`

而不是被泛技能描述占满前排。

### 2. 提高回答质量

希望最终回答：

- 更聚焦候选人的代表性经历
- 更符合用户的真实阅读预期
- 不只是“总结了几个相关词”，而是“把最重要的项目经验说出来了”

### 3. 让回答更贴近真实业务口径

简历问答不是通用百科问答，它更强调：

- 代表性
- 可信度
- 结构化归因
- 对候选人真实能力画像的贴近程度

因此后续优化要尽量让系统具备这种业务感知。

---

## 已经明确的优化方向

基于当前判断，后续可以重点从以下几个方向推进：

### 方向 1：适度提高 `topK`

建议先从较保守的范围开始，例如：

- `topK = 5 -> 8`
- 或 `topK = 5 -> 10`

目的不是盲目堆上下文，而是：

- 给后续重排更多候选片段
- 减少“项目片段被截断在候选池外”的概率

### 方向 2：增加检索后二次重排

对于“经验类问题”，可以提高这些 section 的权重：

- `projects`
- `work_experience`

相对降低这些 section 的优先级：

- `skills`
- `core_strengths`

这样可以让召回结果更符合“项目优先”的业务预期。

### 方向 3：按问题类型切换 prompt 策略

后续可以考虑把 prompt 做成“策略模式”：

- 技能类问题：优先技能块
- 项目类问题：优先项目块
- 经验类问题：优先项目 + 工作经历
- 总结类问题：允许技能、项目、经历综合归纳

这样模型就不会总是用同一种回答姿势处理所有问题。

### 方向 4：必要时增强简历源数据的语义表达

如果 `my-resume` 确实是 AI Agent 相关项目，  
那简历原文里也应该更明确表达这些关键词与事实，例如：

- AI 工作台
- SSE 流式输出
- 多 Agent 工作流
- Prompt 策略实验
- RAG / 检索增强
- Agent 协作与验证闭环

否则从语义召回角度，系统很容易先抓住字面上更像问题本身的技能块。

---

## 当前结论

这次问题可以先归纳为一句话：

> **当前 resume RAG demo 的主要问题，不是“检索不到”，而是“检索结果不够针对，导致回答虽然相关，但不够贴近用户真正想看的核心经历”。**

因此，下一步优化重点应明确转向：

- 从“链路跑通”升级到“检索更准”
- 从“回答不胡说”升级到“回答更像一个懂简历场景的助手”
